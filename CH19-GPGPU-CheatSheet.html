§19.1 CUDA Basics

CUDA Programming Model

CUDA DefinitionCompute Unified Device Architecture — NVIDIA's parallel computing platform and programming model. CUDA C extends C/C++ with GPU execution abstractions.

THREE SECTIONS OF A CUDA PROGRAM

  1. Host code — runs on CPU; serial logic, orchestration
  2. Device code (kernel) — runs on GPU; massively parallel
  3. Data transfer code — moves data between host and device memory
KernelData-parallel function executed by thousands of threads simultaneously on the GPU. Should have few or no branching statements — branching forces serial execution in hardware.
Why Thousands of Threads?Maximises GPU core utilisation and available speedup. Thread count typically in the thousands.

Warp — bundle of 32 threads with consecutive IDs that start at the same instruction address. The fundamental GPU scheduling unit.

SIMT = Single Instruction Multiple Threads

§19.1 CUDA Memory Types

GPU Memory Hierarchy — Complete Reference Table

Memory Type Location Access Type Scope Lifetime Relative Speed Typical Use
Registers On-chip (per SM, 32k × 32-bit in Fermi) R/W Single thread only (no sharing) Thread lifetime ⚡ Fastest (1–2 cycles) Thread-private variables, loop counters
Shared Memory On-chip (per SM, up to 48 kB in Fermi) R/W All threads in the same block (SM) Block lifetime ⚡ Fast (~same as registers, ~ns) Inter-thread data sharing within SM; tiling for matrix multiply; reduce redundant global reads
Local Memory Off-chip (GDDR DRAM — despite name) R/W Single thread only Thread lifetime 🐢 100–150× slower than registers Register spill area (automatic); large arrays private to thread
Global Memory Off-chip (GDDR5 DRAM, up to 6 GB in Fermi) R/W All threads + host (CPU) Application lifetime 🐢 100–150× slower than registers Main data store; input/output arrays; host↔device transfers
Constant Memory Off-chip (dedicated 64 kB cached on-chip) R only All threads + host Application lifetime ⚡ Fast when broadcast (all threads same addr) Kernel parameters, lookup tables, coefficients — read-only data broadcast to all threads
Texture Memory Off-chip (cached via texture cache) R only All threads + host Application lifetime ⚡ Fast for 2D spatial locality Image data, 2D spatial access patterns; sampler hardware for filtering
L1 Cache On-chip per SM (configurable: up to 48 kB) Auto-managed Per-SM Auto ⚡ Fast Register spill cache; automatic caching of local/global accesses
L2 Cache On-chip (768 kB shared across all SMs in Fermi) Auto-managed All SMs Auto Moderate Shared L2 between all SMs; L1 misses + L1/shared mem config
Shared vs L1 Config (Fermi)The 64 kB on-chip SRAM per SM is configurable: 48 kB shared + 16 kB L1, or 16 kB shared + 48 kB L1. Programmer chooses based on workload. Optimising shared memory use is the single biggest GPGPU programming lever — eliminating 100–150× off-chip accesses.
Register Spill ChainThread registers → L1 cache (spill) → L2 cache → Local memory (off-chip DRAM). Programmer must minimise register usage per thread to keep all registers on-chip.

Design PhilosophyCPU: optimised to minimise latency for sequential code. GPU: optimised to maximise throughput for parallel data-parallel workloads. GPU hides latency via massive thread-level parallelism context-switching.

Warp Scheduling & Latency Hiding

Zero-Overhead Context SwitchingEach warp's register state is always resident in the register file — no saving/restoring. Switching to a new warp costs 0 cycles.

Main Sources of SM Under-utilisation:

  • Structural HazardLimited SFUs, double-precision multiplication — only 4 SFUs for 32-thread warp
  • Branch DivergenceBranches within a warp cause serial execution of paths
  • Memory LatencyOff-chip DRAM: 100–150× slower than registers — warp stalls and is swapped out
Optimal Block SizeBlock size > total CUDA cores per SM; must be multiple of 32 (warp size) to achieve near-optimal SM utilisation.

CUDA Core Details (Fermi)

  • Each CUDA core = INT unit pipeline + FP unit pipeline (only one active per cycle)
  • INT: 32-bit, 64-bit integer and logic/bitwise operations
  • FP unit: single-precision FP (FP32); double-precision (FP64) requires 2 cores
  • Fermi upgraded to IEEE 754-2008 (was 754-1985) — adds FMA (Fused Multiply-Add): single rounding at end, better accuracy
  • FMA compresses to 1 cycle: 32 FP32 FMA/cycle/SM or 16 FP64 FMA/cycle/SM
  • Kepler fix: added dedicated DP units per SM to avoid ÷2 penalty on FP64
Max Registers per Thread (Fermi)32k × 32-bit / SM ÷ max active warps per SM ÷ 32 threads/warp = 64 registers per thread (CUDA Compute Capability 2.x). Exceeding spills to L1→L2→local DRAM.

Shared Memory: The Key Optimisation

Why Shared Memory?Off-chip global memory is 100–150× slower. Programmers explicitly load data into per-SM shared memory to avoid repeated off-chip reads — unlike CPU where cache management is automatic.

THREE WAYS SHARED MEMORY ACCELERATES CODE

  1. Reuse — load data once from global to shared; all threads in block read shared (e.g. matrix-matrix multiplication tiling)
  2. Coalescing helper — select threads load global→shared to eliminate redundant reads; other threads read from shared
  3. Coalesced access — programmer arranges global accesses to be coalesced (sequential addresses) for maximum bandwidth
Synchronisation Required!After writing to shared memory, all threads must call __syncthreads() before any thread reads — prevents RAW hazards.

Shared data lifetime = block lifetime. Once all threads of a block complete, shared memory is invalid.

§19.4 Intel Gen8 GPU (Integrated Graphics)

Execution Unit (EU)

7
SMT Threads/EU
2
SIMD FPUs/EU
128
GRF Regs/Thread
  • EU = SMT architecture with 7 simultaneous threads; superscalar pipeline design
  • 2 SIMD Floating-Point Units — each performs simultaneous add + multiply per cycle (FMA)
  • Branch unit — dedicated to branch instructions
  • Send unit — dedicated to memory operations
  • Each thread has 128 general-purpose registers (GRF), each storing 32 bytes (SIMD 8-element vector of 32-bit)
  • Total GRF per EU: 7 threads × 128 regs × 32 bytes = 28 kB per EU
  • Can issue up to 4 different instructions simultaneously from different threads
  • Thread arbiter dispatches instructions to functional units
  • Per-thread architectural state in dedicated ARF (Architecture Register File)

Subslice — 8 EUs

Simultaneous Threads
  • Contains 8 EUs + local thread dispatcher unit + dedicated instruction caches
  • Sampler unit — texture & image surface sampling with local L1+L2 sampler cache; supports bilinear, trilinear, anisotropic filtering; fixed-function compression/decompression
  • Data port — efficient R/W operations, attempts to consolidate reads from different threads to cache line size
  • Each subslice = 8 EUs × 7 threads = 56 simultaneous hardware threads with dedicated resources

§19.5 When to Use a GPU as a Coprocessor

GPU Candidate Workload Criteria

Ideal GPU CandidateCode that is massively data-parallel — the same computation on thousands of elements concurrently, with no/minimal data dependency between iterations.

CHECKLIST: IS YOUR CODE GPU-SUITABLE?

  • ✓ ParallelLarge iteration loops performing the same calculation on different data elements (e.g. matrix arithmetic)
  • ✓ No DependencyNo data dependency between loop iterations — iterations are independent
  • ✓ Significant FractionParallelisable section makes up large % of total runtime (Amdahl's Law — maximise f)
  • ✓ Lightweight ThreadsEach thread = small snippet of code (few branches, little state)
  • ✗ Control-HeavyLots of branching, conditional logic, pointer chasing → poor fit
  • ✗ Serial Critical PathIf serial fraction is large, GPU offloading gives minimal speedup
  • ⚠ Transfer CostCPU↔GPU PCIe transfer time must be small relative to GPU computation time
Profiling StepUse gprof (GNU) or nvprof (NVIDIA visual) to identify hotspots before attempting GPU acceleration. Profile on representative input data.

Key Terms Quick Reference

CUDA / Programming Terms

  • CUDA Compute Unified Device Architecture — NVIDIA GPU programming model
  • Kernel Data-parallel function executed on GPU; launched by host
  • Thread Single instance of kernel = one CUDA core assignment
  • Block Group of threads assigned to one SM; up to 1024 threads
  • Grid All blocks launched in one kernel call
  • Warp 32 consecutive threads; fundamental HW scheduling unit
  • SIMT Single Instruction Multiple Threads — all threads in warp execute same instruction
  • GPGPU General-Purpose Computing using GPU — non-graphics workloads on GPU

GPU Hardware Terms

  • SM Streaming Multiprocessor — contains 32 CUDA cores + warp scheduler + shared mem
  • CUDA Core GPU processor core; each has INT + FP pipeline
  • SFU Special Function Unit — sin, cos, reciprocal, sqrt in 1 cycle; 4/SM
  • GigaThread Global thread block scheduler distributing blocks to SM warp schedulers
  • GDDR5 Graphics DDR5 — high-bandwidth off-chip memory (384-bit bus in Fermi)
  • EU Execution Unit — Intel Gen8 equivalent of SM; 7-thread SMT, 2 SIMD FPUs
  • Subslice Intel Gen8: 8 EUs + dispatcher + sampler + data port
  • GTI Graphics Technology Interface — connects Gen8 GPU to SoC ring

Performance & Architecture Terms

  • FMA Fused Multiply-Add — single-rounding combined ×+ op; IEEE 754-2008
  • Warp Divergence Branches in warp force serial execution of different paths
  • Register Spill Thread uses more registers than available → overflow to L1→local DRAM
  • Memory Coalescing Adjacent threads access consecutive addresses → served in 1 transaction
  • Bank Conflict Two addresses in same memory bank → serialised access; n-way conflict = n× throughput loss
  • Occupancy Ratio of active warps to max warps per SM — key performance metric
  • Roofline Model Visualises performance limit: compute-bound (FLOPs peak) or memory-bound (bandwidth peak)
