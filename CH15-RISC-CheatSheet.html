§15.1 Historical Motivation & Semantic Gap

The semantic gap = difference between operations in HLLs and those in hardware. CISC architects tried to close this gap with more instructions, more addressing modes, and HLL constructs in hardware.

CISC Design Rationale
  • Ease the compiler writer's task
  • Improve execution efficiency via microcode
  • Support more complex and sophisticated HLLs
  • Smaller program size → fewer memory accesses
The Problem
Compilers tend to favor simpler instructions anyway. Complex instructions rarely matched exactly, so CISC complexity often went unused. Studies showed RISC I code was comparable in size to VAX/PDP-11/M68000 code.
RISC Counter-Argument
A simpler architecture supports more effective optimizing compilers and enables faster hardwired execution of the instructions that actually dominate real programs.

§15.1 Procedure Calls & Design Implications

Tanenbaum / Berkeley RISC Findings [TANE78, KATE83]
  • 98% of dynamically called procedures passed < 6 arguments
  • 92% of procedures used < 6 local scalar variables
  • Programs stay within a narrow window of procedure depth
  • References to local variables are highly localized
Three Design Implications for RISC
  1. Large registers or compiler optimization: Keep local scalars in registers. Minimize register–memory operations by exploiting locality.
  2. Efficient instruction pipelines: High proportion of branches/calls makes naïve pipelining inefficient. Prefetch often wasted.
  3. Simple, high-performance instruction primitives: Predictable cost in time, code size, and energy. Hardwired, no microcode.
Bottom Line
Supporting HLLs means optimizing for what programs actually do — not closing the semantic gap with hardware complexity.

§15.2 Register Windows — Concept

Instead of saving/restoring registers on every procedure call, the processor switches to a different fixed-size window of physical registers. Adjacent windows overlap to pass parameters without data movement.

Window Structure (per procedure level)
┌─────────────────────────────┐ │ PARAMETER registers │ ← Passed in from caller │ (= caller's TEMP regs) │ (physical overlap) ├─────────────────────────────┤ │ LOCAL registers │ ← Private to this procedure │ │ ├─────────────────────────────┤ │ TEMPORARY registers │ ← Passed out to callee │ (= callee's PARAM regs) │ (physical overlap) └─────────────────────────────┘
Key Mechanism
Temporary registers at level J are physically identical to parameter registers at level J+1. Parameters are passed by a CWP pointer increment — zero data movement needed.
Berkeley RISC Numbers
  • 8 windows × 16 registers each = 128 physical registers
  • Pyramid: 16 windows × 32 registers = 512 physical registers
  • With 8 windows: save/restore needed on only ~1% of calls [TAMI83]

§15.2 Circular Buffer & CWP/SWP

Since procedure depth fluctuates in a narrow range, register windows are organized as a circular buffer. Only the most recent N–1 activations are held in registers.

Pointer Definitions
PointerMeaning
CWPCurrent Window Pointer — points to the window of the currently active procedure. Register references are offset by CWP.
SWPSaved Window Pointer — points to the window most recently saved to memory.
Overflow/Underflow Handling
Overflow (too many calls)
When CWP is incremented and becomes equal to SWP, an interrupt fires. The oldest window (param + local portions only) is saved to memory. SWP advances.
Underflow (too many returns)
When CWP is decremented and becomes equal to SWP, an interrupt fires. The saved window is restored from memory. SWP decrements.
N-Window Buffer Capacity
N-window register file holds N−1 procedure activations Only param+local portions saved (NOT temporaries) Temporaries are callee's params — callee saves them if needed

§15.2 Global Variables & Register File vs Cache

Two Options for Global Variables
  • Memory-based: Compiler assigns globals to memory locations. Simple but slow for frequently accessed globals.
  • Global register set: Fixed set of registers (e.g., r0–r7) available to all procedures. Registers 8–31 are window-relative. Linker assigns hot globals to global registers.

Register File Advantage
For local scalars, window-based register file is clearly superior: simple decode, no set-associative tag comparison, no block transfer overhead.

§15.3 Compiler-Based Register Optimization

When only a small number (16–32) of registers is available, the compiler must optimize register allocation. HLL programs use symbolic variables; the compiler maps them to physical registers.

Graph Coloring Algorithm [CHAI82]
  1. Assign each program quantity a symbolic/virtual register
  2. Analyze program to build a register interference graph: nodes = symbolic registers; edge between two nodes if both are live at the same time
  3. Attempt to color the graph with n colors (n = number of physical registers), such that no two adjacent nodes share a color
  4. Nodes sharing the same color → assigned to the same physical register
  5. Nodes that cannot be colored → assigned to memory; compiler inserts LOAD/STORE as needed (spilling)
Example: 6 Symbolic → 3 Physical Registers
Symbolic: A B C D E F (6 vars) Physical: R1 R2 R3 (3 available) A ──── time ──────────── B ───────── time ─────── C ──── time ──── D time ──── E time ───────── F ──── time ──────────── ← spilled (conflicts all) Coloring: A,D → R1 | B → R2 | C,E → R3 | F → memory
Research Finding [BRAD91a]
With reasonably sophisticated optimization, marginal improvement beyond 32 registers. Little benefit beyond 64 even with simple optimization.

§15.4 RISC Architecture Characteristics

Core RISC Properties
  • One instruction per cycle: Machine cycle = fetch 2 reg operands + ALU + store result. RISC instructions ≈ microinstruction speed. No microcode needed → hardwired control.
  • Register-to-register operations: Only LOAD/STORE access memory. Simplifies instruction set and control unit. E.g., RISC has 1–2 ADD variants; VAX has 25.
  • Simple addressing modes: Primarily register addressing + displacement + PC-relative. Complex modes synthesized in software.
  • Simple instruction formats: Fixed length (typically 4 bytes), aligned on word boundaries. Fixed opcode field position. Opcode decode and register operand access can occur in parallel.
  • Large register file or compiler optimization: Minimize register–memory traffic.
  • Load/store architecture: All computation on registers; memory access only via explicit LOAD/STORE.
Benefits of Simple Instruction Formats
  • Fixed fields → simultaneous opcode decode + register access
  • Word-length fetch → optimized instruction fetching
  • Word-aligned → no instruction crosses page boundary
  • Simplified control unit → faster execution
Hardwired vs Microprogrammed
RISC instructions are simple enough to be hardwired directly. No need to access a microprogram control store → eliminates control store access latency on every instruction.

§15.5 RISC Pipeline Stages

Instruction Types & Their Stages
Instruction TypeStagesDescription
Register–RegisterI → EI: Fetch. E: ALU op on reg inputs, store to reg
Load / StoreI → E → DI: Fetch. E: Calculate mem address. D: Mem access
BranchI → EI: Fetch. E: Evaluate condition, update PC
4-Stage Pipeline (E split into E1+E2)
Stages: I | E1 | E2 | D I: Instruction fetch E1: Register file read E2: ALU operation + register write D: Data memory access (loads/stores) Up to 4 instructions in flight simultaneously → max 4× speedup
Pipeline Hazards
Data dependency: Instruction needs operand modified by preceding instruction → delay (NOOP). Branch hazard: Branch updates PC → fetched instructions may be wrong → pipeline flush or NOOP.
Pipeline Timing (sequential vs pipelined)
Sequential: 2N + D stages (N instr, D mem accesses) 2-stage: ~N + D stages (with wait states for 1-port mem) 3-stage: ~(N + D)/2 + branch penalty 4-stage: ~N/4 + penalty (data + branch delays)

§15.5 Delayed Branch & Delayed Load

Delayed Branch Concept

A delayed branch does not take effect until after the execution of the following instruction. The slot immediately after the branch is the delay slot.

Table 15.8 — Normal vs Delayed Branch (Optimized)
AddrNormal Branch+ NOOP (RISC)Optimized Delayed
100LOAD X, rALOAD X, rALOAD X, rA
101ADD 1, rAADD 1, rAJUMP 105
102JUMP 105JUMP 106ADD 1, rA ← delay slot
103ADD rA, rBNOOPADD rA, rB
105STORE rA, ZSTORE rA, ZSTORE rA, Z
How Optimization Works
Move the ADD (which comes before the JUMP logically) into the delay slot. ADD executes while JUMP is resolving. Result: 2 fewer clock cycles. No special pipeline-flush circuitry needed. Berkeley RISC / IBM 801 fill delay slots successfully 70–90% of the time.
Delayed Load

The target register of a LOAD is locked. Processor continues until that register is needed, then idles if load not complete. Compiler rearranges instructions to do useful work in the load latency window.

Loop Unrolling [BACO94]
  • Replicate loop body u times; iterate by step u
  • Reduces loop overhead (fewer test+branch)
  • Increases instruction parallelism (fills pipeline slots)
  • Improves register/data cache/TLB locality
Conditional Branch Caution
For conditional branches, instruction before/after cannot blindly be swapped if the condition is modified by that instruction. Compiler must verify no dependency, or insert NOOP.

§15.6 MIPS R4000 — Architecture & Instruction Set

Overview
  • Developed by MIPS Technology Inc. (inspired by Stanford MIPS [HENN84])
  • 64-bit architecture: all internal/external paths, addresses, registers, ALU (vs R2000/R3000 which were 32-bit)
  • 32 × 64-bit general-purpose registers
  • Up to 128 KB on-chip cache (split: instruction + data)
  • Chip partitioned: CPU core + memory management coprocessor
  • No condition codes — conditions stored in GP registers (simplifies pipelining and register optimization)
Instruction Set Principles
  • All instructions: single 32-bit fixed format
  • All data operations: register to register only
  • Memory access: pure LOAD / STORE only
  • Only addressing mode: 16-bit offset from a base register
  • r0 always contains 0 (useful for synthesizing operations)

§15.7 SPARC — Register Windows

SPARC (Scalable Processor Architecture) — developed by Sun Microsystems, directly inspired by Berkeley RISC I.

Register Window Layout
  • Each window: 32 logical registers (r0–r31)
  • Physical registers: 8 global (r0–r7) + 8 windows × 24 window-specific = 136 total (for 8-window impl.)
  • In registers (r24–r31 / logical): Passed in from calling procedure. Physically overlapped with caller's out registers.
  • Local registers (r16–r23): Private to current procedure, not shared.
  • Out registers (r8–r15): Passed out to called procedure. Physically overlapped with callee's in registers.
  • Global registers (r0–r7): Accessible from all windows.
  • r0 hardwired to 0 (useful for synthesis)

§15.8 RISC vs CISC Controversy

Two Categories of Assessment
Quantitative
Compares program size and execution speed on comparable technology. Mostly done by RISC proponents [PATT82b, HEAT84, PATT84] — generally favorable to RISC.
Qualitative
Examines HLL support quality and VLSI real-estate efficiency. Inherently subjective. Results ambiguous and disputed from both sides [COLW85a, WALL85, PATT85b].
Problems with RISC/CISC Comparison
  • No pair with comparable technology, cost, compiler sophistication, and OS support
  • No definitive test program set; performance varies by workload
  • Hard to separate hardware effects from compiler quality effects
  • Most RISC studies used "toy" machines, not commercial products
  • Commercial "RISC" machines often contain CISC features (not pure)

Modern Convergence (1990s onward)
RISC machines grew more complex (PowerPC). CISC machines adopted RISC techniques: deeper register files, aggressive pipelining, RISC-like micro-ops (Pentium Pro+). The controversy has largely been rendered moot by architectural convergence.

RISC Performance Arguments
  • Simple instructions → hardwired, no control store latency
  • Simpler control unit → more chip area for cache and ALU
  • Fixed formats → parallel decode + register fetch
  • Load/store architecture → straightforward pipelining
  • More GP registers → less memory traffic

  • Faster interrupt response (check between elementary ops)

Exam Topics & Tricky Points

Why temporary registers aren't saved on window overflow
The temporary registers of procedure A are physically identical to the parameter registers of procedure B. If B saves its in-registers (which are A's temporaries), those values are preserved by B — A doesn't need to save them separately.
N windows ≠ N procedure depths
An N-window file can hold only N−1 activations. The Nth window overlaps with the oldest, causing overflow when it would be overwritten.
Why delayed branch works for unconditional but not blindly for conditional
Unconditional branches always jump → instruction before branch can always go in delay slot. Conditional branches may or may not jump → instruction moved to delay slot must not depend on (or modify) the condition being tested.
RISC pipeline efficiency formula trap
"Sequential: 2N + D" counts 2 cycles per instruction (fetch + execute) plus D extra cycles for memory data access. Pipelined versions overlap these, but data dependencies and branches subtract from ideal speedup.
CISC code size advantage overstated
Studies show RISC code size is comparable to or only slightly larger than CISC. Key reason: CISC uses longer opcodes (more instructions → more opcode bits), and RISC favors register ops (fewer bits than memory addresses).
SPARC r0 = 0 synthesis tricks
MOV src, dst → OR src, %r0, dst
CLR dst → OR %r0, %r0, dst
NEG dst → SUB %r0, src, dst
COMPARE s1, s2 → SUBcc s1, s2, %r0 (set flags, discard result)
MIPS: no condition codes → no special pipeline logic
Conditions stored in GP registers means the same dependency-tracking hardware used for data dependencies handles condition values automatically. No special condition-code forwarding needed.
