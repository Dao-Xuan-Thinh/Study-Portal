CPU: Central Processing Unit — §5.1

The CPU is the brain of the computer — it fetches, decodes, and executes instructions

The CPU has three main internal components that work together every clock cycle to process program instructions.

Three Main Components
  • ALU (Arithmetic Logic Unit): performs integer arithmetic, bitwise logic, and comparisons on operands fetched from registers
  • Control Unit (CU): decodes instructions, generates control signals, coordinates data flow between all components
  • Registers: small, ultra-fast storage directly inside the CPU — the only data the ALU can operate on directly
Register Types
  • Data registers: hold operands and results (e.g., general-purpose registers R0–R15, EAX–EDX in x86)
  • Address registers: hold memory addresses (SP = stack pointer, BP = base pointer, SI/DI = index registers)
  • Status register (flags): condition codes set by ALU — zero flag (Z), carry (C), overflow (V), sign/negative (N)
  • Program counter (PC): holds the address of the next instruction to fetch; auto-incremented after each fetch
  • Instruction register (IR): holds the current instruction being decoded and executed by the CU
  • Memory address register (MAR): address output to memory bus — set before any read or write operation
  • Memory data register (MDR): bidirectional buffer — holds data read from memory or data about to be written

Condition Flag Meanings
  • Z (Zero): set when result = 0; used for equality comparison after CMP
  • N (Negative / Sign): set when MSB of result = 1; indicates negative in two's complement
  • C (Carry): set when unsigned addition overflows or subtraction borrows from MSB
  • V (Overflow): set when signed result overflows the word size (e.g., 127 + 1 in 8-bit)
Control Unit Design Approaches
  • Hardwired CU: combinational logic gates directly implement control signals — extremely fast, inflexible, common in RISC
  • Microprogrammed CU: a ROM/PLA maps opcodes to sequences of micro-operations — easier to design/update; used in CISC (x86 internally translates complex instructions to micro-ops)

Main Memory — §5.2

Main memory (RAM) stores both programs and data currently in use — volatile

Memory is modelled as a large array of bytes. Every byte has a unique numeric address starting at 0. The CPU accesses memory through MAR (address) and MDR (data).

Address Space
  • n address bits → 2ⁿ addressable locations (each 1 byte wide)
  • 32-bit addresses → 2³² = 4,294,967,296 bytes = 4 GB address space
  • 64-bit addresses → 2⁶⁴ = 16 exabytes theoretical address space
  • Physical memory installed may be much less than the maximum address space
Address space = 2^(address bus width) bytes
Memory Access Protocol
  • Read: CPU puts address in MAR → memory reads that location and puts value in MDR → CPU reads MDR
  • Write: CPU puts address in MAR and data in MDR → memory stores MDR value at address in MAR
Memory Types
  • RAM (Random Access Memory): volatile; read + write; DRAM used for main memory, SRAM used for cache
  • ROM (Read-Only Memory): non-volatile; factory-programmed; used for BIOS, firmware, boot code
  • EEPROM / Flash: electrically erasable; non-volatile; used for SSDs, USB drives, firmware updates
  • Cache (L1/L2/L3): small, fast SRAM on or near the CPU die; exploits temporal and spatial locality

Endianness
  • Big-endian: most-significant byte at the lowest address — SPARC, IBM mainframes, network byte order (TCP/IP)
  • Little-endian: least-significant byte at the lowest address — x86, x86-64; ARM (configurable)
  • Matters when exchanging binary data between systems or casting byte pointers to multi-byte types
Locality of Reference
Caches work because programs exhibit temporal locality (recently used data used again soon) and spatial locality (data near recently-used data accessed soon). This is why a small L1 cache dramatically improves average memory access time.

I/O Subsystem — §5.3

I/O devices connect the computer to the external world — keyboards, displays, disks, network
I/O Device Categories
  • Input only: keyboard, mouse, scanner, microphone, barcode reader, camera
  • Output only: monitor/display, printer, speakers, projector
  • Both input and output: touchscreen, disk drive, network interface card, USB flash drive, modem
I/O Controllers

Each I/O device has a dedicated controller chip. The CPU communicates only with the controller, not the device directly. The controller handles timing, buffering, format conversion, and error detection.

  • Controller registers: data register (transfer buffer), status register (ready/busy/error), control register (commands)
  • CPU reads status register to check if device is ready; writes command register to initiate operations
I/O Addressing
  • Memory-mapped I/O: controller registers appear at specific memory addresses — CPU uses normal LOAD/STORE instructions to access them (used by ARM, RISC-V)
  • Port-mapped I/O (isolated I/O): separate I/O address space with dedicated IN/OUT instructions — used by x86
Three I/O Methods
  • Programmed I/O (polling): CPU executes a busy-wait loop reading the device status register — wastes CPU cycles, but simple
  • Interrupt-driven I/O: device signals the CPU when it is ready; CPU handles other tasks in the meantime — efficient for slow devices (keyboard, serial)
  • DMA (Direct Memory Access): DMA controller autonomously transfers blocks of data between device and memory; only interrupts CPU when entire transfer finishes — ideal for disks, network, GPU
I/O Method Comparison
MethodCPU InvolvementEfficiency
Polling100% busy-waitPoor — CPU wasted on checking
Interrupt-drivenOnly at completionGood — CPU free between I/O events
DMAOnly at start and endBest — for bulk transfers
Interrupt Mechanism Steps
  1. Device finishes operation and asserts interrupt request line (IRQ)
  2. CPU finishes current instruction, then checks interrupt flag
  3. CPU saves registers (PC, flags) on the stack — preserves context
  4. CPU jumps to the Interrupt Service Routine (ISR) address from the interrupt vector table
  5. ISR services the device (reads data, clears interrupt flag)
  6. CPU restores saved registers (IRET) and resumes original program
Modern Systems
Modern systems use interrupts and DMA together: the DMA controller handles the bulk data transfer to RAM while the CPU runs other processes, then fires an interrupt to notify the CPU that the transfer is complete.

Subsystem Interconnection: The Bus — §5.4

A bus is a shared communication pathway connecting CPU, memory, and I/O

The bus is the fundamental interconnect in classical computer designs. Being shared means only one device may transmit at a time — devices must arbitrate for ownership.

Three Types of Bus Lines
  • Data bus: carries actual data — bidirectional; width = word size (8/16/32/64-bit); wider = more data per cycle
  • Address bus: carries memory or I/O address — unidirectional (CPU → memory/I/O); width determines maximum addressable space
  • Control bus: carries control signals — read/write enable, interrupt request, DMA request, bus grant, clock, reset
Bus Bandwidth
Bus bandwidth (bytes/s) = bus width (bytes) × bus clock frequency

Example: 64-bit bus at 100 MHz = 8 bytes × 100,000,000 = 800 MB/s

Synchronous vs Asynchronous Bus
  • Synchronous bus: all events clocked by a shared bus clock — simple, fast, but all devices must match the clock speed; prone to clock skew on long buses
  • Asynchronous bus: uses handshaking signals (request/acknowledge) instead of a shared clock — more flexible, works with mixed-speed devices; slightly slower overhead
Bus Arbitration (Multiple Masters)
  • Daisy-chain: bus grant signal passes through devices in series — simple wiring, priority = physical position; far devices get low priority
  • Centralized parallel: dedicated arbiter chip; each device has its own bus request line to the arbiter — fair priority schemes possible
  • Distributed (self-selection): each device monitors bus and applies its own priority logic — no central arbiter; higher complexity

Von Neumann Bottleneck
A single shared bus between CPU and memory limits throughput — especially as CPUs get faster while memory latency stays high. The memory wall is a fundamental constraint: even a 4 GHz CPU stalls for ~400 cycles waiting for a DRAM access (~100 ns).
Modern Alternatives
  • PCIe: point-to-point serial lanes; each device has a dedicated link to a root complex — eliminates bus contention
  • HyperTransport / QPI (QuickPath): CPU-to-CPU and CPU-to-memory links for multi-socket systems
  • AXI / AMBA: on-chip bus protocol used inside ARM-based SoCs for connecting CPU cores, peripherals, and memory controllers

Program Execution: Fetch-Decode-Execute — §5.5

The CPU repeatedly cycles through fetch → decode → execute until the program ends
Detailed Cycle Steps
  1. Fetch: PC → MAR; memory[MAR] → MDR → IR; PC is incremented to point to the next instruction in sequence
  2. Decode: Control unit reads the opcode field of IR; determines the operation type, addressing modes, and identifies source/destination operands
  3. Evaluate operands: if operands are in memory, calculate effective addresses and perform additional memory reads (indirect addressing adds extra cycles)
  4. Execute: ALU performs the operation on the operands; result is stored in destination register or written to memory via MAR/MDR; condition flags updated
  5. Repeat from step 1 — or handle interrupt if interrupt flag set and interrupts enabled
Clock and Performance
  • Each stage takes one or more clock cycles — the clock is the heartbeat of the CPU
  • CPI (Cycles Per Instruction): metric of how many cycles each instruction uses on average
  • RISC architectures target CPI = 1 for most instructions; complex CISC instructions may take 10–100+ cycles
CPU time = Instruction Count × CPI × Clock Period
CPU time = (IC × CPI) / Clock Frequency
Pipelining — The Assembly Line Analogy

Pipelining overlaps execution of multiple instructions in different stages simultaneously. Throughput approaches 1 instruction/cycle even if each instruction takes 5 cycles.

  • Classic 5-stage RISC pipeline: IF (Instruction Fetch) → ID (Instruction Decode) → EX (Execute) → MEM (Memory access) → WB (Write-Back)
  • At any instant, 5 different instructions are in different pipeline stages simultaneously
Pipeline Hazards
Hazard TypeCauseSolution
Data hazard (RAW)Instruction reads register written by previous not-yet-finished instructionForwarding (bypassing) or pipeline stall (bubble)
Control hazardBranch instruction — next instruction unknown until branch resolvesBranch prediction; delayed branch slots
Structural hazardTwo instructions need the same hardware resource simultaneouslyResource duplication or stall
Modern CPU Enhancements
Modern high-end CPUs execute multiple instructions per cycle (superscalar — 4–8 way), reorder instructions dynamically (out-of-order execution, Tomasulo algorithm), and predict branches with >95% accuracy to keep deep pipelines (20+ stages) filled at all times.

Different CPU Architectures — §5.6

Different architectures make different trade-offs in ISA design, register count, and memory access
CISC — Complex Instruction Set Computer
  • Many complex instructions, variable-length encoding, rich addressing modes
  • Can perform memory-to-memory operations in a single instruction (e.g., ADD [mem1], [mem2])
  • Historically designed to close the "semantic gap" between high-level languages and hardware
  • Examples: x86, x86-64 (Intel Core, AMD Ryzen, Xeon, EPYC)
  • Pros: powerful instructions reduce code density; Cons: complex decoder, variable-length makes pipelining harder
RISC — Reduced Instruction Set Computer
  • Small number of simple instructions, fixed-length encoding (uniformly 32-bit), load/store architecture
  • Only LOAD and STORE access memory; ALL other operations work only on register operands
  • Large register file (16–32+ registers) reduces memory traffic
  • Examples: ARM, MIPS, RISC-V, PowerPC, Apple Silicon (M-series)
  • Pros: simple uniform pipeline, high clock speed, compiler-friendly; Cons: more instructions needed for same task, more registers used
CISC vs RISC Feature Comparison
FeatureCISC (x86-64)RISC (ARM64)
Instruction countFewer, more complexMore, simpler
Instruction lengthVariable (1–15 bytes)Fixed (4 bytes)
Memory accessMany ops can access memoryLoad/store only
Register count8 (x86-32) / 16 (x86-64)31 general-purpose

Harvard vs Von Neumann Architecture
  • Von Neumann: single unified memory holds both instructions and data; one bus — simpler but creates instruction/data fetch conflict; used by all general-purpose CPUs
  • Harvard: physically separate instruction memory and data memory with separate buses — allows simultaneous instruction fetch and data access; used in microcontrollers (PIC, AVR), DSPs
  • Modified Harvard: shared DRAM but separate L1 instruction cache and L1 data cache — modern CPUs use this; gets most of Harvard's benefit with Von Neumann simplicity
Modern x86 Reality
Modern x86 processors are CISC externally (for backward compatibility) but translate CISC instructions into internal RISC-like micro-operations before execution. The pipeline itself is essentially RISC inside — best of both worlds.

A Simple Computer Example — §5.7 (Forouzan's Model)

A simple conceptual CPU with 16 instructions illustrates how real processors work
Forouzan's Simple Computer Specs
  • 16 instructions — 4-bit opcode field (opcodes 0–15)
  • 256 bytes of memory — 8-bit addresses (0x00–0xFF)
  • 16 general-purpose registers — R0 through R15, each 4-bit register ID
  • 16-bit instruction word — exactly one 16-bit word per instruction
  • 16-bit data words — each register holds 16 bits
Instruction Format
[opcode 4-bit | operand 12-bit]

The 12-bit operand field encodes register IDs (4-bit each), memory addresses (8-bit), or immediate values depending on the instruction type. E.g., a 3-register instruction uses three 4-bit fields.

Scaling to Real CPUs
This simple model captures the essence of all real processors. The difference is scale: x86-64 has ~2000+ instructions, 16 visible + hundreds of internal registers, 20+ pipeline stages, out-of-order execution with 200+ instruction windows, and branch prediction at >95% accuracy.

Chapter 5 Big Picture
A computer system = CPU + Main Memory + I/O Subsystem, linked by buses. The CPU fetch-decode-execute cycle drives all computation. Architectural choices (RISC vs CISC, Harvard vs Von Neumann, bus design, cache strategy) reflect fundamental trade-offs between cost, speed, power, and programmability.

Cache Memory — Deep Dive

Cache exploits locality: recently-used and nearby data are likely to be used again soon

Cache sits between CPU registers and main DRAM. Because DRAM latency (~100 ns) is ~100× slower than the CPU clock cycle (~0.25 ns at 4 GHz), a small fast cache dramatically reduces average memory access time.

Two Types of Locality
  • Temporal locality: a memory location accessed now will likely be accessed again soon (loops, frequently-called functions)
  • Spatial locality: if address X is accessed, addresses near X (X+1, X+2 …) are likely to be accessed soon (arrays, sequential code)
Cache Hit vs Miss
  • Cache hit: requested data is found in cache → fast access (~4 ns for L1)
  • Cache miss: data not in cache → must fetch from next cache level or DRAM → slow access
  • On a miss, an entire cache line (typically 64 bytes) is loaded from DRAM — exploits spatial locality
Average Access Time = Hit Rate × T_cache + (1 − Hit Rate) × T_DRAM
Cache Mapping Strategies
StrategyHow It WorksPros / Cons
Direct-mappedEach memory block maps to exactly one cache line (addr mod cache_size)Simple, fast lookup; suffers thrashing with conflicting accesses
Fully associativeAny block can go into any cache lineNo thrashing; complex search logic (CAM hardware); expensive
N-way set-associativeCache divided into sets; each block maps to one set but any of N lines within that setBest trade-off; most common (L1: 4–8 way, L2: 8–16 way)
Cache Replacement Policies
  • LRU (Least Recently Used): evict the line that has not been accessed for the longest time — best hit rate, hardware overhead to track order
  • FIFO (First In First Out): evict the oldest loaded line regardless of access pattern — simple
  • Random: evict a randomly chosen line — simple hardware, surprisingly competitive with LRU
Write Policies
  • Write-through: every write to cache also writes immediately to DRAM — memory always consistent; high bus traffic
  • Write-back: write to cache only; dirty line written to DRAM only when evicted — less bus traffic; requires dirty bit; main cache design used today

Performance Analysis & Key Metrics

Performance = how fast a computer executes programs — measured by execution time or throughput
CPU Performance Equation
CPU Time = IC × CPI × (1 / f)
  • IC (Instruction Count): total instructions executed — determined by algorithm + ISA + compiler
  • CPI (Cycles Per Instruction): hardware efficiency — improved by pipelining, caches, out-of-order execution
  • f (Clock frequency): CPU speed in Hz — limited by heat, power, and signal propagation
  • To improve performance: reduce any one of IC, CPI, or (1/f) — but they are interdependent
Amdahl's Law — Limits of Speedup
Speedup = 1 / ((1 − P) + P / S)
  • P = fraction of program that can be parallelized / improved
  • S = speedup factor applied to that fraction
  • The serial fraction (1 − P) is the bottleneck — limits overall gain even with infinite speedup on P
  • Example: if 90% of code can be sped up 10×, max speedup = 1 / (0.1 + 0.9/10) = 5.26×
Bandwidth vs Latency
MetricDefinitionMeasured In
LatencyTime for one operation to complete (request → response)nanoseconds (ns)
BandwidthMaximum data transfer rate (throughput)GB/s, MB/s
ThroughputOperations completed per unit timeMIPS, GFLOPS
MIPS and FLOPS
  • MIPS (Millions of Instructions Per Second): MIPS = f / (CPI × 10⁶) — misleading across different ISAs (more MIPS ≠ more useful work)
  • FLOPS (Floating-Point Operations Per Second): measures floating-point throughput; relevant for scientific/ML workloads
  • Modern CPUs: ~hundreds of GFLOPS (single-precision); modern GPUs: tens of TFLOPS
Factors Affecting Real Performance
  • Memory latency: cache misses cause CPU to stall — often the dominant bottleneck in data-intensive programs
  • Branch misprediction: incorrect speculative execution must be flushed — costs 15–20 cycles per misprediction on deep pipelines
  • I/O bound vs CPU bound: a program limited by disk/network speed will not benefit from a faster CPU
  • Compiler optimization: good compilers can dramatically reduce IC and exploit instruction-level parallelism
Rule of Thumb
Measure before optimizing. Profile the program to find the actual bottleneck (CPU, memory, I/O, branch) before applying architecture-level optimizations — Amdahl's Law ensures only the bottleneck fraction matters.
