§14.1 Processor Organization

The processor must: fetch instructions, interpret/decode, fetch data, process data (ALU), write results. Requires small internal memory — registers — to hold state between steps.

Key Internal Components (Fig 14.1 / 14.2)
  • ALU: performs all arithmetic and logic operations on data
  • Control Unit (CU): directs movement of data & instructions; controls ALU operation
  • Registers: top of memory hierarchy — above cache; small and fast
  • Internal processor bus: moves data between registers and ALU (ALU only works on data in registers)
  • System bus: connects processor to memory and I/O via Control / Data / Address buses
ALU Elements (Fig 14.2)
  • Complementer (subtraction support)  |  Arithmetic & Boolean logic unit
  • Shifter  |  Status flags (carry, zero, overflow, sign)
Structural Analogy
Processor internal structure mirrors the whole computer: {CU, ALU, registers} connected by buses — just as the computer has {processor, memory, I/O} connected by a system bus.

§14.2 User-Visible Registers

Registers accessible to the programmer via machine-language instructions. Four categories:

  • General-purpose: may hold data or addresses; orthogonal = any register with any opcode; some architectures impose restrictions (dedicated FP, stack regs)
  • Data registers: hold data only; cannot be used for address computation
  • Address registers: hold addresses — includes index registers, segment base registers, stack pointers
  • Condition codes (flags): bits set by hardware after operations; read implicitly; programmer cannot write directly; typically part of the PSW control register
Optimal Register Count
Studies show 8–32 user-visible registers is optimal. Fewer → more memory references. More → diminishing returns on reference reduction. RISC uses hundreds (Ch. 15).
Register Length Constraints
  • Address registers: must be ≥ largest possible address
  • Data registers: wide enough for most data types; two adjacent regs may hold double-width values
MC68000 vs Intel 8086 (Fig 14.3)
  • MC68000: clean split — 8 data regs (D0–D7) + 9 address regs (A0–A7); orthogonal; two A7 SPs (user/OS)
  • 8086: every register special-purpose; AX=accumulator, CX=count, BX=base, DX=data; compact encoding but less flexible
  • 80386: 32-bit widening of 8086 embedded original layout for backward compatibility

§14.4 Pipeline Performance Formulas

Cycle Time
t = max[tᵢ] + d  = tₘ + d

tᵢ = delay of stage i  |  tₘ = max stage delay  |  d = latch delay (tₘ ≫ d)

Total Execution Time — k stages, n instructions
T(k,n) = [k + (n − 1)] × t
  • First instruction: k cycles to complete
  • Each remaining (n−1) instruction: +1 cycle each
  • Verified: k=6, n=9 → [6+8]×1 = 14 cycles ✓ (Fig 14.10)
Speedup Factor vs. No Pipeline
Sₖ = T(1,n)/T(k,n) = nkt / [k+(n−1)]t = nk / [k+(n−1)]
  • As n → ∞: Sₖ → k  (k-fold speedup — theoretical max)
  • As k → ∞: Sₖ → n  (limited by instruction count without branches)
Practical Limits on More Stages
  • Buffer-to-buffer overhead + latch delay per stage
  • Control logic complexity grows enormously with stage count
  • Branch flushes negate gains from deeper pipelines
  • Logically dependent sequential instructions stall often at more stages

Pipeline Hazards (§14.4)

A pipeline hazard (stall / bubble) forces the pipeline to pause. Three types:

1. Resource Hazards (Structural)
  • Two+ in-flight instructions need the same hardware resource simultaneously
  • Example: single-port memory — FI and FO stages conflict → one stalls for 1 cycle
  • Example: single ALU — two instructions reach EI together
  • Fix: duplicate resources (multiple memory ports, multiple ALUs)
2. Data Hazards
TypeAliasCondition
RAWTrue dependencyRead before preceding write completes (most common)
WARAntidependencyWrite completes before a preceding read finishes
WAWOutput dependencyTwo writes to same location arrive out of order
RAW Example (Fig 14.16)
ADD EAX, EBX — writes EAX at end of its EI stage (cycle 5).
SUB ECX, EAX — needs EAX at start of its FO stage (cycle 4).
→ 2 stall cycles inserted. Without bypass hardware, pipeline idles.
3. Control Hazards (Branch Hazards)

Pipeline fetches wrong instructions after a conditional branch → must flush. Primary source of pipelined processor inefficiency. Addressed by branch prediction strategies.

Branch Strategies — All 5 Approaches

1. Multiple Streams

Replicate early pipeline stages; fetch both branch target and sequential instruction simultaneously. Drawbacks: contention for registers/memory; nested branches multiply needed streams. (IBM 370/168, IBM 3033)

2. Prefetch Branch Target

On recognizing a conditional branch, prefetch the target address into a buffer alongside the sequential path. If branch taken → target already ready. (IBM 360/91)

3. Loop Buffer

Small very-high-speed memory in the fetch stage holding the n most recently fetched sequential instructions. Three benefits: (1) prefetched instructions available without memory latency; (2) short forward branches (IF-THEN sequences) likely already in buffer; (3) loop bodies fetched from memory only on first iteration — all subsequent iterations run from buffer. (CDC 6600, 7600, CRAY-1)

4. Branch Prediction
MethodTypeNotes
Predict never takenStaticMost popular; always fetch sequentially
Predict always takenStaticBetter when branches taken >50% of the time
Predict by opcodeStaticReported >75% accuracy
1-bit historyDynamicRecords last outcome; 2 errors per loop (entry + exit)
2-bit history (Fig 14.18/19)DynamicRequires 2 consecutive mispredictions to switch; better for loops
Branch history tableDynamicCache of {branch addr + history bits + target addr}; enables prefetch of target before branch resolves
(m,n) correlatorDynamicUses last m global branches + n-bit local history; correlation-based; most accurate
5. Delayed Branch

Compiler reorders instructions to fill the pipeline slot after the branch with useful work. The "delay slot" executes regardless of branch outcome. Detailed in Ch. 15 (RISC).

Intel 80486 — 5-Stage Pipeline

Classic real-world pipelined processor. Sustains ~1 instruction per clock cycle.

5 Stages (§14.4)
  1. Fetch: independently fills two 16-byte prefetch buffers from cache or external memory; ~5 instructions per 16-byte load; variable-length instructions (1–11 bytes)
  2. Decode 1 (D1): decodes opcode + addressing-mode from first 3 bytes; determines total instruction length; directs D2 to get displacement/immediate
  3. Decode 2 (D2): expands opcode into ALU control signals; handles complex addressing mode calculations
  4. Execute (EX): ALU operations, cache access, register update
  5. Write Back (WB): updates registers and status flags; result sent to cache + write buffer simultaneously if memory write
Pipeline Timing Cases (Fig 14.21)
  • No delay: reg-to-reg ops flow through cleanly at 1 instr/cycle
  • Pointer load delay (+1 cycle): if a loaded register is used as base address in the very next instruction — D2 stage needs the value while WB is still writing it; bypass paths save 1 extra cycle (only 1 stall, not 2)
  • Branch taken: compare updates condition codes in WB; bypass to EX of jump; speculative prefetch of branch target during jump EX; if branch not taken → discard speculative fetch and continue sequentially

MMX Registers — Aliasing (Fig 14.24)

No new registers added. MM0–MM7 alias the low 64 bits (mantissa) of each FP register. On MMX write → bits [79:64] set to all-1s (NaN/∞) to prevent FP misinterpretation. EMMS instruction must clear tag word before returning to FP operations.

Control Registers CR0–CR4 (Fig 14.23) Key Bits
  • PE (CR0 bit 0): Protection Enable — real→protected mode
  • EM (CR0 bit 2): Emulation — FP instructions cause interrupt if no FPU
  • WP (CR0 bit 16): Write Protect — supervisor can't write user read-only pages
  • PG (CR0 bit 31): Paging enable
  • CR2: Page fault linear address
  • CR3: Page directory base (PDBR), PCD + PWT bits
  • CR4: Extended feature control (VME, PAE, PSE, OSFXSR, etc.)

x86 Interrupt Processing (§14.5)

Two Event Classes
  • Interrupts: asynchronous; hardware signal on INTR or NMI pin
  • Exceptions: synchronous; from executing a specific instruction
Sources
SourceMaskable?
INTR pin (maskable HW interrupt)Yes — IF flag must be set
NMI pin (non-maskable HW interrupt)No — always processed
Processor-detected exceptionN/A — from execution error
Programmed (INT/INTO/BOUND)N/A — software-generated
Key Interrupt Vector Table Entries (Table 14.3)
  • 0 = Divide error   1 = Debug   2 = NMI   3 = Breakpoint
  • 6 = Undefined opcode   7 = Device not available   8 = Double fault
  • 14 = Page fault   16 = FP error   17 = Alignment check
  • 32–255 = User INTR vectors (activated by INTR signal + vector number on bus)
Interrupt Handling — 6-Step Sequence
  1. If privilege level changes → push SS + ESP to stack
  2. Push EFLAGS to stack
  3. Clear IF and TF (disable further INTR and single-step)
  4. Push CS + EIP to stack
  5. If error code → push error code
  6. Load interrupt vector into CS:EIP → execute ISR

IRET restores all saved values; resumes from point of interrupt. Priority: Class 1 (traps on prev instr) > Class 2 (external) > Class 3 (fetch faults) > Class 4 (decode faults) > Class 5 (execute faults).

§14.6 ARM Processor Organization

ARM RISC Characteristics
  • Moderate uniform register file (more than some CISC, fewer than many RISC)
  • Load/Store model: only LDR/STR access memory; all other operations use registers only
  • Fixed 32-bit instructions (ARM standard) / 16-bit (Thumb)
  • Barrel Shifter: shifts/rotates one source register Rm inline, within the same instruction cycle → increases ALU power without extra instructions
  • Auto-increment / auto-decrement addressing for efficient loops
  • Conditional execution on almost all instructions → fewer branch instructions → fewer pipeline flushes
Simplified ARM Data Path (Fig 14.25)
  • Data bus: load/store data OR instruction fetches from external memory
  • Register file R0–R15: all 32-bit; byte/halfword sign-extended to 32 bits on load
  • R15 = PC; R14 = LR (link register); R13 = SP (stack pointer)
  • Barrel Shifter: preprocesses Rm value before it enters ALU (shift/rotate)
  • ALU: inputs from Rn and shifted Rm → result to Rd
  • Multiply/Accumulate: separate unit with Acc register for partial products
  • CPSR: current program status register with N/Z/C/V condition flags + control bits

ARM Processor Modes (§14.6)

ARM supports 7 modes — more than most processors, enabling fine-grained OS protection policies.

ModeClassEntry / Use
UserUnprivilegedDefault application mode; no protected resource access; cannot change mode except via exception
SystemPrivilegedNot entered by exception; uses User-mode registers; for privileged OS tasks that can be interrupted by any exception
SupervisorExceptionSoftware interrupt (SWI instruction) — standard user→OS call mechanism
AbortExceptionMemory fault: prefetch abort (instr fetch fault) or data abort (data access fault)
UndefinedExceptionUnrecognized opcode reaches execute stage
IRQExceptionExternal interrupt via IRQ pin; interruptible only by FIQ
FIQExceptionFast interrupt via FIQ pin; highest priority; private R8–R12 eliminate register save overhead; cannot be interrupted
Two Advantages of Many Modes
(1) OS can tailor system software to diverse circumstances. (2) Dedicated registers per mode allow faster context switches — no need to save/restore registers for short ISRs.

ARM Register Organization & Exceptions

37 Total Registers (Fig 14.26)
  • R0–R7, R15 (PC), CPSR: shared by all modes — always visible
  • R8–R12: shared by all modes except FIQ, which has private R8_fiq–R12_fiq
  • R13 (SP), R14 (LR): each exception mode has its own private copy (R13_svc, R13_abt, R13_und, R13_irq, R13_fiq etc.)
  • SPSR: each exception mode has a Saved PSR to preserve CPSR at exception entry
  • At any time: 16–18 registers visible (16 numbered + CPSR + optionally SPSR)
ARM CPSR/SPSR Bit Fields (Fig 14.27)

Bits [31:16] — User flags: N (neg), Z (zero), C (carry), V (overflow), Q (SIMD saturation), J (Jazelle), GE[3:0] (SIMD ≥)

Bits [15:0] — System control (privileged): E (endianness), A/I/F (abort/IRQ/FIQ disable), T (Thumb mode), M[4:0] (current mode)
