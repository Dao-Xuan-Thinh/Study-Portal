Mode-by-Mode Deep Dive

Immediate

Operand is embedded in the instruction itself. Stored in twos complement; sign-extended when loaded into a register. No memory access for operand. Useful for constants and initialization.

Direct

Address field = exact memory address. Simple but address field is small → limited to a narrow range of memory. Common in early computers; rare today.

Indirect

Address field points to a memory word that contains the real address. Unlocks full N-bit address space (2ᴺ). Costs an extra memory cycle. Multilevel/cascaded variant: EA = (…(A)…) — each level adds a memory ref.

Register & Register Indirect
  • Register: operand is in a CPU register (3–5 bit field → 8–32 registers). Very fast, no memory access
  • Register Indirect: register holds the address of the operand in memory. One memory ref; full word-length address; 1 fewer ref than indirect
Stack

Implied addressing — no address field needed. Stack pointer (SP) register tracks top-of-stack. Push/pop automatically update SP. LIFO queue; ideal for expression evaluation and subroutine calls.

Displacement Addressing

EA = A + (R)   [address field + register contents]

Requires two address fields in the instruction. Combines capabilities of direct addressing (the constant A) and register indirect addressing (the register R). Three major uses:

1. Relative (PC-Relative) Addressing
  • Implicit register = Program Counter (PC)
  • EA = A + (PC) — A is a signed twos complement offset
  • Effective address is relative to the current instruction
  • Exploits locality — nearby addresses need fewer bits
  • Used for branches and jumps in position-independent code
2. Base-Register Addressing
  • Register holds a base (segment start) address; A is an unsigned displacement/offset
  • Register reference may be explicit or implicit
  • Supports segmentation — one segment-base register per segment
  • If N registers, K-bit offset: can reference N regions of 2ᴷ words
3. Indexing
  • A = base memory address (in instruction); Register = positive offset/index
  • Opposite interpretation from base-register: A is the fixed base, R increments
  • Ideal for iterative array traversal — increment index register each loop
  • Autoindexing: hardware auto-increments/decrements index reg after each use

Indexing Variants: Pre vs. Post Indexing

Autoindexing
EA = A + (R)   then   (R) ← (R) + 1

Register automatically increments after each reference. Enables clean loop iteration without separate increment instructions.

Postindexing (index after indirect)
EA = (A) + (R)
  1. Fetch the address stored at memory location A → get a direct address
  2. Add the register (index) value to that address → final EA

Useful when A is a pointer variable (e.g., pointer to start of a process control block) and R is the field offset within that block.

Preindexing (index before indirect)
EA = (A + (R))
  1. Calculate A + (R) first → intermediate address
  2. Fetch the operand address from that intermediate memory location → final EA

Useful for multi-way branch tables: A = start of address table, R = index into table. The table entry contains the actual branch target.

💡 Key Difference
Postindexing: pointer first, then index. Preindexing: index first, then indirect lookup. Most ISAs offer only one of the two.

Segment register → descriptor table → base address. Final linear address = segment base + effective address. Then (if paging) → physical address via MMU.

ARM Addressing: Load/Store Indexing (Fig 14.3)

ARM is a load/store architecture — only load and store instructions access memory. All memory addressing done via base register + offset. Three indexing methods:

1. Offset (No update)
Address = base_reg + offset   (base not modified)

Base register unchanged after access. Example: STRB r0, [r1, #12] → address = r1 + 12; r1 stays the same.

2. Pre-index (Write-back)
Address = base_reg + offset   then   base_reg ← updated

Address formed same as offset, but the computed address is written back into the base register. Signified by ! in assembly: STRB r0, [r1, #12]!

3. Post-index
Address = base_reg   then   base_reg ← base_reg + offset

Memory is accessed at the original base address; then the base register is updated with the offset added. Example: STRB r0, [r1], #12

Scaled Register Offset

Offset register can be shifted before adding to base. Shift operators: LSL, LSR, ASR, ROR, RRX (rotate with carry). Shift amount specified as immediate in instruction.

ARM Load/Store Multiple (Fig 14.4)

LDM / STM instructions load or store a subset of all general-purpose registers in a single instruction. A 16-bit field in the instruction has one bit per register.

Instruction Format Design Principles

Instruction Length Trade-offs
  • Longer → more opcodes, operands, address modes, larger address range
  • Shorter → fewer bits wasted; faster fetch (16-bit fetched 2× faster than 32-bit)
  • Should equal or be a multiple of the memory bus width (avoid partial fetches)
  • Should be a multiple of character size (typically 8 bits) and fixed-point word length
Bit Allocation Trade-offs
  • More opcodes → fewer bits for addressing
  • Variable-length opcodes — short common opcodes, longer rare ones; remaining bits used for addressing
  • Registers vs memory: 3–5 bits per register ref (8–32 regs); studies recommend 8–32 user-visible registers
  • Register sets: 1 general-purpose set (32+ regs) is typical; specialized sets (x86) reduce bits per reference
  • Address range: displacement addressing extends range to register size; larger displacement = longer instruction
  • Granularity: byte addressing needs more address bits than word addressing for same memory size
💡 Orthogonality
Addressing mode independent of opcode. Any mode usable with any instruction. Seen in PDP-11, ARM. Reduces programmer burden.

Design Principles
  • Orthogonality: EA calculated the same way regardless of opcode
  • Completeness: identical instruction sets for all data types (int, fixed, float)
  • Direct addressing: 18-bit address field makes direct addressing practical (avoids base+displacement burden)
  • 9-bit opcode → up to 512 ops; 365 defined
  • 16 general-purpose registers (4-bit field); same registers used as index registers
  • Indirect + indexing both supported for addresses > 2¹⁸
Trade-off
Ease of programming at the cost of less efficient space utilization — a conscious, justified design choice.

PDP-11 & VAX — Variable-Length Formats

PDP-11 (16-bit, variable-length)

Eight 16-bit GPRs (R7 = PC, R6 = SP). 13 different instruction formats. Zero, one, and two-address types.

  • Opcode: 4 to 16 bits (variable)
  • Each register reference: 6 bits = 3-bit mode + 3-bit register number
  • Orthogonal: mode tied to operand, not opcode → any mode with any instruction
  • Base: 16-bit instruction; extended to 32 or 48 bits with appended memory addresses
VAX — Maximally Flexible Format

Two design goals: (1) each instruction has its "natural" number of operands; (2) all operands equally general in addressing.

  • 1 or 2 byte opcode; 0–6 operand specifiers per instruction
  • Instructions range from 1 byte to 37 bytes
  • Each operand specifier: 1 byte minimum; 4-bit mode field + 4-bit register; extendable with constant bytes
  • 12 different addressing modes per operand
  • Literal mode: 00 in top 2 bits → 6-bit literal embedded in operand byte
  • Index mode prefix byte for indexed addressing
  • Supports memory-to-memory operations in a single instruction (contrast: x86 cannot)

x86 vs VAX Key Difference
x86: addressing mode in opcode sequence → only ONE memory operand per instruction. VAX: mode per operand → memory-to-memory in one instruction. x86 is more compact but less flexible.

ARM Thumb & Thumb-2 Instruction Sets

Thumb (16-bit re-encoding of ARM)

Designed for better code density on narrow memory buses. Subset of ARM ISA re-encoded into 16-bit instructions. Space savings achieved by:

  1. Unconditional only → no 4-bit condition field (saves 4 bits)
  2. All ALU ops update flags → no S bit (saves 1 bit). Total: 5 bits saved
  3. Reduced opcode: 2-bit opcode + 3-bit type (saves 2 bits)
  4. Smaller register file: only r0–r7 (3-bit field vs 4-bit) + no rotate on immediate. Saves remaining 9 bits
Thumb-2 (Enhanced — Cortex-M default ISA)
  • Intermixes 16-bit Thumb and new 32-bit instructions freely
  • 32-bit Thumb-2 instructions are mostly unconditional
  • New If-Then (IT) instruction provides ARM-style conditional execution for up to 4 following instructions
  • Code density ≈ Thumb; performance ≈ ARM (98% of ARM on benchmarks)
  • Eliminates need to choose ARM (performance) vs Thumb (size)

Variable-Length Instructions

Advantages
  • Efficient and compact representation of diverse operations
  • Large opcode space without wasting bits on simple instructions
  • Mixed operand counts and addressing modes in the same ISA
  • Short instructions for common cases → better code density
Disadvantages / Costs
  • Increases processor decode complexity
  • Harder to pipeline (instruction boundaries unknown until decoded)
  • RISC/superscalar processors exploit fixed-length for improved IPC
Fetch Strategy

Processor doesn't know next instruction's length before decoding. Typical approach: fetch enough bytes for the maximum possible instruction length. Often fetches more than one instruction per cycle — beneficial for prefetch pipelines.

Length Constraint Rule
All instruction lengths should be integral multiples of the word/character length (e.g., 8-bit multiples for 8-bit character systems)
Variable-Length Opcodes

Minimum opcode length + extension bits for rare operations. Extension bits borrowed from addressing fields → simpler instructions get more address bits; complex instructions get fewer. Used in x86 (1–3 byte opcodes) and VAX (1–2 byte opcodes).
