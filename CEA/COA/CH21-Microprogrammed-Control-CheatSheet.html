§21.1 — Basic Concepts

Motivation & History of Microprogramming

M. V. Wilkes proposed microprogramming in 1951 as an organized, systematic alternative to complex hardwired control unit design.

The Problem with Hardwired Control
  • Must implement logic for: sequencing μOPs, executing μOPs, interpreting opcodes, testing ALU flags
  • Difficult to design, debug, and test
  • Inflexible — adding a new machine instruction requires redesigning the hardware
The Microprogramming Solution

Replace complex logic circuits with a small program (microprogram) stored in control memory. The control unit becomes a simple microprogram sequencer + executor.

Microprogram = sequence of microinstructions
Microinstruction = one set of control signals (one row)
Control memory = ROM holding the microprogram
Historical Milestone
IBM System/360 (announced April 1964) used microprogramming for all but the largest models, making it mainstream. Previously, major manufacturers had dismissed the technique as impractical.
Firmware = Midway between HW and SW
Microprogram is called firmware: easier to write than hardware circuits, harder than software. Can be updated without hardware redesign.

Control Word · Microinstruction · Microprogram

Each micro-operation only requires the CU to set some control lines ON (1) and others OFF (0). This can be encoded as a binary control word.

Execution Rules for a Microinstruction
  1. Turn ON all control lines where bit = 1; leave OFF all lines where bit = 0. This executes a set of micro-operations.
  2. If the condition field evaluates FALSE → fetch the next sequential microinstruction.
  3. If the condition evaluates TRUE → fetch the microinstruction at the address field.

Microprogrammed Control Unit (Fig 21.4)

The microprogrammed CU replaces combinational logic with a program stored in control memory. Same I/O interface as hardwired CU — just different implementation.

Control Unit Structure
▲ IR / ALU Flags / Clock — INPUTS
Sequencing Logic
generates next CAR address
Control Memory (ROM)
microprogram stored here
Control Buffer Register (CBR)
holds current microinstruction
▼ Control Signals — OUTPUTS (to CPU + system bus)
One Clock Pulse Cycle
  1. Sequencing logic issues READ to control memory at address in CAR.
  2. Microinstruction at that address → loaded into CBR.
  3. Left portion of CBR drives control signal lines directly (horizontal) or via decoder (vertical).
  4. Sequencing logic reads CBR's next-address info + ALU flags → loads new address into CAR.
Three Sequencing Decisions per Cycle
  • Increment: CAR ← CAR + 1 (sequential execution)
  • Branch: CAR ← address field in CBR (jump microinstruction)
  • Map from IR: CAR ← f(opcode) via upper decoder (start of machine instruction's microroutine)

Two Decoders in the CU (Fig 21.4)
  • Upper decoder: maps opcode from IR → starting control memory address of that instruction's microroutine
  • Lower decoder: only used for vertical microinstructions — decodes encoded fields into individual control signals

Wilkes Control (1951) — First Microprogrammed CU

Wilkes's original design used a diode matrix. Each row = one microinstruction; activated by a pulse on that row's address line.

Matrix Layout
  • Part 1 of row: control signals for the current μOP (dot = signal active)
  • Part 2 of row: address of the next microinstruction to activate
Operation
  1. Register I holds current row address → feeds decoder
  2. Clock pulse activates decoder → pulses one row of matrix
  3. Row part 1 generates control signals (micro-operations)
  4. Row part 2 OR instruction register opcode → loaded into Register II
  5. Second clock transfers Register II → Register I (two registers prevent feedback instability)
Key Difference from Fig 21.1a
No Increment
In Wilkes's scheme, the next address is embedded in every microinstruction — no incrementer. To branch, a row contains two alternative addresses controlled by a flag (conditional signal). This requires more bits per microinstruction.
Wilkes Example Machine Registers
  • A, B, C, D: Accumulator (split into halves), shift register
  • E (MAR), F (PC), G: control-only registers (not user-visible)
  • 38 microinstructions define the complete control unit for the example machine

Advantages & Disadvantages

Advantages of Microprogramming
  • Simpler design: CU logic reduces to sequencer + control memory — simpler than hardwired logic
  • Lower cost: cheaper and less error-prone to implement
  • Flexibility: new instructions added by updating ROM/RAM, not redesigning circuits
  • Error correction: bugs fixed via firmware update rather than chip respin
  • Emulation: run another ISA's instructions by providing its microprogram
  • Regularity: control memory behaves uniformly, making design systematic
Disadvantages
  • Slower: must access control memory every microinstruction cycle; hardwired is faster with comparable technology
  • Extra hardware: control memory, CAR, CBR, sequencer all add complexity + cost
  • Propagation delay: vertical format requires additional decode logic — more delay than horizontal
Current Relevance
ArchitectureControl Strategy
CISC (e.g., x86)Historically microprogrammed; modern CPUs use hybrid (RISC core + microcode for complex instr)
RISCHardwired — simpler ISA makes this practical and fast
Pentium 4Most instructions → RISC-like μOPs (hardwired); some → microprogram
§21.2 — Microinstruction Sequencing

Sequencing Design Considerations

Every microinstruction cycle, the sequencer must determine the address of the next microinstruction. Two competing design goals:

Goal 1: Minimize control memory SIZE
          (fewer address bits = smaller memory)
Goal 2: Minimize address GENERATION TIME
          (execute microinstructions as fast as possible)
Three Categories for "Next Address"
#SourceWhen Used
1IR (opcode mapping)Exactly once per machine instruction — at start of execute cycle, opcode maps to microroutine start address
2Next sequential (+1)Most common — when executing in-order microinstructions
3Branch (conditional / unconditional)At end of subcycles (fetch → execute), loops, conditional actions. ~1 in 3–4 microinstructions is a branch
Branch Frequency
Studies show ~1 in every 3–4 microinstructions is a branch. Cannot optimize only for sequential access — branch efficiency is critical.

Address Generation Techniques (Table 21.3)

Beyond the three format-based approaches, address generation techniques can be categorized as explicit (address in microinstruction) or implicit (computed by logic).

CategoryTechniqueHow It Works

Conditional Branch Condition Sources
  • ALU condition codes (zero, overflow, carry, sign)
  • Parts of the opcode or address mode fields of the machine instruction
  • Sign bit or other bits of a selected register
  • Status bits internal to the control unit

LSI-11 Microinstruction Sequencing

The LSI-11 (PDP-11 single-board) uses a 22-bit microinstruction and 2K × 22-bit control memory. Demonstrates residual control (subroutine facility).

Five Next-Address Mechanisms
  1. Next sequential: CAR ← CAR + 1 (default)
  2. Opcode mapping: start of each instruction cycle, opcode determines starting microroutine address
  3. Subroutine facility: 1 dedicated bit in every microinstruction; when set, loads 11-bit return register with updated CAR. A subsequent return microinstruction loads CAR from return register.
  4. Interrupt testing: specific microinstructions test for pending interrupts; if found, determines next address
  5. Conditional/unconditional branch: 4-bit test code tests ALU condition codes; if false → +1; if true → load 8 low-order bits from microinstruction (256-word page branch)
IBM 3033 — Addition-Based Addressing (Fig 21.9)

13-bit CAR divided into two parts:

Bits 00–07 (high): copied from BA field of microinstruction
                   → selects a 32-microinstruction BLOCK
Bits 08–12 (low): each bit set by a 4–7-bit condition field
                  → specifies OFFSET within that block

§21.3 — Microinstruction Execution

Microinstruction Execution Overview

Each microinstruction cycle = fetch + execute. The execute phase generates control signals from the microinstruction bits, causing micro-operations in the CPU.

Control Unit Organization (Fig 21.10)
  • Sequencing logic module: generates next CAR address using IR, ALU flags, CAR (for +1), and CBR (for branch address)
  • Control logic module: generates control signals from bits in the current microinstruction (CBR); complexity depends on degree of encoding
  • Clock: drives all timing in the microinstruction cycle
Control Signal Types Generated
TypeExamples
Internal control signalsOpen gate PC→MAR; activate ALU add; move MBR→IR
External control signalsMemory READ/WRITE; I/O read/write on system bus
Emulation
By loading a different microprogram, the same CPU hardware can execute a completely different ISA. E.g., an IBM 360 model could emulate a prior IBM 1401 architecture via a different microprogram.

Microinstruction Encoding — Direct vs. Indirect (Fig 21.11)

Why Encode?

With K total control signals, pure unencoded format needs K bits. But many combinations are physically impossible or mutually exclusive:

  • Two sources cannot be gated to the same destination simultaneously
  • A register cannot be both source and destination in one time unit
  • Only one ALU function can be active at a time
  • Only one pattern on the external control bus at a time

With Q valid combinations (Q < 2ᴷ), we only need log₂Q bits minimum. In practice, use more than log₂Q for programmability, but fewer than K.

Field-Based Encoding (Direct, Fig 21.11a)
Field 1
Field 2
Field 3
…
Fld N
  • Each field → its own decoder → its own subset of control signals
  • N fields → N simultaneous independent actions per microinstruction
  • Actions within one field are mutually exclusive (only one code per field at a time)
  • Actions from different fields can occur simultaneously
Direct vs Indirect Encoding
TypeMechanismTrade-off
DirectEach field decoded independently → control signalsSimpler logic, single decode level
IndirectOne field determines interpretation of another field (two-level decoding)More compact, but adds propagation delay (two decode stages)
Indirect Encoding Example
ALU has 8 arithmetic ops + 8 shift ops. Use 1-bit field to select arithmetic vs. shift; then 3-bit field for the specific operation. Total = 4 bits instead of 4 bits for direct (same here, but can be more efficient in complex cases).
Field Design Rules
  1. Organize into independent fields — actions from different fields must be able to occur simultaneously.
  2. Within each field, the alternative actions must be mutually exclusive — only one can occur at a given time.
Encoding Approaches
  • Functional: one field per function type (e.g., all data sources for accumulator in one field)
  • Resource: one field per hardware resource (ALU field, I/O field, memory field)

Key Terms & Definitions

  • Microinstruction: one control word; one row of control memory; activates a set of control lines
  • Microprogram (Firmware): sequence of microinstructions implementing the CU behavior; between HW and SW
  • Control memory: ROM/RAM holding all microprograms; addressed by CAR
  • CAR (Control Address Register): holds address of next microinstruction
  • CBR (Control Buffer Register): holds currently executing microinstruction
  • Horizontal μinstruction: unencoded; 1 bit per control line; wide; fast; hard to program
  • Vertical μinstruction: encoded; few bits → decoder → signals; narrow; slower; easier to program
  • Mapping: implicit address generation — opcode → microroutine start address via special logic
  • Residual control: use previously saved address from return register/stack for next microinstruction
  • Emulation: microprogramming an ISA on hardware designed for a different ISA
  • Hard microprogramming: fixed microprogram in ROM; optimized for performance
  • Soft microprogramming: writable control store; user-programmable; flexible
  • Writable Control Store (WCS): RAM-based control memory; allows dynamic microprogramming
  • Microroutine: set of microinstructions implementing one machine-level instruction
  • Field: group of bits in a microinstruction encoding mutually exclusive actions
