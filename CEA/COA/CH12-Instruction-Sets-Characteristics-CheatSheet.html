Machine Instruction Elements

A machine instruction is a binary-encoded command that the processor fetches and executes. The full collection of instructions a processor can execute is its instruction set.

Four Required Elements
  • Operation Code (Opcode) — binary code specifying the operation (ADD, SUB, LOAD…)
  • Source Operand Reference — one or more inputs for the operation
  • Result Operand Reference — destination for the computed result
  • Next Instruction Reference — address of next instruction to fetch (usually implicit via PC)
Operand Locations
  • Main / Virtual Memory — full address must be supplied
  • Processor Register — unique name/number; fewer bits needed
  • Immediate — value is embedded directly in the instruction field
  • I/O Device — I/O module + device ID (or memory-mapped address)

Instruction Types

Any program written in a high-level language must be translatable to machine language. The instruction set must be sufficient to express all HLL constructs.

HLL→Machine Expansion Example
BASIC: X = X + Y → 3 Machine Instructions
1. LOAD R, 513 ; R ← mem[513] (X)
2. ADD R, 514 ; R ← R + mem[514] (Y)
3. STOR 513, R ; mem[513] ← R
Key Point
One HLL statement → multiple machine instructions. HLL is algebraic; machine language is move-based.

Instruction Set Design Issues

One of the most analyzed aspects of computer design — fundamental decisions remain disputed even today (see RISC vs. CISC in Chapter 15).

Five Core Design Dimensions
  • Operation Repertoire — how many ops, how complex
  • Data Types — types upon which operations act
  • Instruction Format — length, field sizes, # of addresses
  • Registers — count and usage of processor registers
  • Addressing Modes — how operand addresses are specified
Design Pathway (this book)
  • CH12 → Data types & operation repertoire
  • CH13 → Addressing modes & instruction formats
  • CH15 → RISC architecture & design philosophy
Key Insight
All five dimensions are highly interrelated and must be considered together — no single dimension can be optimized in isolation.

Operand Type: Logical Data

An n-bit unit is treated as n independent 1-bit items (Boolean values), each being 0 (false) or 1 (true).

Two Advantages of Bit-Oriented View
  1. Store Boolean arrays with maximum memory efficiency (1 bit per element)
  2. Manipulate individual bits — e.g., shift significand in software float, extract BCD digits from IRA bytes
Common Bitwise Masking Techniques
AND as mask (isolate lower nibble)
R1 = 1010 0101
R2 = 0000 1111
R1 AND R2 = 0000 0101 ← masked
XOR with all-1s = ones complement
R1 = 1010 0101
R2 = 1111 1111
R1 XOR R2 = 0101 1010
Type Polymorphism
The same data can be logical, numeric, or text — determined by the operation applied, not a stored type tag.

Most ARM implementations omit FP hardware (saves power/area). FP can be done in software, or via optional coprocessor supporting IEEE 754 single & double precision.

Transfer of Control: Branches & Skips

Updates the Program Counter to a new address rather than incrementing sequentially. Essential for loops, decisions, and subroutines.

Three Reasons Transfer-of-Control is Needed
  1. Repeat instructions (loops) for iterative data processing
  2. Conditional execution based on test results
  3. Modular programming via procedures/subroutines

Transfer of Control: Procedure Calls

A procedure is a self-contained program invoked from a larger program. The two key instructions are CALL and RETURN.

Why Procedures?
  • Economy — reuse code; reduce memory footprint
  • Modularity — decompose large tasks into manageable units
Three Return Address Storage Methods
MethodCALL X does…
Register (RN)RN ← PC+Δ; PC ← X
Start of Procmem[X] ← PC+Δ; PC ← X+1
Top of Stackpush(PC+Δ); PC ← X — supports re-entrancy
Re-entrancy
Register & start-of-procedure methods do not support reentrant (concurrent/recursive) calls. Only the stack method fully supports re-entrancy and nesting.
Nested Procedure Execution (Fig. 12.8)
Address layout

4800: [PROC2]
4xxx: RETURN → caller
Three Important Properties
  1. A procedure can be called from multiple locations
  2. Procedures can call other procedures (arbitrary nesting depth)
  3. Every CALL is matched by exactly one RETURN

Endianness (Appendix 12A)

Defines byte order in which a multi-byte value is stored in memory.

Little-Endian (x86)
Value 0x12345678 at address 100
Addr 100: 0x78 (LSB first)
Addr 101: 0x56
Addr 102: 0x34
Addr 103: 0x12 (MSB last)
Big-Endian (e.g., network order)
Value 0x12345678 at address 100
Addr 100: 0x12 (MSB first)
Addr 101: 0x34
Addr 102: 0x56
Addr 103: 0x78 (LSB last)
Bi-Endian (ARM)

ARM supports both via the E-bit in the Program Status Register. The SETEND instruction switches endianness at runtime. Address of each byte in memory is fixed; only the register byte lane assignment differs.

Condition Codes

All ARM data processing instructions can be made conditional by appending a 2-bit condition suffix (EQ, NE, GT, LT, GE, LE, CS, CC…) — eliminates many branch instructions.

Shift in Data Path

Every ARM data processing instruction includes a barrel shifter on the second operand: ADD R0, R1, R2 LSL #2 computes R0 = R1 + (R2 × 4).

Load/Store Multiple
LDM/STM can transfer up to 17 registers in a single instruction — useful for procedure prologue/epilogue.

Instruction Format Principles

Instruction length and field layout are critical design decisions affecting program size, memory bandwidth, and implementation complexity.

Format Design Variables
  • Instruction length — total bits (8, 16, 32, 64-bit common)
  • Number of addresses — 0, 1, 2, or 3 address fields
  • Field sizes — bits for opcode vs. operand specifiers
  • Addressing mode — may consume extra bits
Bit Allocation Trade-offs
Opcode bits: k → up to 2ᵏ distinct operations Operand bits: n → up to 2ⁿ addressable locations Register bits: r → up to 2ʳ registers (r << n)
Register vs. Memory Reference
Register references need far fewer bits than memory references (e.g., 3 bits for 8 registers vs. 32 bits for 4GB memory). This is a key motivation for register-rich ISAs.
Fixed vs. Variable Length
TypeProCon
FixedSimple decode; aligned fetchWasted bits for simple ops
VariableCompact encoding; efficientComplex decode; alignment issues
