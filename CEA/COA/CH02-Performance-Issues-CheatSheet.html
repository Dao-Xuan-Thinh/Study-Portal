§2.1 · Microprocessor Speed

Moore's Law has driven exponential transistor growth, enabling 4–5× performance improvement every ~3 years since Intel launched x86 in 1978. Raw speed potential is only realized if the processor is fed a constant, uninterrupted stream of instructions.

Key CPU Techniques to Sustain Throughput
  • Pipelining — overlaps multiple instruction stages (fetch → decode → execute …) simultaneously, like an assembly line
  • Branch Prediction — prefetches likely instruction branches to avoid stalls; advanced versions look multiple branches ahead
  • Superscalar Execution — issues >1 instruction per clock cycle using multiple parallel pipelines
  • Data-Flow Analysis — schedules instructions by data-dependency order, independent of program order, eliminating unnecessary delays
  • Speculative Execution — executes instructions ahead of their appearance using branch prediction & data-flow analysis; results held in temporary locations
Goal
Execute many instructions per cycle (IPC > 1), rather than many cycles per instruction (CPI > 1).

§2.1 · Performance Balance

Performance balance: adjusting organization and architecture to compensate for the mismatch among capabilities of different components. The processor–memory interface is the most critical bottleneck.

Processor–Memory Gap Solutions
  • Make DRAMs wider (more bits per access) rather than deeper, with wide bus data paths
  • Change DRAM interface — include on-chip caches or buffering schemes on DRAM
  • Add cache hierarchy (L1/L2/L3) between processor and main memory; on-chip cache has dramatically lower access latency
  • Increase interconnect bandwidth — higher-speed buses, hierarchical bus structures
I/O Bottleneck Solutions
  • Caching and buffering schemes for high-throughput peripherals
  • Higher-speed interconnect buses and elaborate interconnection structures
  • Multiple-processor configurations to distribute I/O demand
Two Evolving Factors
Performance improvement rates differ greatly by component type (CPU, bus, memory, peripherals); AND new applications constantly change instruction/data-access demand patterns.

§2.1 · Chip Organization & Architecture

Three Approaches to Increase Processor Speed
  • Increase hardware speed — shrink logic gate size → more gates per chip → shorter signal propagation → higher clock rate
  • Increase cache size & speed — on-chip cache drops access time dramatically; Pentium: ~10% of chip area; modern chips: >50% for cache
  • Improve processor organization — pipelining, superscalar, and other parallel execution techniques
Obstacles to Simply Raising Clock Speed
  • Power — higher clock & density → higher power density (W/cm²); heat dissipation becomes critical design constraint
  • RC Delay — thinner wires increase resistance; closer wires increase capacitance; RC product rises as components shrink
  • Memory Latency & Throughput — DRAM latency and bandwidth lag behind processor speed growth
Consequence
Clock speed gains plateaued ~2004. Chip area freed by Moore's Law is now used for multiple cores and larger caches, not faster single-core logic. ~¾ of non-cache area is pipeline control & buffering.

§2.2 · Multicore Architecture

Multiple processors (cores) placed on the same chip sharing a large cache. Avoids power-dissipation wall of higher clock speeds by trading clock frequency for parallelism.

Performance Scaling
Within one processor, performance scales roughly as √(complexity increase). But doubling the number of processors almost doubles performance — if software exploits parallelism.
Cache Hierarchy Evolution
  • Initially: L1 per core, L2 & L3 shared by all cores
  • Modern: L2 is commonly also private to each core
  • Power consumption of memory logic << processing logic → justifies larger caches per chip
  • Trajectory: 2-core → 4 → 8 → 16 → many cores (MIC)
Strategy

Use two simpler processors rather than one complex processor. The hardware parallelism gain outweighs the simplicity penalty per core.

§2.2 · Many Integrated Core (MIC)

MIC represents a leap to >50 cores per chip — a homogeneous collection of general-purpose processors on a single die. The term was coined to distinguish this regime from standard multicore (2–16 cores).

Key Characteristics
  • Homogeneous cores — all general-purpose, identical architecture
  • Massive on-chip parallelism requires highly parallel software
  • Challenge: developing software to exploit 50+ cores efficiently
  • Example: Intel Xeon Phi product line
Software Challenge
The hardware parallelism far outpaces available parallel software. Amdahl's Law limits gains from additional cores if the sequential fraction of code is non-trivial.

§2.2 · GPGPU Architecture

A GPU (Graphics Processing Unit) is a core designed for parallel operations on graphics data. GPGPU = General-Purpose Computing on GPUs: using GPU architecture for non-graphics workloads.

GPU vs CPU Design Philosophy
AttributeCPUGPU

Optimization targetLow latency, serialHigh throughput, parallel

Use Cases for GPGPU
  • Scientific simulations (repetitive floating-point operations on large arrays)
  • Machine learning / deep learning (matrix multiplications)
  • Signal & image processing
  • Blurs the CPU/GPU boundary — vector processing on general data sets

§2.3 · Amdahl's Law — Core Formula

Proposed by Gene Amdahl (1967). Quantifies the theoretical maximum speedup when parallelizing a program on N processors.

Variables
  • T — total single-processor execution time
  • f — fraction of code that is infinitely parallelizable (0 ≤ f ≤ 1)
  • (1–f) — fraction that is inherently sequential
  • N — number of parallel processors
Speedup Formula (Multiprocessor)
Speedup = T_single / T_parallel

Speedup = [T(1–f) + Tf] / [T(1–f) + Tf/N]

Speedup = 1 / [(1–f) + f/N]
Two Key Conclusions
  • When f is small → parallel processors have little effect (sequential bottleneck dominates)
  • As N→∞: Speedup is bounded by 1/(1–f) — hard ceiling regardless of processors added
Pessimism Note (Gustafson 1988)
These conclusions are too pessimistic in practice — servers, databases, and simulations scale problem size with available processors, making f effectively larger.

§2.3 · Amdahl's Law — Generalized Enhancement

Amdahl's Law generalizes to any system enhancement (not just parallel processors): cache improvement, faster memory, faster I/O, etc.

General Speedup Definition
Speedup = Performance_after / Performance_before
= Execution_time_before / Execution_time_after … (2.1)
Enhancement Speedup Formula

If feature f is used a fraction f of the time and is sped up by factor SU_f:

Speedup = 1 / [(1–f) + f/SU_f]
Example 2.1 — FP Hardware Speedup
Setup
40% of time is floating-point; FP module sped up by factor K.
Speedup = 1 / (0.6 + 0.4/K)

Max speedup (K→∞) = 1/0.6 = 1.67×
Insight
No matter how fast you make the FP unit, overall speedup is capped at 1.67× because 60% of work is non-FP. Optimizing the non-bottleneck provides diminishing returns.

§2.3 · Little's Law

Proposed by John Little (1961). A fundamental queuing-theory relationship applicable to any steady-state system with no item leakage.

Variables
  • L — average number of items in the system at any one time
  • λ (lambda) — average arrival rate of items (items per unit time)
  • W — average time each item spends in the system
Little's Law Formula
L = λ · W
Proof Intuition

When an item arrives it finds on average L items ahead of it. When it leaves, it also leaves L items behind (steady state). It spent time W in the system. In that time, λW items arrived. Therefore L = λW.

Key Properties
  • Requires no assumptions about service-time distribution, arrival-time distribution, or service order/priority
  • Applies to single-server, multi-server, single-queue, multi-queue topologies
  • Valid whenever the system is in statistical steady state

§2.3 · Little's Law — Computer System Applications

Multicore Application (from Litt11)

In a multicore system with shared memory: define λ as total thread request rate for memory; L as average stopped threads waiting; W as average response time. Little's Law provides a quantitative measure of whether user requirements are being met and how much improvement is needed.

Revival
Little's Law experienced a revival in interest due to multicore performance analysis and cloud system throughput modeling.

§2.4 · Clock Speed & Cycle Time

Clock Fundamentals
  • Quartz crystal generates constant sine wave → converted to digital voltage pulses
  • Clock rate / clock speed = pulse frequency, measured in Hz (cycles/second)
  • Clock cycle / tick = one increment (pulse) of the clock
  • 1 GHz processor = 1 billion pulses/second
Key Relationships
Clock rate: f (Hz = cycles/second)
Cycle time: τ = 1/f (seconds/cycle)

Example: f = 1 GHz → τ = 1 ns
Clock Rate Constraints
  • Clock rate is NOT arbitrary — must allow voltage levels to settle for accurate signal values
  • Physical layout determines propagation delays; some paths take longer than others
  • Operations must be synchronized so all values are stable before the next clock edge
Clock Speed ≠ Performance
Different processors with the same clock speed execute different amounts of work per cycle (IPC varies). Clock speed alone cannot compare across architectures.

§2.4 · CPI & Execution Time Formulas

Key Variables
  • f — clock frequency (Hz)
  • τ — clock cycle time = 1/f (s)
  • I_c — instruction count (# of instructions executed for a program)
  • CPI — average cycles per instruction
  • CPI_i — cycles for instruction type i; I_i — count of type-i instructions
CPI Formula (Mixed Instruction Types)
CPI = Σ(CPI_i × I_i) / I_c … (2.2)
Basic Execution Time
T = I_c × CPI × τ = I_c × CPI / f
Refined Execution Time (with Memory)
T = I_c × [p + (m × k)] × τ

p = processor cycles to decode & execute
m = number of memory references
k = memory_cycle_time / processor_cycle_time
5 Performance Factors
I_c, p, m, k, τ — influenced by: Instruction Set Architecture, Compiler Technology, Processor Implementation, Cache & Memory Hierarchy.

§2.4 · MIPS, MFLOPS & Worked Example

MIPS Rate Formula
MIPS = I_c / (T × 10⁶) = f / (CPI × 10⁶) … (2.3)
MFLOPS Rate Formula
MFLOPS = (# FP operations executed) / (Execution time × 10⁶)
Example 2.2 — CPI & MIPS Calculation
Setup
2M instructions on 400-MHz processor
Instruction TypeCPI_iMix %Contribution
Arithmetic & logic160%0.60
Load/store (cache hit)218%0.36
Branch412%0.48
Memory ref (cache miss)810%0.80
Average CPI—2.24
MIPS = (400 × 10⁶) / (2.24 × 10⁶) ≈ 178 MIPS
MIPS Limitation
MIPS is architecture-dependent. A RISC may execute 4× more instructions than a CISC for the same high-level statement yet achieve the same wall-clock performance (CISC 1 MIPS vs RISC 4 MIPS — both equal work in equal time).

§2.5 · Three Mean Formulas

Given n positive real numbers (x₁, x₂, …, xₙ):

Arithmetic Mean (AM)
AM = (x₁ + x₂ + … + xₙ) / n
= (1/n) · Σ xᵢ … (2.4)

Functional form: f(x) = x
Geometric Mean (GM)
GM = ⁿ√(x₁ · x₂ · … · xₙ)
= (∏ xᵢ)^(1/n)
= exp[(1/n) · Σ ln(xᵢ)] … (2.5)

Functional form: f(x) = ln(x)
Harmonic Mean (HM)
HM = n / Σ(1/xᵢ) … (2.6)

Functional form: f(x) = 1/x
Fundamental Inequality
AM ≥ GM ≥ HM (equality iff all xᵢ are equal)
Weighted Harmonic Mean (for Rates)
WHM = Σ Zⱼ / Σ tᵢ … (2.7)

where Zᵢ = operation count, tᵢ = execution time

§2.5 · When to Use Each Mean

MeanUse WhenPropertyExample Application
Arithmetic Measuring execution times Proportional to sum of values; proportional to total time Average runtime across benchmark runs; comparing simulation times
Harmonic Measuring execution rates (MIPS, MFLOPS) Inversely proportional to total execution time Mean MFLOPS across benchmarks; rate comparisons
Geometric Normalized performance ratios (benchmark comparisons) Consistent regardless of reference machine; equal weight to all values in multiplicative sense SPEC score computation; normalizing to reference machine
Arithmetic Mean of Rates is WRONG
Using AM on MFLOPS or MIPS data gives incorrect comparisons. A MFLOPS value is a rate (Z/t). AM of rates ∝ sum(1/tᵢ), not inversely proportional to Σtᵢ. Use HM for rates.
SPEC's Choice: Geometric Mean
SPEC uses GM for benchmark scores because: (1) Consistent regardless of reference machine; (2) Less biased by outliers than HM or AM; (3) Performance ratios follow lognormal distributions — GM is the back-transformed average of a lognormal.

§2.5 · GM Properties & Consistency

Equal Weight Property (GM)

A 10% change in any data value produces the same relative change in GM: GM × ⁿ√1.1. In contrast, AM is dominated by large values — a 10% change in the largest value dominates; a 10% change in the smallest is negligible.

GM of Ratios (Key Identity)
GM(Zᵢ/tᵢ) = GM(Zᵢ) / GM(tᵢ) … (2.8)
Consistency Property

GM gives the same relative ranking of systems regardless of which system is used as the normalization reference. AM does NOT have this property — changing the reference machine can change the ranking.

Example 2.6 — Non-Monotonic AM vs GM
Data SetAMGM
Set c (uniform 1–11)6.005.95 (lower AM, higher GM)
Set d (large-number bias)7.556.42

AM(d) > AM(c) but GM(c) > GM(d) — GM and AM can disagree on ordering (GM is non-monotonic w.r.t. AM).

HM Outlier Sensitivity
HM is highly sensitive to small-value outliers. A single very small value drives HM far below the "central" tendency of the data — misleading for skewed distributions.

§2.6 · Benchmark Principles

MIPS and MFLOPS are inadequate for cross-architecture comparisons due to ISA differences. The solution: run the same programs on different machines and compare execution times.

CISC vs RISC — Why MIPS Fails
Example 2.8
Statement: A = B + C (all in main memory)
CISC: 1 instruction → add mem(B), mem(C), mem(A)
RISC: 4 instructions → load, load, add, store
Same wall-clock time → CISC=1 MIPS, RISC=4 MIPS — but equal real-world work!
Desirable Benchmark Properties (Weicker 1990)
  1. Written in a high-level language — portable across different machines
  2. Representative of a particular programming domain (systems, numerical, commercial)
  3. Can be measured easily — clear start/end, reproducible results
  4. Has wide distribution — accepted by industry and academic communities
Benchmark Benefits
  • Guides customers deciding which system to buy
  • Helps vendors/designers identify performance targets
  • Enables reproducible, objective comparisons across platforms

§2.6 · SPEC Organization & CPU2006

SPEC = Standard Performance Evaluation Corporation — industry consortium defining and maintaining standardized benchmark suites since late 1980s.

SPEC CPU2006 — Industry Standard for CPU-Intensive Apps
  • 5th generation SPEC CPU suite (replaced CPU2000, CPU95, CPU92, CPU89)
  • 12 integer benchmarks (CINT2006) — C and C++ programs
  • 17 floating-point benchmarks (CFP2006) — C, C++, and Fortran programs
  • Over 3 million lines of code total
  • All benchmarks drawn from real-life applications (not synthetic loops)
  • Reference machine: Sun Ultra Enterprise 2, 296-MHz UltraSPARC II (1997)
  • A conforming run takes ~12 days on the reference machine
SPEC Key Terms
TermDefinition
BenchmarkHigh-level language program, compilable on any platform implementing the compiler
System Under Test (SUT)The system being evaluated
Reference MachineBaseline system; establishes reference times for all benchmarks
Base MetricRequired; standard compiler with default settings — comparable results
Peak MetricOptional; maximum optimization allowed (different flags per benchmark, feedback-directed optimization)
Speed MetricTime to complete a single task — single-task comparison
Rate MetricTasks completed per unit time (throughput); exploits multiple processors

§2.6 · SPEC Evaluation Process (Fig. 2.7)

Three-Step Calculation for Integer Benchmarks
  1. Run each program 3 times — select the median runtime. (Three runs eliminate variation from disk access, OS kernel scheduling artifacts.)
  2. Normalize each result — compute the ratio: reference time / SUT time:
    rᵢ = T_ref_i / T_sut_i … (2.9)
    Higher ratio = faster machine. T_ref is the execution time on the reference machine.
  3. Compute Geometric Mean of all 12 (integer) or 17 (FP) ratios:
    r_G = (∏ rᵢ)^(1/12) [for CINT2006]
Four Integer Score Variants
MetricTuningMeasure
SPECint2006PeakSpeed (single task)
SPECint_base2006BaseSpeed (single task)
SPECint_rate2006PeakThroughput (multi-task)
SPECint_rate_base2006BaseThroughput (multi-task)
Why Geometric Mean for SPEC?
Consistent across reference machines, less biased by outliers, matches lognormal distribution of performance ratios.

Chapter 2 — Key Takeaways

§2.1 — Design for Performance
  • Microprocessor speed has grown via Moore's Law + architectural tricks (pipeline, superscalar, speculation)
  • Memory & I/O cannot keep pace → performance balance problem → cache hierarchy, wider buses
  • Clock scaling limited by power/RC-delay → shift to multicore
§2.2 — Parallel Architectures
  • Multicore: 2–16+ cores, shared/private cache hierarchy, homogeneous
  • MIC: 50+ cores, requires massively parallel software
  • GPGPU: hundreds of simple cores optimized for data-parallel throughput
§2.3 — Laws
  • Amdahl's Law: Speedup = 1/[(1–f)+f/N]; sequential fraction (1–f) is the hard ceiling
  • Little's Law: L = λW; simple, universal, assumption-free queuing relationship
§2.4 — Metrics
  • Execution time (T = I_c × CPI × τ) is the most honest metric
  • MIPS is ISA-dependent; MFLOPS is workload-dependent
§2.5 — Means
  • AM for times, HM for rates, GM for normalized ratios
  • AM ≥ GM ≥ HM; SPEC uses GM for consistency & outlier robustness
§2.6 — SPEC
  • CPU2006: 12 integer + 17 FP benchmarks from real applications
  • Run 3×, take median → normalize to reference machine → GM of ratios
  • Base (default) and Peak (optimized) variants; Speed and Rate variants
