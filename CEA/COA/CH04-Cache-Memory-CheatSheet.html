§4.1 Memory System Characteristics

Key Characteristics (Table 4.1)
CharacteristicValues
LocationInternal (registers, cache, RAM) / External (disk, tape)
CapacityWords or Bytes; block size for external
Unit of TransferWord (internal) / Block (external, larger)

Physical TypeSemiconductor · Magnetic · Optical
CharacteristicsVolatile/Nonvolatile · Erasable/Nonerasable
Performance Parameters
  • Access time (latency): Time from address presented → data available
  • Cycle time: Access time + recovery time before next access (system bus concern)
  • Transfer rate: = 1/cycle time (random) or Tn = TA + n/R (non-random)
T_n = T_A + n/R // T_A = avg access time, n = bits, R = transfer rate (bps)
Access Methods Compared
  • Sequential: Linear order only (tape); variable time
  • Direct: Shared R/W + direct to vicinity + sequential search (disk)
  • Random: Any location independently; constant time (RAM, some cache)
  • Associative: Match on content, not address; simultaneous all-word compare (cache)

The hierarchy works because of locality of reference — the processor accesses lower levels far less frequently, making small fast upper levels sufficient.

§4.1 Locality of Reference

Definition (Denning, 1968)
Memory references by the processor tend to cluster. Over a short period, the processor works with fixed clusters; over a long period, clusters change.
Two Types of Locality
TypeDefinitionExploited By
TemporalRecently accessed locations likely accessed again soonKeep in cache (LRU, etc.)
SpatialLocations near a referenced location likely accessed soonLarger cache lines; prefetching
Why Locality Exists
  • Sequential execution: next instruction follows last (except branches/calls)
  • Iterative loops: small instruction set repeated many times
  • Subroutines: program stays in narrow procedure-invocation depth
  • Data structures: arrays and tables accessed in clustered fashion
Example: Two-Level Memory (Example 4.1)
Worked Example
L1: 1000 words, T₁ = 0.01 ms  |  L2: 100,000 words, T₂ = 0.1 ms
Hit ratio H = 0.95:
Tavg = 0.95×0.01 + 0.05×(0.01+0.1) = 0.0095 + 0.0055 = 0.015 ms
≈ 7× closer to L1 speed than L2 speed
Disk Cache Benefits
  • Clusters disk writes → fewer large transfers → better disk performance
  • Data pending write-out retrieved from software cache at memory speed

§4.2 Cache Memory Principles

Core Concept
Cache combines access time of expensive fast memory with the large size of slower cheap memory. Small fast cache holds copies of recently used main memory blocks.
Structure Terminology
TermDefinition
BlockFixed-size unit of main memory (K words); M = 2ⁿ/K total blocks
LineCache storage unit; holds K words of data + tag + control bits
TagPortion of memory address identifying which block occupies a line
Line sizeLength of line excluding tag and control bits
Dirty bitControl bit; 1 = line modified since loaded (write-back)
Cache Read Operation Flow
  1. CPU generates Read Address (RA)
  2. Check if block containing RA is in cache
  3. HIT: Deliver word directly to CPU (cache-speed access)
  4. MISS: Access main memory → load block into cache line
  5. Deliver word from cache line to CPU
Hardware Transparency
Cache is invisible to programmer and typically to the OS. It is a hardware staging device between main memory and processor registers.
Cache Hit Organization

On HIT: data/address buffers disabled; communication only between CPU and cache — no system bus traffic. On MISS: address put on bus, data returned through data buffer to both cache and CPU.

§4.2 Hit Ratio & Access Time

Key Definitions
SymbolMeaning
HHit ratio = fraction of accesses found in cache (M1)
(1–H)Miss ratio = fraction not found in cache
T₁ = T_cCache access time
T₂ = T_mMain memory access time
T_sAverage (system) access time
Primary Formula — Simultaneous Load (Eq. 4.2)
T_s = H·T₁ + (1-H)·(T₁ + T₂) = T₁ + (1-H)·T₂ // Assumes miss → T₁ + T₂ (load to cache then deliver)
Alternative Form — Sequential Load
T_avg = h·T_c + (1-h)·T_m // Some texts omit T_c on miss (assumes re-execute) // T_avg = T_c + (1-H)·T_m [Eq. 4.2 variant, Prob 4.26]
Access Efficiency (Eq. 4.4)
T₁/T_s = 1 / [1 + (1-H)·(T₂/T₁)] // r = T₂/T₁; on-chip cache: r ≈ 25–50 // off-chip L2 cache: r ≈ 5–15 // disk vs main memory: r ≈ 1000
Performance Target
For T_s ≈ T₁, need hit ratio H close to 1. With r = T₂/T₁ = 25–50 (typical on-chip), need H ≥ ~0.9 for good efficiency.
Cost Formula (Eq. 4.3)
C_s = (C₁·S₁ + C₂·S₂) / (S₁ + S₂) // C_s ≈ C₂ requires S₁ << S₂ // Works because locality → high H with small S₁
Two-Level Cache (Problem 4.27)
T_a = H₁·T_c1 + (1-H₁)·H₂·T_c2 + (1-H₁)·(1-H₂)·T_m // H₁ = L1 hit ratio, H₂ = combined L1+L2 hit ratio

§4.3 Direct Mapping

Core Rule
Each main memory block maps to exactly one cache line. Simplest and cheapest to implement.
Mapping Formula
i = j mod m // i = cache line number // j = main memory block number // m = number of lines in cache
Address Structure (s+w bits total)
TAG (s−r bits)
Most significant
LINE (r bits)
Identifies cache line
WORD (w bits)
Offset in block
Parameter Summary
Address length = s + w bits Addressable units = 2^(s+w) words/bytes Block/line size = 2^w words/bytes Blocks in memory = 2^s Lines in cache = m = 2^r Cache size = 2^(r+w) words/bytes Tag size = s − r bits
Worked Example (Fig 4.10) — 24-bit address, 64kB cache, 4B blocks
Example 4.2a
Cache: 16K = 2¹⁴ lines (r=14), block=4B (w=2), memory=16MB (24-bit addr, s=22)
Tag = 8 bits (s−r = 22−14), Line = 14 bits, Word = 2 bits
Address 16339C: Tag=16 (hex), Line=0CE7, Word=0
Line 0 holds blocks: 000000, 010000, …, FF0000
Disadvantage: Thrashing
If two heavily-used blocks map to the same line, they are continually swapped → low hit ratio. Fixed mapping = no flexibility.

§4.3 Associative (Fully) Mapping

Core Rule
Any main memory block can be loaded into any cache line. Maximum flexibility; hardware-expensive (parallel tag search).
Address Structure (s+w bits total)
TAG (s bits)
Uniquely identifies memory block
WORD (w bits)
Offset in block
Key Difference
No LINE field — cache logic must simultaneously examine every line's tag for a match (content-addressable memory / CAM).
Parameter Summary
Address length = s + w bits Addressable units = 2^(s+w) words/bytes Block/line size = 2^w words/bytes Blocks in memory = 2^s Lines in cache = undetermined (no address field) Tag size = s bits
Worked Example (Fig 4.12) — same 24-bit system
Example 4.2b
Same system: 16MB memory (24-bit), 4B blocks (w=2), so s=22
Tag = 22 bits, Word = 2 bits
Address 16339C: binary 0001 0110 0011 0011 1001 1100
Tag (22 MSBs) = 058CE7 hex, Word = 0
Any line can hold this block — no line field needed.
Disadvantage: Complex Hardware
All cache line tags must be compared in parallel. Requires CAM (Content Addressable Memory) — expensive for large caches. Full replacement flexibility available.

§4.3 Set-Associative Mapping

Core Rule
Cache divided into v sets, each with k lines. Block maps to a specific set (direct), but can go into any line within that set (associative). Best of both worlds.
Mapping Formulas
m = v × k (total lines = sets × lines-per-set) i = j mod v (set number for block j) // i = cache set number, j = main memory block number // m = total lines, v = sets, k = ways (lines/set)
Address Structure (s+w bits total)
TAG (s−d bits)
Identifies block within set
SET (d bits)
v = 2^d sets
WORD (w bits)
Offset
Parameter Summary
Address length = s + w bits Block/line size = 2^w words/bytes Blocks in memory = 2^s Lines per set = k (ways) Number of sets = v = 2^d Total cache lines = m = k × 2^d Cache size = k × 2^(d+w) words/bytes Tag size = s − d bits
Worked Example (Fig 4.15) — 2-Way Set-Associative
Example 4.2c
Same system: 24-bit addr, 4B blocks (w=2), 16K lines, k=2 ways
v = 16K/2 = 8192 = 2¹³ sets → d=13, Tag = 24−13−2 = 9 bits
Address structure: [9-bit Tag | 13-bit Set | 2-bit Word]
Set 0 holds any of: blocks 000000, 008000, …, FF8000
Both lines in set examined in parallel for tag match.

Hit Ratio vs Associativity (GCC Simulation, Fig 4.16)
  • Direct → 2-way: significant improvement up to 64kB cache size
  • 2-way → 4-way: modest additional improvement (most pronounced at 4kB)
  • Beyond 4-way: diminishing returns; better to increase cache size
  • Beyond ~32kB cache: increasing size yields no significant performance gain
Victim Cache (Reducing Direct-Map Conflicts)

Small fully-associative cache (4–16 lines) between direct-mapped L1 and next memory level. Stores recently evicted lines for fast re-use. Reduces conflict misses without full associativity overhead.

§4.3 Replacement Algorithms

Applies To
Only needed for associative and set-associative caches. Direct mapping has only one possible destination — no choice.
Four Common Algorithms
AlgorithmRuleImplementationPerformance
LRUReplace line least recently used (longest time with no reference)USE bit per line (2-way); list of indexes (fully assoc)Best (exploits temporal locality)
FIFOReplace line that has been in cache the longestRound-robin / circular bufferGood; simple
LFUReplace line with fewest referencesCounter per lineGood; moderate complexity
RandomPick any candidate line at randomTrivial hardwareOnly slightly worse than usage-based
LRU for 2-Way Set-Associative
Each line has a USE bit. On access to line A in set: USE(A)=1, USE(other)=0 On replacement: evict line with USE=0
Intel 80486 Pseudo-LRU (4-way)

3 bits (B0, B1, B2) per set of 4 lines (L0–L3):

  • B0: 0 → L0/L1 least recent, 1 → L2/L3 least recent
  • B1: used to choose within L0/L1
  • B2: used to choose within L2/L3
  • True LRU would need 6 bits (log₂(4!) = log₂(24) ≈ 4.6 → 6 bits)
Implementation Note
LRU is most popular due to best hit ratio and relatively simple implementation. All algorithms must be hardware-implemented for speed.

§4.3 Write Policy

The Two Write Policies
PolicyMechanismProsCons
Write-Through Every write goes to both cache AND main memory simultaneously Main memory always valid; easy coherency; simple High memory traffic; potential bus bottleneck
Write-Back Write only to cache; set dirty bit. Write to memory only on eviction (if dirty) Minimizes memory writes; less bus traffic Main memory may be stale; complex I/O; coherency issues
Write-Back Dirty Bit Logic
On write: set dirty_bit = 1 for that cache line On eviction: IF dirty_bit = 1 THEN write line back to memory IF dirty_bit = 0 THEN discard (memory already valid)
Example 4.3 — Write-Back Break-Even
When is Write-Back Better?
Line = 32 bytes = 8 words; write time/word = 30 ns
Write-back eviction: 8 × 30 = 240 ns (once per eviction)
Write-through: 30 ns per update
Break-even: 240/30 = 8 writes per line before eviction
→ Write-back wins if avg writes per line > 8
Cache Coherency Problem

When multiple CPUs/I/O share memory, a write in one cache invalidates others.

Coherency Solutions
  • Bus watching (write-through): Controllers monitor bus writes and invalidate matching entries
  • Hardware transparency: Writes propagate to all caches automatically
  • Noncacheable memory: Shared memory excluded from caching (always misses)
Write Frequency
Writes ≈ 15% of memory references (general); up to 33–50% for HPC (vector/matrix).

§4.3 Line Size

Design Trade-off
Larger lines exploit spatial locality but reduce the number of lines in the cache, which can hurt performance by evicting useful data.
Effect of Block Size on Hit Ratio
  • Small → medium size: Hit ratio ↑ (more spatial locality exploited)
  • Very large size: Hit ratio ↓ (fewer blocks fit in cache; distant words unlikely needed)
  • Two competing effects: (1) Fewer blocks → higher eviction rate; (2) Each word farther from reference → less likely needed
Practical Guidelines
System TypeTypical Line Size
General purpose8 to 64 bytes (optimal range per studies)
HPC systems64 and 128 bytes (most common)
Pentium 4 L1 D-cache64 bytes
Pentium 4 L2/L3 cache128 bytes
No Universal Optimum
Optimal line size depends on locality characteristics of specific program workload. No single value is optimal for all applications.
Miss Penalty with Block Transfer (Problem 4.28)
// First-word: T_first, each additional: T_each // Line = n words: Miss penalty (non-burst) = 1 + n × T_access Miss penalty (burst) = 1 + T_first + (n-1) × T_each

§4.3 Number of Caches — Multi-Level & Split

Multi-Level Cache Rationale
  • On-chip L1: Eliminates external bus for most accesses; fastest possible; internal data paths much shorter
  • L2 (off/on-chip): Catches L1 misses before slow DRAM; SRAM matches bus speed (zero wait-state)
  • L3: Added for large database applications; now on-chip in most modern CPUs
L2 Effectiveness Rule
L2 Sizing
L2 must be at least 2× L1 size to significantly improve total hit ratio. If same size, contents mirror L1 — no benefit.
Modern Hierarchy
LevelLocationSize Range
L1On-chip16–128 kB
L2On-chip (now)256 kB–4 MB
L3On-chip (now)4–96 MB
L4Multi-chip (IBM)Shared
Unified vs. Split Cache
TypeDescriptionAdvantage
UnifiedSingle cache for both instructions and dataHigher hit rate (auto load-balancing); one design
Split (Harvard)Separate I-cache and D-cache at same levelEliminates contention between fetch/decode and execute units
Modern Trend
Split caches at L1 (eliminates pipeline stalls); unified caches at L2/L3. Critical for superscalar/pipelined designs that prefetch instructions while executing data operations simultaneously.
Contention Problem (Unified L1)

When instruction prefetcher and execution unit simultaneously request cache access, prefetcher is stalled → pipeline stall → performance loss. Split cache eliminates this entirely.

Intel Cache Evolution (Table 4.4)
  • 386: External cache only (off-chip)
  • 486: On-chip L1 (8kB unified) + external L2
  • Pentium: Split L1 (I+D) to eliminate contention
  • Pentium Pro: Backside bus for L2 (dedicated, faster)
  • Pentium II: L2 moved on-chip
  • Pentium III: External L3 added
  • Pentium 4: L3 moved on-chip

P4 Innovation
Unlike all previous designs, the P4 instruction cache sits after the decode logic — it stores decoded micro-operations, not raw x86 instructions. Decoding is done separately, enabling better superscalar scheduling. 256-bit interface to execution units.

Processor Core Components
  • Fetch/Decode unit: Fetches instructions from L2, decodes to micro-ops, stores in L1 I-cache
  • Out-of-order execution logic: Schedules micro-ops based on data dependencies; speculative execution
  • Execution units: Execute micro-ops using L1 D-cache; results in registers
  • Memory subsystem: L2 + L3 caches + system bus (64-bit) for L1/L2 misses and I/O
Write Policy
Default: Write-Back
Data written to main memory only on eviction when dirty. Can be dynamically configured to write-through via control registers (CD/NW bits).
Special Cache Instructions
  • INVD — Invalidate (flush) internal cache; signal external cache to invalidate
  • WBINVD — Write-back then invalidate internal; write-back then invalidate external
Data Path Widths
  • L1 D-cache ↔ Execution units: 256-bit internal bus
  • System bus (main memory): 64-bit
  • Both L2 and L3 are eight-way set-associative, 128-byte lines

Complete Formula Set
// Average access time (Eq 4.2): T_s = T₁ + (1-H) × T₂ // Access efficiency (Eq 4.4): T₁/T_s = 1 / [1 + (1-H) × (T₂/T₁)] // Average cost per bit (Eq 4.3): C_s = (C₁×S₁ + C₂×S₂) / (S₁ + S₂) // N-level generalization (Problem 4.16): T_s = T₁ + (1-H₁)T₂ + (1-H₁)(1-H₂)T₃ + ... = sum_{i=1}^{N} T_i × product_{j=1}^{i-1}(1-H_j)
Locality Effects on Hit Ratio
  • No locality: H = S₁/S₂ (linear function of relative size)
  • Moderate locality: H > S₁/S₂ (sub-linear improvement)
  • Strong locality: H ≫ S₁/S₂ (very small M1 achieves high H)
  • Empirical: 1K–128K word cache yields H > 0.75 regardless of total memory size
Performance Goals
GoalRequirement
T_s ≈ T₁ (fast)H close to 1; strong locality required
C_s ≈ C₂ (cheap)S₁ ≪ S₂
Both simultaneouslyRequires strong locality — the fundamental justification for cache hierarchy
Key Insight
Strong locality means a small M1 achieves both goals simultaneously. The same principle applies to cache, virtual memory, and disk cache.

Key Terms & Quick Reference

Essential Vocabulary
  • Cache hit: Requested data found in cache
  • Cache miss: Not found; must fetch from memory
  • Cache line: Basic unit of cache storage (K words + tag + control)
  • Cache set: Group of lines in set-associative
  • Tag: Address bits identifying block in a line
  • Hit ratio (H): Fraction of accesses that hit
  • Miss ratio: 1 − H
  • Dirty bit: Indicates line modified (write-back)
  • Thrashing: Repeated eviction of needed blocks (direct map)
  • Temporal locality: Recently used → used again soon
  • Spatial locality: Nearby addresses → used soon
  • L1 cache: Level 1, on-chip, fastest
  • L2 cache: Level 2, larger, slower
  • L3 cache: Level 3, largest on-chip
  • Logical cache: Uses virtual addresses
  • Physical cache: Uses physical addresses
  • Unified cache: Combined I+D cache
  • Split cache: Separate I-cache and D-cache
  • Write-through: Write to cache and memory
  • Write-back: Write to cache only; memory on eviction
  • Victim cache: Small fully-assoc buffer for evicted lines
  • MMU: Memory management unit (virtual→physical)

Critical Design Relationships
  • Direct: i = j mod m (line = block mod total-lines)
  • Set-assoc: i = j mod v (set = block mod num-sets); m = v × k
  • Direct is special case of set-assoc: k=1, v=m
  • Fully-assoc is special case: k=m, v=1

  • Write% ≈ 15% general; up to 50% HPC workloads
