§17.1 — Multiple Processor Organizations

Flynn's Taxonomy

ClassFull NameDescriptionExamples
SISDSingle Instruction, Single DataClassical uniprocessor — one instruction stream operates on one data streamTraditional scalar CPU
SIMDSingle Instruction, Multiple DataOne instruction operates on multiple data elements in parallel; exploits data-level parallelismGPU, SSE/AVX, vector processors
MISDMultiple Instruction, Single DataMultiple instruction streams operate on the same data; rarely used in practiceSystolic arrays (some fault-tolerant systems)
MIMDMultiple Instruction, Multiple DataMultiple processors, each with own instruction stream and data stream; most general modelSMP, clusters, multicore
Dominant Model
Flynn's taxonomy — MIMD is the dominant architecture in modern parallel systems. It encompasses both shared-memory (SMP) and distributed-memory (cluster/MPP) organizations.

MIMD: Shared-Memory

  • Single physical memory accessible to all processors
  • Communication via shared memory reads/writes
  • Requires cache coherence protocol to maintain consistency
  • SMP = Uniform Memory Access (UMA)
  • NUMA = Non-Uniform Memory Access
Programming
Easiest to program; shared address space means no explicit message passing needed.

MIMD: Distributed-Memory

  • Each processor has its own local memory
  • Communication via message passing over interconnect
  • No global address space shared between nodes
  • Examples: Clusters, MPP (Massively Parallel Processors)
Scalability
Scales to thousands of nodes; no memory bus bottleneck.

Memory Organization Models

ModelFull NameAccess TimeDescription
UMA Uniform Memory Access Equal latency All processors access all memory with equal latency; classic SMP architecture; single memory bus or crossbar
NUMA Non-Uniform Memory Access Varies by location Memory physically distributed among processors; local memory is fast, remote memory is slower; CC-NUMA adds coherence
COMA Cache-Only Memory Architecture Migration-based All local memory acts as cache; data migrates to where it is used; no fixed home location; complex directory management
§17.2 — Symmetric Multiprocessors (SMP)

SMP Defining Characteristics

  1. Two or more similar processors of comparable capability
  2. Share main memory and I/O facilities via a common bus or interconnect
  3. All processors have equal access to I/O devices
  4. All processors can perform the same functions (symmetric)
  5. Single integrated OS manages all processors and resources

SMP Advantages

  • Performance: Parallel execution of concurrent tasks
  • Availability: If one processor fails, others continue
  • Incremental growth: Add processors to increase throughput
  • Scaling: Products can be offered with different processor counts
  • Transparent to user: OS manages all processors seamlessly
Note
These are potential benefits; realizing them depends on OS and application design.

SMP Bus / Interconnect

  • Time-shared bus: Simple; all processors share single bus; bottleneck at scale
  • Multiple buses: Reduce contention; more complex arbitration
  • Crossbar switch: Full non-blocking connectivity; expensive
  • Each CPU typically has L1 + L2 + L3 cache hierarchy
Cache Coherence Problem
Local caches can hold stale copies of shared data — hardware must maintain coherence across all caches.

OS Design Considerations for SMP

Concurrency & Scheduling
  • Simultaneous concurrent processes: OS kernel must be reentrant
  • Scheduling: Assign processes to specific processors or pool
  • Synchronization: Mutual exclusion and event management across CPUs
Memory & Reliability
  • Memory management: Protection and paging across shared memory
  • Reliability & fault tolerance: Graceful degradation on CPU failure
  • Process migration: Move processes between processors for load balance

Amdahl's Law

Speedup(N) = 1 / ((1 - f) + f/N) N = number of processors f = fraction of code parallelizable (1-f) = sequential fraction
  • As N → ∞, max speedup = 1/(1−f)
  • f = 0.95 → max speedup = 20×
  • f = 0.90 → max speedup = 10×
  • Small sequential portions dominate at large N
Implication
Sets theoretical ceiling on parallel speedup; sequential fraction is the fundamental bottleneck.
§17.3 — Cache Coherence & MESI Protocol

Cache Coherence Problem

In SMP systems each processor has private caches. Multiple cached copies of the same memory block can diverge, making the system incoherent — different processors see different values for the same address.

  • Write-back policy: Updated value stays in cache; other caches hold stale copy until writeback
  • Write-through policy: Updates propagate to memory, but other CPU caches may not see it until they reload
Hardware Solution Required
Software-only solutions are too slow; hardware snoopy or directory protocols are used.

Coherence Approaches

📀 Software Approach
  • Compiler marks shared data as non-cacheable
  • Compiler inserts flush/invalidate instructions at synchronization points
  • No hardware overhead; significant performance cost
⚙️ Hardware Approach
  • Directory protocols: Central directory tracks which caches hold each block
  • Snoopy protocols: Each cache controller monitors (snoops) the shared bus
  • Transparent to software; preferred for SMP

Snoopy Protocol Strategies

❌ Write-Invalidate (used by MESI)
  • On write: invalidate all other cached copies
  • Next access by other CPU = cache miss → fetch from memory/writer
  • Fewer bus transactions for repeated writes to same location
  • Preferred for most shared-memory systems
📢 Write-Update (Broadcast)
  • On write: broadcast updated value to all caches holding the block
  • All copies immediately up-to-date
  • More bus traffic; good for frequently-read shared data

MESI Protocol — Four States

StateNameDirty/CleanOther Caches Hold It?Action on Access
M Modified Dirty No — only this cache Writeback to memory before allowing any other access; processor can read/write freely
E Exclusive Clean No — only this cache Processor can write → transitions to M silently (no bus transaction needed)
S Shared Clean Yes — multiple caches Read is a hit; to write, must issue BusUpgr to invalidate other copies → transitions to M
I Invalid N/A Possibly (unknown) Any access = cache miss; must issue BusRd or BusRdX to fetch block; state determined by snoop responses
Encoding
Only a 2-bit state field per cache line is needed to encode all four MESI states.

Bus Transactions
BusRd — request to read a block
BusRdX — read with intent to modify (RWITM); invalidates all copies
BusUpgr — upgrade S to M by invalidating all other sharers
Why MESI over MSI?
The Exclusive state avoids a bus transaction when a processor writes to a block it loaded but no other cache has touched. Without E, every write from S would need BusUpgr even if the processor is the sole holder.
§17.4 — Multithreading & Chip Multiprocessing (CMP)

Thread & Process Concepts

  • Process: Program in execution; owns address space, file handles, OS resources; expensive context switch
  • Thread: Dispatchable unit within a process; shares address space; cheaper switch (no TLB flush)
  • Thread switch may not require kernel mode; saves/restores only register state + PC
  • Process switch: Full OS involvement; flush TLB, save entire process state
  • Multiple threads → ILP + TLP (thread-level parallelism)
MIPS Rate = f × IPC f = clock frequency (MHz/GHz) IPC = instructions per clock cycle

Multithreading Approaches

ApproachAlso Known AsMechanismBest ForTrade-off
Interleaved MT Fine-grain / Barrel processing Switch thread every cycle; zero-cycle switch; pipeline always sees a ready instruction Hiding ALL pipeline latencies (cache miss, branch, etc.) Single-thread latency increases; needs many threads to fill pipeline
Blocked MT Coarse-grain MT Run one thread until it stalls (e.g., L2/L3 miss); switch to another thread Hiding long-latency memory stalls Short stalls not hidden; pipeline must flush on switch
Simultaneous MT SMT / Hyper-Threading (Intel) Issue instructions from multiple threads in the same clock cycle; fills issue slots from any ready thread Wide superscalar with diverse workloads; high ILP + TLP Shared resources; one thread can starve another; cache pollution
Chip Multiprocessing CMP / Multicore Multiple complete processor cores on one die; each core is independent True parallelism; thread isolation; power efficiency vs. frequency scaling Per-core performance lower than single high-frequency core; more complex coherence

SMT — Hyper-Threading Detail

  • Exploits the wide superscalar pipeline that is underutilized by a single thread
  • Each SMT core holds multiple thread contexts (register files, PC, status)
  • OS sees 1 physical CPU = multiple logical CPUs
  • Each cycle, issue logic selects instructions from any ready thread
  • Blocked threads' slots are transparently filled by other threads' instructions
  • Shared: execution units, last-level cache, TLB
Performance Gain
~20–30% throughput improvement over single-thread mode on typical workloads; highly workload-dependent.

SMT vs CMP Comparison

FeatureSMT (Hyper-Threading)CMP (Multicore)
Physical structureOne physical core, multiple thread contextsMultiple independent physical cores on one die
Thread isolationWeak — share execution units & cachesStrong — each core has private L1/L2
ILP exploitationVery high — fills entire wide issue widthModerate — per-core ILP limited
Latency hidingExcellent — fills stall cycles from other threadsNo — stalls affect that core only
Single-thread perf.Slightly reduced (resource sharing)Full single-thread performance
ScalabilityLimited (2–4 threads per core typical)Scales well (2, 4, 8, 16+ cores)
Modern Practice
Modern CPUs combine CMP + SMT: e.g., 8-core CPU with 2 SMT threads per core = 16 logical processors.
§17.5 — Clusters

Cluster Definition & Benefits

A cluster is a group of interconnected, whole computers working together as a unified computing resource that can create the illusion of a single machine. Each node runs its own OS instance.

  • Availability: If one node fails, the cluster survives (failover)
  • Performance: Parallel workloads distributed across nodes
  • Scalability: Add nodes incrementally as demand grows
  • Cost: Built from commodity x86 servers; excellent price/performance
Failover / Failback
Failover: Surviving nodes take over workload of failed node automatically.
Failback: When failed node recovers, workload migrates back.

Cluster Configurations

🔒 Passive Standby

One active node does all work; standby is idle. On failure, standby takes over. Simple but wastes standby resources.

🔙 Shared Nothing

Each node owns and manages partitioned disk volumes. No shared storage; high availability through replication. Most common for databases (e.g., Oracle RAC alternative).

💾 Shared Disk

All nodes can access a common disk array. Requires distributed lock manager to prevent write conflicts. Higher availability; more complex.

⚡ Active / Mutual Takeover

All nodes are active and serve different workloads simultaneously. On failure, surviving nodes take over failed node's work. Best resource utilization.

Blade Servers

  • Blade modules slide into a common chassis
  • Each blade = complete server (CPU, RAM, NIC, sometimes local disk)
  • Chassis provides shared power, cooling, networking, management
  • Very high density: dozens of servers in 10U chassis
  • Interconnect: 100GbE / 40GbE backplane or InfiniBand
Cable Reduction
Shared chassis backplane dramatically reduces cabling complexity vs. rack-mount servers; also reduces power and cooling overhead.

Cluster Middleware & Single System Image

  • Single system image (SSI): Hide individual nodes; appear as one machine
  • Single entry point: One address/interface to the entire cluster
  • Single file hierarchy: Global namespace for all files across nodes
  • Single control point: Centralized admin for monitoring & management
  • Single virtual network: All nodes share a common logical network
  • Single memory space: Distributed shared memory abstraction
  • Single job management: Global scheduler assigns work to nodes

Parallel Programming Models

  • Parallelizing compiler: Automatically extracts loop/data parallelism; transparent to programmer
  • Parallelized application (MPI): Programmer explicitly uses MPI calls to communicate between nodes
  • Parametric computing: Many independent runs of same program with different parameters (embarrassingly parallel)
  • Message passing (MPI/PVM): Explicit send/receive; portable across clusters and MPPs

Clusters vs SMP Comparison

§17.6 — NUMA (Non-Uniform Memory Access)

NUMA Motivation & Organization

NUMA bridges the gap between tightly-coupled SMP and loosely-coupled clusters. It provides a single global address space while distributing physical memory across processor nodes for scalability beyond what a single bus can support.

CC-NUMA (Cache-Coherent NUMA)
  • Multiple processors per node + per-node directory
  • Single system-wide address space visible to all processors
  • Local memory: Fast (same as SMP latency)
  • Remote memory: Slower (crosses interconnect)
  • Hardware directory tracks block ownership across nodes
CC-NUMA Node Structure: [ CPU0 CPU1 ] <--- Local L1/L2 caches [ Local Memory ] <--- Fast access [ Directory ] <--- Tracks remote sharers | [ Interconnect ] <--- High-speed fabric | [ Other Nodes... ] <--- Remote (slower) access

NUMA Advantages, Disadvantages & Comparison

✔ Advantages
  • Scalable beyond SMP limits (32–256+ processors)
  • Transparent shared address space (no MPI needed)
  • Local memory accesses remain fast
  • NUMA-aware OS can pin threads near their data
❌ Disadvantages
  • Remote accesses degrade performance significantly
  • Directory overhead (memory + bandwidth)
  • OS must be NUMA-aware for best performance
  • Applications must be tuned for data locality

§17.7 — Cloud Computing

Cloud Essential Characteristics (NIST)

  • Broad network access: Capabilities available over the network; accessible via standard mechanisms (phones, tablets, laptops)
  • Rapid elasticity: Resources can be provisioned and released quickly; appear unlimited to consumer
  • Measured service: Usage automatically monitored, controlled, reported — pay-per-use
  • On-demand self-service: Consumer provisions compute, storage without human interaction with provider
  • Resource pooling: Provider pools resources; multi-tenant model; consumer unaware of exact physical location
Authority
NIST SP 800-145 is the authoritative definition of cloud computing used industry-wide.

Cloud Service Models

🏢 IaaS — Infrastructure as a Service

Consumer gets: raw compute, storage, network. Consumer controls: OS, middleware, applications, data. Provider controls: physical hardware, hypervisor.

AWS EC2 Azure VMs Google Compute Engine

⚽ PaaS — Platform as a Service

Consumer deploys apps on managed platform. Consumer controls: app code and configuration. Provider manages: runtime, OS, servers, storage.

Google App Engine Heroku Azure App Service

💻 SaaS — Software as a Service

Consumer uses hosted applications via browser/API. Consumer controls: only app-specific configuration. Provider manages everything.

Gmail Office 365 Salesforce

Cloud Deployment Models

🌐 Public Cloud

Owned and operated by cloud provider; available to general public. Low cost Security concern

🏠 Private Cloud

Exclusive use by one organization; on-premises or hosted. Better control Higher cost

🏛 Community Cloud

Shared by organizations with common concerns (compliance, mission, policy). Cost-shared; meets sector-specific needs (e.g., government, healthcare).

⚡ Hybrid Cloud

Two or more distinct cloud infrastructures bound by standardized technology. Cloud bursting: overflow public workloads during peaks. Most flexible model.

Virtualization's Role in Cloud

  • Resource pooling via hypervisor (Type 1: VMware ESXi, Hyper-V; Type 2: VirtualBox)
  • Rapid elasticity: Spin up new VMs in seconds from templates
  • Multi-tenancy with isolation: Multiple customers on same physical host, isolated by hypervisor
  • Live migration: Move running VMs between physical hosts without downtime
  • Blade servers provide the high-density physical substrate for VM hosting
Fundamental Enabler
Without virtualization (hypervisors), the resource pooling and rapid elasticity characteristics that define cloud computing would not be achievable at scale.
