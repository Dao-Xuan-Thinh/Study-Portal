§7.1 External Devices

An external device attaches to the computer via a link to an I/O module. The link carries control, status, and data signals. Why not connect peripherals directly to system bus?

  • Wide variety of peripherals with different operation methods
  • Data transfer rate of peripherals often much slower than memory/CPU
  • Some peripherals are faster → mismatch causes inefficiency
  • Peripherals use different data formats and word lengths
Three Categories of External Devices
CategoryDescriptionExamples
Human-ReadableFor communicating with usersVDT, keyboard, printer
Machine-ReadableFor communicating with equipmentDisk, tape, sensors, actuators
CommunicationFor communicating with remote devicesModems, NICs, terminals
External Device Internals
  • Transducer — converts electrical ↔ physical energy (input/output)
  • Buffer — temporarily holds data during transfer (8–16 bits for serial; larger for block devices)
  • Control logic — controls device in response to I/O module commands

§7.2 I/O Module Functions

The I/O module interfaces the system bus to one or more peripheral devices. It hides the details of timing, formats, and electromechanics from the CPU.

Five Major Functions
  • Control & Timing — coordinates flow between internal resources and external devices; arbitrates bus access
  • Processor Communication — command decoding, data exchange, status reporting (BUSY/READY), address recognition
  • Device Communication — issues commands, monitors status, exchanges data with peripheral
  • Data Buffering — absorbs speed mismatch; buffers bursts from memory, sends at device rate
  • Error Detection — detects mechanical/electrical faults; checks parity bits; reports errors to CPU
I/O Module Structure (Figure 7.3)
  • Data Registers — buffer data transferred to/from the module
  • Status/Control Registers — report device state; accept processor commands
  • I/O Logic — interfaces control lines, address lines, data lines to system bus
  • External Device Interface Logic — one per controlled device; handles Data, Status, Control signals
Terminology
I/O Controller / Device Controller — primitive, requires detailed CPU control (microcomputers). I/O Channel / I/O Processor — high-level interface, executes its own programs (mainframes).

§7.3 Programmed I/O

CPU directly controls the I/O operation by issuing commands and busy-waiting (polling) on the status register until the device is ready. No interrupts are used.

Four I/O Command Types
  • Control — activate peripheral, tell it what to do (e.g., rewind tape, position disk head)
  • Test — check status (powered on? ready? errors?)
  • Read — obtain data from peripheral into I/O module's buffer; CPU retrieves via data bus
  • Write — send byte/word from data bus through I/O module to peripheral
Memory-Mapped vs Isolated (I/O-Mapped) I/O
FeatureMemory-MappedIsolated I/O
Address SpaceShared with memorySeparate I/O address space
Bus LinesSingle RD/WR linesSeparate MEM and I/O cmd lines
InstructionsFull memory instruction setSpecial IN/OUT instructions only
Address RangeMemory space reducedFull memory + full I/O range
AdvantageRich instruction set, efficient codeNo memory space wasted
Critical Drawback
CPU is tied up in a polling loop checking status — wastes processor cycles; transfer rate limited by CPU polling speed.

§7.4 Interrupt-Driven I/O

CPU issues I/O command then continues executing other instructions. The I/O module signals the CPU via an interrupt when it is ready. CPU saves context, runs the ISR, then restores context.

Interrupt Processing Sequence (Figure 7.6)
  1. Device issues interrupt signal to processor over control line
  2. Processor finishes current instruction before responding
  3. Processor tests for interrupt, sends ACK to device → device clears interrupt signal
  4. Processor saves PSW + PC onto the system control stack
  5. Processor loads PC with entry address of interrupt-handling routine (ISR)
  6. ISR saves all general-purpose registers onto stack (full process state)
  7. ISR processes the interrupt — reads data from I/O module, checks status, sends ACKs
  8. ISR restores all saved registers from stack
  9. ISR restores PSW and PC from stack → resumes interrupted program
Device Identification Techniques
  • Multiple Interrupt Lines — one line per device; impractical for many devices; each line may still have multiple devices
  • Software Poll — CPU branches to ISR, polls each I/O module via TESTIO command or status register; simple but slow
  • Daisy Chain (Hardware Poll / Vectored) — all modules share one interrupt line; ACK propagates down the chain; requesting module places vector on data bus; CPU uses vector as pointer to device-specific ISR
  • Bus Arbitration (Vectored) — I/O module wins bus before raising interrupt; only one active at a time; places vector on data lines after CPU ACK
Priority Schemes
  • Multiple lines → CPU picks highest priority line
  • Software poll → poll order determines priority
  • Daisy chain → physical position in chain = priority
  • Bus arbitration → explicit priority scheme in arbitration logic
Interrupt Types
  • Maskable — can be disabled by CPU; used for normal device I/O
  • Non-Maskable (NMI) — cannot be disabled; reserved for critical events (power failure, memory error)
  • Vectored — device supplies vector (address of ISR) on data bus
  • Non-Vectored — all interrupts go to one fixed address; ISR must poll to identify device

Intel 82C59A — PIC

The 82C59A Programmable Interrupt Controller manages interrupts for the Intel 80386, which has only a single INTR and INTA line.

Key Facts
  • Handles up to 8 I/O modules per chip
  • Cascade mode: up to 8 slaves × 8 devices = 64 devices total
  • Accepts interrupt requests → determines highest priority → raises INTR to CPU
  • CPU responds on INTA → 82C59A places vector on data bus
Programmable Priority Modes
  • Fully Nested — fixed priority IR0 (highest) … IR7 (lowest)
  • Rotating — equal-priority devices; serviced device receives lowest priority (round-robin)
  • Special Mask — CPU can inhibit specific device interrupts selectively
Intel 8255A — Programmable Peripheral Interface

General-purpose I/O module with 24 I/O lines in three 8-bit ports (A, B, C). Used for both programmed and interrupt-driven I/O.

  • Mode 0 — basic I/O; each port independently input or output
  • Mode 1 — strobe I/O; Port C lines = handshaking + IRQ control
  • Mode 2 — bidirectional; Port A as data, Port B as direction control
Key Signals
CS Chip Select · RD Read · WR Write · A0/A1 Port Select (00=A, 01=B, 10=C, 11=Control) · RESET → all ports set to input mode

§7.5 Direct Memory Access (DMA)

The DMA controller takes over bus control from the CPU to transfer entire blocks of data directly between I/O device and main memory — without CPU involvement during the transfer.

Why DMA?
Drawbacks of Programmed & Interrupt I/O
(1) Transfer rate limited by CPU polling/servicing speed. (2) Every word of data must pass through CPU registers — CPU tied up for each word of the entire block.
DMA Controller Registers (Figure 7.12)
  • Address Register — starting memory address for transfer
  • Data Count Register — number of words to transfer; decremented each cycle
  • Control Logic — manages R/W direction, device address, bus control signals
  • Data Register — temporary buffer for data in transit
DMA Configurations (Figure 7.14)
  • (a) Single-bus, detached DMA — DMA + I/O both on system bus; 2 bus cycles per word (slow)
  • (b) Single-bus, integrated DMA-I/O — DMA has private path to I/O modules; 1 bus cycle per word
  • (c) I/O bus — I/O modules connected to DMA via I/O bus; DMA uses system bus only for memory
Cycle Stealing
DMA pauses CPU for one bus cycle at a time (between instruction steps). Not a full interrupt — CPU does not save context. CPU executes more slowly but DMA completes transfer.

DMA Operation Steps (Intel 8237A)

The Intel 8237A DMA Controller interfaces to the 80x86 family and DRAM. It is a fly-by controller — data does not pass through the DMA chip itself (transfers between I/O port and memory only).

Transfer Sequence (Memory → Disk)
  1. Peripheral (disk controller) pulls DREQ high to request DMA service
  2. DMA asserts HRQ (hold request) on CPU's HOLD pin
  3. CPU finishes current bus cycle, asserts HLDA (hold acknowledge) — grants bus to DMA
  4. DMA asserts DACK (DMA acknowledge) → tells peripheral to begin data transfer
  5. DMA places memory address on address bus, activates MEMR (read from memory) + IOW (write to I/O); decrements count, increments address pointer; repeats until count = 0
  6. DMA deasserts HRQ → CPU regains bus control; DMA sends interrupt to notify completion
8237A Registers
RegisterPurpose
CommandEnable/disable DMA, memory-to-memory transfer (ch.0→ch.1), address hold
StatusD0–D3: TC (terminal count) reached per channel; D4–D7: pending DMA requests
ModeSelect channel; transfer direction (read/write/verify); single/block/demand/cascade mode; auto-init
Single MaskEnable/disable (mask) DREQ for a specific channel
All MaskEnable/disable all 4 channels at once
Address Reg ×4Starting memory address for each channel
Count Reg ×4Number of bytes to transfer for each channel
4 Independent Channels
8237A has 4 independently programmable channels (0–3). Cascade mode allows multiple 8237As to expand beyond 4 channels.

DMA Transfer Modes

Three Transfer Modes
Mode 1: Burst Mode
  • DMA seizes bus and transfers entire block before releasing
  • CPU is blocked for duration of transfer
  • Fastest total throughput
  • High CPU latency — not suitable for real-time systems
Mode 2: Cycle Stealing
  • DMA steals one bus cycle at a time from CPU
  • DMA inserts wait cycles between CPU memory accesses
  • CPU slows down but is not completely blocked
  • Most commonly used mode
Mode 3: Transparent / Interleaved
  • DMA transfers data only when CPU is not using the bus
  • No CPU slowdown at all
  • Most complex — requires monitoring CPU bus usage
  • Slowest DMA throughput; zero CPU impact
DMA Breakpoints in Instruction Cycle

DMA can steal cycles at any of these points: after Fetch Instruction, after Decode, after Fetch Operand, after Execute, or after Store Result. Interrupts can only occur at end of instruction.

Fly-By Limitation
8237A can only transfer between one I/O port and one memory address — not two memory locations or two I/O ports. Memory-to-memory uses a temporary register via two channels.

§7.6 Direct Cache Access (DCA)

DMA performance is insufficient for 10-Gbps / 100-Gbps Ethernet rates. DCA allows the I/O function to place/read data directly in the last-level cache (L2 or L3) — bypassing main memory bottlenecks.

Problem: Traditional DMA + Network I/O
  • NIC receives packet → DMA writes to main memory
  • Cache lines for new data are invalidated → cache miss when CPU processes packet
  • CPU stalls on memory reads → performance degrades under heavy network load
  • Data copied between system buffer ↔ application buffer wastes cycles
Intel Direct Data I/O (DDIO) — Xeon E5 Family
Input (NIC → Host)
  1. NIC initiates memory write; invalidates cache lines for system buffer
  2. DDIO deposits incoming data directly into last-level cache (not main memory)
  3. CPU reads packet data from cache — no main memory access needed
Output (Host → NIC)
  1. TCP/IP process creates packet and stores it in allocated L3 cache space (not main memory)
  2. NIC reads data from L3 cache directly — no main memory evictions
DDIO Cache Write Policies
Cache Hit → write-back update (write update). Cache Miss → write-allocate into cache without writing to main memory — data need not be retained after NIC reads it.
Intel Xeon Ring Architecture
Xeon E5-2600/4600: up to 8 cores, shared 20 MB L3 (sliced per core). Ring connects cores + L3 + PCIe + IMC. DMA-with-cache: MCH checks L3 before going to DDR3 on reads — avoids off-chip memory access for cached data.

§7.7 I/O Channels & Processors

An I/O channel is an extension of DMA — it can execute its own I/O instruction programs stored in main memory, completely freeing the CPU from I/O management.

Evolution of the I/O Function
  1. CPU direct control — CPU controls peripheral directly (simple microcontrollers)
  2. I/O module + programmed I/O — CPU somewhat divorced from device details
  3. I/O module + interrupt I/O — CPU free to run other programs while I/O proceeds
  4. DMA — I/O module accesses memory directly; CPU only involved at start/end
  5. I/O Channel — I/O module executes its own instruction set stored in memory; CPU only issues "execute I/O program" command
  6. I/O Processor — full computer with local memory; manages large sets of I/O devices independently
Two Types of I/O Channels
TypeUse CaseBehaviour
Selector Channel High-speed devices (disk, tape) Dedicated to one device at a time; multiplexes between high-speed I/O controllers
Byte Multiplexor Channel Multiple low-speed devices (terminals) Interleaves bytes from multiple devices simultaneously (e.g. A1B1C1A2C2…)
Block Multiplexor Channel Multiple high-speed devices Interleaves blocks (not bytes) from several high-speed devices
Channel Command Word (CCW)
The I/O channel executes a channel program consisting of Channel Command Words stored in main memory. Each CCW specifies: operation, data address, count, and chaining flags. CPU uses a single "start channel" instruction.

§7.9 IBM zEnterprise EC12 I/O Structure

The zEC12 uses a dedicated I/O subsystem that completely offloads I/O processing from the main application processors.

Physical Layout
  • A Frame (right) — Processor cage (4 processor books) + I/O cage (I/O hardware, multiplexors, channels)
  • Z Frame (left) — Batteries, power supplies, support elements, I/O drawers
  • Each processor book = MCM (6 multicore chips + 2 storage control chips) + memory + I/O connections
Channel Subsystem Structure (Figure 7.19)
ElementDetails
Channel Subsystem (CSS)Up to 4 CSSs; each CSS = 1 dedicated core acting as System Assist Processor

HSA (Hardware System Area)32 GB reserved system memory for I/O config; not part of customer memory

I/O Interconnects
  • InfiniBand (HCA fanout) — processor book → I/O cages & drawers; supports ESCON, Ethernet (1/10 Gbps)
  • PCIe fanout — processor book → PCIe switches → I/O device controllers (Ethernet, Fiber Channel)
  • Each book: up to 8 HCA+PCIe fanouts × 32 connections = 256 connections/book

I/O Module Communication Sequence

General sequence for CPU-initiated data transfer from external device to memory (§7.2):

  1. CPU interrogates I/O module — checks status of attached device
  2. I/O module returns device status (READY / NOT-READY / ERROR)
  3. If device READY: CPU issues READ command to I/O module
  4. I/O module obtains unit of data (8 or 16 bits) from external device
  5. Data transferred from I/O module to CPU over data bus
  6. CPU writes data to main memory
Bus Arbitration Note
With a shared system bus, each CPU ↔ I/O module interaction requires one or more bus arbitration cycles.
Processor Communication Functions (I/O Module)
  • Command Decoding — interprets commands from control bus (e.g., READ SECTOR, SEEK, SCAN)
  • Data Exchange — transfers data over the data bus
  • Status Reporting — returns BUSY/READY signals and error conditions
  • Address Recognition — each I/O module has unique address(es) for its device(s)
Error Detection Methods
  • Parity Bit — 8th bit set so total 1s = even (even parity) or odd; detects single-bit errors
  • Mechanical/Electrical fault detection — paper jam, bad disk track, etc.
  • IRA character = 7 bits + 1 parity bit = 1 byte

Exam Tip
Interrupt vs DMA distinction: Both use interrupts at completion. The key difference is that in interrupt I/O, every word passes through CPU; in DMA, the entire block transfers without CPU involvement — CPU only interrupted once per block.

Interrupt Context Save/Restore Detail

When interrupt occurs after instruction at location N (Figure 7.7):

On Interrupt (Hardware Phase)
Stack ← PSW, PC (= N+1)
  • Stack Pointer updated to new top
  • PC loaded with ISR entry address (T)
  • CPU proceeds to fetch instruction at T
ISR Entry (Software Phase)
Stack ← All General-Purpose Registers
  • ISR saves all registers it may modify
  • Stack pointer points to saved register block
ISR Processing
  • Read data from I/O module
  • Check status, handle errors
  • Send additional commands/ACKs if needed
ISR Exit (Software Phase)
Registers ← Stack (pop all saved registers)
Return from Interrupt (Hardware)
PSW, PC ← Stack → resume at instruction N+1
Why Save All State?
Interrupts can occur at any point in any program — unpredictably. The interrupted program must resume exactly as if nothing happened. Even one unsaved register would cause program corruption.

Processor State During DMA
CPU does NOT save any context during cycle stealing — it simply pauses for one bus cycle. Compare to interrupt where full context save occurs. DMA is far less disruptive per-transfer.

I/O Channel: Program & Architecture

An I/O channel executes a channel program in main memory without CPU involvement. CPU only issues a "start I/O channel" instruction.

Channel Command Word (CCW) Fields
FieldPurpose
Operation CodeType of operation: READ, WRITE, SENSE, CONTROL, etc.
Data AddressStarting address in main memory for data buffer
CountNumber of bytes to transfer
FlagsChain data (CD): link to next CCW; Chain command (CC): chain commands; Suppress length indication (SLI)
Channel Program Example
CCW1: READ addr=0x1000 count=512 CC=1
CCW2: READ addr=0x1200 count=512 CC=0

CPU issues one "start channel" command pointing to CCW1. Channel executes both READs autonomously, then interrupts CPU when both are done.

Selector vs Multiplexor (Summary)
ChannelDevicesMethod
SelectorFew high-speed (disk, tape)One device exclusively at a time
Byte MultiplexorMany low-speed (terminals)Byte-interleaved across multiple devices
Block MultiplexorMultiple high-speedBlock-interleaved; handles concurrent block transfers
Channel vs DMA
DMA transfers a block defined at initiation time. I/O channel executes a full program — can handle sequences of transfers, error recovery, conditional branching — a true offload processor.

Key Design Principle
The evolution of I/O techniques consistently moves processing burden away from the CPU — from busy-wait polling → interrupt-per-word → block DMA → cache-direct DCA → fully autonomous I/O channel with its own instruction set. Each step increases CPU availability for application code.
