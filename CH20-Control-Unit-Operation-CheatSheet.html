§20.1 — Micro-Operations

Micro-Operations: Concept & Hierarchy

A micro-operation (μOP) is the most fundamental, atomic operation a processor performs — movement of data into or out of a register, or an ALU computation. Each instruction cycle decomposes into a sequence of μOPs.

Key Principle
Only FETCH and EXECUTE always occur. INDIRECT only if indirect addressing. INTERRUPT only if enabled interrupt pending at end of execute.
Four Categories of μOPs
  • Register → Register: transfer data between internal registers
  • Register → External: transfer from register to system bus (e.g., address bus, data bus)
  • External → Register: read from memory/I/O into register
  • ALU operation: arithmetic or logic using register inputs, result to register

Why Y and Z?
ALU is combinatorial — no storage. Y holds 2nd operand. Z holds result before it can be written to the bus (to prevent feedback loops).

μOP Grouping & Timing Rules

Multiple μOPs may execute in the same clock time unit if they do not conflict. This parallelism reduces instruction cycle time.

Two Grouping Rules
Rule 1 — Sequence
Proper order must be maintained: MAR ← (PC) must precede MBR ← Memory because the memory read uses the address already in MAR.
Rule 2 — No Conflicts
Cannot read AND write the same register in the same time unit. E.g., MBR ← Memory and IR ← MBR cannot be simultaneous — result would be unpredictable.
Parallel Example (t2 of Fetch)
t2: MBR ← Memory ← read instruction
t2: PC ← (PC) + I ← increment PC
These two DO NOT conflict — safe to do together
Notation Key
  • t1, t2, t3 — successive clock time units
  • X ← (Y) — copy contents of Y into X
  • MBR ← Memory — read from memory at address in MAR
  • Memory ← (MBR) — write MBR contents to memory at MAR
  • I — instruction length in address units

Fetch Cycle μOPs

Occurs at the beginning of every instruction cycle. Retrieves the next instruction from memory into the IR.

Standard 3-Step Fetch
t1:MAR ← (PC)
t2:MBR ← Memory|PC ← (PC) + I
t3:IR ← (MBR)
Step-by-Step Explanation
  1. t1: Move PC → MAR. MAR is the only register connected to address bus, so the desired instruction address must pass through it.
  2. t2: Issue READ on control bus; memory word at MAR address flows to MBR via data bus. Simultaneously, PC is incremented by I (instruction length) to point to next instruction. These two are independent — safe to parallel.
  3. t3: Move MBR → IR. This frees MBR for use in a possible indirect cycle.
Alternative Grouping (t2+t3 merge)
t1: MAR ← (PC)
t2: MBR ← Memory
t3: PC ← (PC)+I  |  IR ← (MBR)
Both forms are valid — rules 1 and 2 are satisfied either way.

Indirect Cycle μOPs

Occurs only when the current instruction uses indirect addressing. Resolves the indirect address before execute.

Indirect Address Resolution
t1:MAR ← (IR(Address))
t2:MBR ← Memory
t3:IR(Address) ← (MBR(Address))
What Happens
  1. t1: The address field of the instruction (which is an indirect address) is copied to MAR.
  2. t2: The word at that indirect address is read from memory into MBR. This word contains the actual effective address of the operand.
  3. t3: The address portion of MBR is written back into IR's address field — replacing the indirect address with the direct effective address.
After Indirect Cycle
The IR now looks as if indirect addressing had never been used. The execute cycle proceeds with a direct address, simplifying its logic.

Interrupt Cycle μOPs

Occurs at end of execute if an enabled interrupt is pending. Saves the PC (return address) and transfers control to the interrupt service routine.

Simple Interrupt Save & Jump
t1:MBR ← (PC)
t2:MAR ← Save_Address|PC ← Routine_Address
t3:Memory ← (MBR)
Step-by-Step Explanation
  1. t1: Current PC (return address from interrupt) is moved to MBR for safe-keeping before PC is modified.
  2. t2: MAR is loaded with the memory location where the return address will be saved; simultaneously, PC is loaded with the starting address of the interrupt service routine (ISR). Two independent transfers — safe to parallel.
  3. t3: MBR (containing the old PC) is written to memory at the Save_Address.
Real Processors
Real interrupt cycles are more complex: may need extra μOPs to determine Save_Address and Routine_Address from interrupt type/priority vectors before steps can execute.

Instruction Cycle Code (ICC)

A 2-bit register (ICC) tracks which phase of the instruction cycle the processor is currently in. The control unit uses ICC to decide which μOP sequence to execute next.

ICCPhaseNext Cycle Logic
00FetchIf indirect bit set → 01 (Indirect)
Else → 10 (Execute)
01IndirectAlways → 10 (Execute)
10ExecuteIf interrupt pending → 11
Else → 00 (Fetch)
11InterruptAlways → 00 (Fetch)
Complete Instruction Cycle Flowchart Logic
  • Start at ICC=00 (Fetch): execute fetch μOPs, then check IR for indirect bit
  • If indirect → ICC=01: execute indirect μOPs, unconditionally advance to Execute
  • ICC=10 (Execute): decode opcode, execute appropriate μOP sequence
  • After execute: test for enabled interrupt; if yes → ICC=11, else → ICC=00
  • ICC=11 (Interrupt): save PC, load ISR address, unconditionally → ICC=00
Purpose of ICC
The ICC tells the control unit "where it is" in the instruction cycle so it knows which set of control signals to generate at each clock step.

§20.2 — Control of the Processor

CU Functional Requirements & Two Tasks

To design the control unit, we identify what it must do (functional requirements) based on the μOPs analysis.

Three-Step Design Process
  1. Define basic elements of the processor (ALU, registers, data paths, external bus, CU)
  2. Describe all micro-operations the processor performs (as done in §20.1)
  3. Determine what signals the CU must generate to cause each μOP to occur
Two Core Tasks of the Control Unit
SEQUENCING: Step the processor through μOPs in the
             correct order based on program being executed
EXECUTION: Generate the specific control signals that
           cause each μOP to be performed
Processor Functional Elements
  • ALU: performs all arithmetic and logic operations
  • Registers: store data internal to processor (PC, IR, MAR, MBR, AC, general purpose)
  • Internal data paths: buses/connections moving data between registers and ALU
  • External data paths: system bus connections to memory and I/O
  • Control unit: orchestrates all of the above

Minimal Nature of Control Unit
The CU runs the entire computer yet never sees the actual data being processed. It controls everything through a small number of control signals to specific gate points.

Intel 8085 — Control Unit Example (Fig 20.7)

The 8085 illustrates real-world control unit structure. Its CU has two sub-modules: (1) Instruction Decoder & Machine Cycle Encoding, (2) Timing and Control.

Key 8085 Components
  • Timing & Control module: core of CU — receives clock, IR, external control signals; outputs internal + external control signals
  • Instruction decoder & machine cycle encoding: translates opcode bits into μOP-selecting signals
  • Incrementer/Decrementer address latch: adds/subtracts 1 from SP or PC without tying up the ALU
  • Interrupt control: handles multiple interrupt priority levels
  • Serial I/O control: SID/SOD — 1-bit serial communication
Machine Cycle vs. State Timing
  • Each instruction cycle = 1–5 machine cycles (fixed per instruction)
  • Each machine cycle = 3–5 states (T1, T2, T3…)
  • Each state = 1 clock cycle
  • Machine cycles ≡ bus accesses: # determined by how many times processor communicates with external devices

Fetch Cycle — Clock by Clock
  • CU step 1: Knowing ICC=00, activates C2 → contents of PC flow through open gate to MAR
  • CU step 2: Activates C5 (MAR→addr bus) + CR (memory read) + C5 gating data bus→MBR + PC increment logic
  • CU step 3: Activates C4 → MBR contents flow to IR
  • CU decision: Examines IR indirect bit; sets ICC=01 or ICC=10 accordingly
§20.3 — Hardwired Implementation

Hardwired Control Unit — Structure

A hardwired (combinational/sequential logic) control unit transforms input signals into output control signals using fixed digital logic circuits. No stored program — logic is "wired in".

Two Implementation Categories
Hardwired → state machine (sequential logic circuits)
Microprogrammed → stored program in control memory (Ch.21)
Hardwired CU = Finite State Machine
  • States correspond to individual time steps within instruction cycles
  • State transitions driven by clock + ICC + decoded IR + flags
  • Each state produces specific output (control signals)
  • Very fast — direct gate delay, no memory access penalty
Three Building Blocks
BlockRole
Decodern-bit opcode → one of 2ⁿ unique outputs; each output selects a specific μOP sequence
EncoderCombines decoder outputs + state + flags → produces correct set of control signals
Sequential LogicClock-driven flip-flops maintain current state (ICC); implements state transitions

Opcode Decoder (Table 20.3)

A decoder converts the n-bit opcode into 2ⁿ individual output lines — only one is active at a time. This gives the CU a unique signal for each instruction type.

Real Decoder Complexity
Real ISA decoders must handle variable-length opcodes, prefix bytes, and addressing mode bits — far more complex than the simple n-input example.
Clock Timing Constraint

The clock period must be long enough for signals to propagate through the longest gate chain in the processor. This sets the maximum clock frequency of the processor.

Key Terms & Definitions

  • Micro-operation (μOP): atomic processor operation; movement of data into/out of a register or ALU computation
  • Microprogram: sequence of microinstructions stored in control memory (Ch.21)
  • Instruction Cycle Code (ICC): 2-bit register tracking phase (00=fetch, 01=indirect, 10=execute, 11=interrupt)
  • MAR: Memory Address Register — holds address for any memory read or write
  • MBR/MDR: Memory Buffer/Data Register — buffers data between CPU and memory
  • Control signals: binary signals emitted by CU that open gates, activate ALU, or drive system bus
  • Hardwired control: CU implemented as finite state machine with combinational + sequential logic
  • State machine: sequential circuit with states + transitions; drives correct outputs per state
  • Decoder: n inputs → 2ⁿ outputs; one output active per input pattern; maps opcode to unique signal
  • Instruction decoding: process by which CU examines opcode in IR to determine which execute-cycle μOP sequence to perform
  • Clock cycle: one time unit; one set of simultaneous μOPs per cycle; period must exceed max gate delay
