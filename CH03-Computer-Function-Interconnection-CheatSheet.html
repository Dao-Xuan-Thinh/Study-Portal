§3.1 — Computer Components

Von Neumann Architecture

Contemporary computer designs are based on concepts developed by John von Neumann at the Institute for Advanced Studies, Princeton.

Three Key Concepts
  • Stored Program: Data and instructions are stored in a single read–write memory
  • Addressable Memory: Contents of memory are addressable by location, without regard to the type of data contained there
  • Sequential Execution: Execution occurs sequentially from one instruction to the next (unless explicitly modified)
Two Major CPU Components
  • Instruction Interpreter: Accepts instruction codes and generates control signals
  • General-Purpose ALU: Performs arithmetic and logic operations based on control signals
Key Insight
Instead of rewiring hardware for each program (hardwired), the programmer supplies a new sequence of instruction codes (software). Same hardware, different control signals.

Top-Level Components & Registers

System Components
  • CPU: Instruction interpreter + ALU; reads instructions & data, writes results, generates control signals
  • Main Memory: N words at addresses 0…N-1; stores both instructions and data (von Neumann)
  • I/O Module: Transfers data between external devices and CPU/memory; contains internal buffers
  • System Bus: Interconnects CPU, memory, and I/O modules

§3.2 — Computer Function: Instruction Cycle

Basic Instruction Cycle

The processing required for a single instruction is called an instruction cycle.

START │ ▼ ┌──────────────────┐ │ FETCH CYCLE │ ← PC → MAR → memory → MBR → IR │ Fetch next │ PC is incremented │ instruction │ └────────┬─────────┘ │ ▼ ┌──────────────────┐ │ EXECUTE CYCLE │ ← Decode IR, perform action │ Execute │ (may access memory/I/O) │ instruction │ └────────┬─────────┘ │ ▼ HALT? ──Yes──► HALT │No └──────────► (loop back to FETCH)
Four Categories of Instruction Actions
  • Processor-Memory: Data transfer between processor and memory
  • Processor-I/O: Data transfer to/from peripheral device via I/O module
  • Data Processing: Arithmetic or logic operation on data
  • Control: Alter sequence of execution (jumps/branches)

§3.2 — Interrupts

Classes of Interrupts

Interrupts are provided primarily as a way to improve processing efficiency — the CPU doesn't waste cycles polling slow devices.

ClassSourceExamples
ProgramInstruction execution conditionArithmetic overflow, division by zero, illegal opcode, memory violation
TimerInternal processor timerAllows OS to perform periodic functions (preemptive multitasking)
I/OI/O controllerNormal I/O completion, service request, error conditions
Hardware FailureHardware faultPower failure, memory parity error
Without Interrupts
CPU must sit idle polling the I/O device. A slow printer could waste thousands of instruction cycles — very inefficient.

Interrupt Handling Procedure

Instruction Cycle with Interrupt Cycle Added
START │ ▼ ┌──────────────┐ │ FETCH CYCLE │ └──────┬───────┘ │ ▼ ┌──────────────┐ │ EXECUTE CYCLE│ └──────┬───────┘ │ ▼ ┌──────────────────────────────┐ │ INTERRUPT CYCLE │ │ Any interrupt pending? │ │ No ──────────────────────► │ (back to FETCH) │ Yes ──► process interrupt │ └──────────────────────────────┘
Interrupt Processing Steps
  1. Suspend current program execution; save context (PC + relevant data) onto system stack
  2. Set PC to starting address of the interrupt handler routine (ISR)
  3. Execute ISR: Determine nature of interrupt; perform required actions (e.g., write more data to I/O module)
  4. Restore context: Pop saved state from stack; resume user program at point of interruption
Key Point
The user program does NOT contain special interrupt code — the processor and OS are entirely responsible for suspending and resuming it transparently.

Multiple Interrupts & Timing

Two Approaches to Multiple Interrupts
Approach 1 — Sequential (Disable Interrupts)
While processing an interrupt, all interrupts are disabled. Pending interrupts are checked after ISR completes and interrupts are re-enabled. Simple but ignores priority.
Approach 2 — Nested (Priority-Based)
Higher-priority interrupts can interrupt a lower-priority ISR. Lower-priority interrupts are held pending. Enables time-critical processing.
Priority Example (t-axis, 3 devices: printer=2, disk=4, comm=5)
t=0 User program begins t=10 Printer interrupt (priority 2) → enter Printer ISR t=15 Comm interrupt (priority 5 > 2) → interrupt Printer ISR Push Printer ISR state → enter Comm ISR t=20 Disk interrupt (priority 4) → HELD (4 < 5, comm running) t=25 Comm ISR completes → restore Printer ISR state BUT disk interrupt (4 > 2) honored immediately → enter Disk ISR t=35 Disk ISR completes → resume Printer ISR t=40 Printer ISR completes → return to User program

§3.3 — Interconnection Structures

Required Interconnection Transfers

The interconnection structure is the collection of paths connecting the modules. It must support all necessary data exchanges:

Transfer TypeDirectionDescription
Memory ReadMemory → ProcessorProcessor reads instruction or data from memory
Memory WriteProcessor → MemoryProcessor writes data unit to memory
I/O ReadI/O → ProcessorProcessor reads data from I/O device via I/O module
I/O WriteProcessor → I/OProcessor sends data to I/O device
DMA (I/O ↔ Memory)I/O ↔ MemoryI/O module exchanges data with memory directly, bypassing processor
Module Interface Requirements
  • Memory: Read/Write operations; address specifies location; data bus carries word
  • I/O Module: Read/Write operations; port address; external data paths; can send interrupt signals
  • Processor: Reads instructions & data; writes data; uses control signals; receives interrupts

Interconnect Topology Comparison

FeatureShared BusPoint-to-Point

Why the Shift?
At higher frequencies, synchronization and arbitration in wide synchronous buses become increasingly difficult. Multi-core chips magnified these problems.
§3.4 — Bus Interconnection

Bus Structure & Line Types

A bus is a shared communication pathway connecting two or more devices. Only one device may transmit at a time (shared medium).

Three Functional Groups of Bus Lines
┌───────────────────────────────────────────┐ │ SYSTEM BUS │ ├───────────────────────────────────────────┤ │ DATA LINES (32/64/128+ bits wide) │ │ ════════════════════════════════════════ │ │ → carry actual data between modules │ │ → width = key performance factor │ ├───────────────────────────────────────────┤ │ ADDRESS LINES (width → memory capacity) │ │ ──────────────────────────────────────── │ │ → specify source/destination of data │ │ → high-order bits: select module │ │ → low-order bits: select location/port │ ├───────────────────────────────────────────┤ │ CONTROL LINES (command + timing signals) │ │ ············································│ │ → control access to data & address lines │ └───────────────────────────────────────────┘
Bus bandwidth = bus_width(bits) × bus_frequency(Hz) / bits_per_transfer
Data Bus Width Impact
If data bus is 32 bits wide but instructions are 64 bits long → processor must access memory TWICE per instruction cycle, halving throughput.

Control Line Signals

Bus Operation Protocol
  1. Obtain bus: Module requests and waits for bus grant
  2. Transfer data: Put address on address lines, data on data lines, assert control signal

Bus Arbitration & Timing

Arbitration Methods
Centralized Arbitration
Single hardware arbiter receives all bus requests; grants bus access. Simpler design but arbiter is a single point of failure.
Distributed Arbitration
Each module participates in selection of next bus master. E.g., daisy-chain: BPRN/BPRO signals propagate priority from highest to lowest device. No single point of failure.
Timing Methods
SYNCHRONOUS BUS (clock-driven): ┌───┐ ┌───┐ ┌───┐ ┌───┐ │ T1│ │ T2│ │ T3│ │ T4│ ┘ └───┘ └───┘ └───┘ └ Address Data ACK Next valid valid rcv'd cycle ASYNCHRONOUS BUS (handshake-driven): Master: ──[ReadReq]──────────────[ReadReq↓]── Slave: ──[DataReady]──[ACK]── (Each event triggers next; no shared clock)
MethodProsCons
SynchronousSimple, fastMust accommodate slowest device; clock skew at high freq.
AsynchronousAccommodates mixed speedsMore complex handshake logic
§3.5 — Point-to-Point Interconnect: Intel QuickPath Interconnect (QPI)

QPI Overview & Key Characteristics

Intel's QuickPath Interconnect (QPI), introduced 2008, is a point-to-point interconnect replacing Front Side Bus in multi-core platforms.

Key Characteristics
  • Multiple direct connections: Pairwise direct links between components — no arbitration needed
  • Layered protocol architecture: Like TCP/IP — Protocol / Routing / Link / Physical layers
  • Packetized data transfer: Data sent as packets with headers and error control codes, not raw bit streams
  • Bidirectional: Simultaneous full-duplex transfer in both directions
Typical Multicore Configuration
┌──────────┐ QPI ┌──────────┐ │ Core A │◄────►│ Core B │ │ +DRAM │ │ +DRAM │ └────┬─────┘ └─────┬────┘ │ QPI QPI │ ▼ ▼ ┌──────────┐ QPI ┌──────────┐ │ Core C │◄────►│ Core D │ │ +DRAM │ │ +DRAM │ └──────────┘ └──────────┘ │ QPI │ QPI ▼ ▼ ┌───────┐ ┌───────┐ │ IOH │ │ IOH │ (I/O Hub) └───┬───┘ └───┬───┘ │ PCIe │ PCIe I/O devices I/O devices
QPI throughput = 6.4 GT/s × 20 bits/transfer = 16 GB/s per direction Total bidirectional capacity = 32 GB/s

┌──────────────────────────────────────────┐ │ QPI Port Physical Structure │ ├──────────────────────────────────────────┤ │ 20 data lanes (TX) + 1 clock lane (TX) │ │ 20 data lanes (RX) + 1 clock lane (RX) │ │ Total: 84 individual links per port │ ├──────────────────────────────────────────┤ │ Each "lane" = PAIR of wires (differential)│ │ Binary value = voltage difference between│ │ the two conductors (LVDS technique) │ ├──────────────────────────────────────────┤ │ 4 quadrants × 5 lanes = 20 data lanes │ │ Can operate at half/quarter width for │ │ power saving or fault tolerance │ └──────────────────────────────────────────┘
Multilane Distribution (80-bit flit → 4× 20-bit phits)
Flit bit stream: b0 b1 b2 b3 ... b79 Lane 0: b0, b20, b40, b60 Lane 1: b1, b21, b41, b61 ... Lane 19: b19, b39, b59, b79 (Round-robin distribution across 20 lanes)
Link Layer — Error & Flow Control
  • CRC Error Detection: Each 80-bit flit has 8-bit CRC over 72-bit payload; mismatch → NAK → retransmit damaged flit + all subsequent
  • Credit-Based Flow Control: Sender given N credits; each flit sent decrements credit; each buffer freed at receiver returns a credit; prevents overflow
§3.6 — PCI Express (PCIe)

PCI Express (PCIe) replaces bus-based PCI for I/O device interconnection. Point-to-point, full-duplex, high-bandwidth, processor-independent.

PCIe Physical Layer: 128b/130b Encoding & Scrambling

Why 128b/130b Encoding? (PCIe 3.0+)

Unlike QPI, PCIe does not use its clock lane for bit synchronization — the receiver must self-synchronize by detecting transitions in the data stream.

Problem
Long strings of identical bits (all 0s or all 1s) → no transitions → receiver clock drifts → loss of synchronization.
Solution: Two-Step Process
Input data │ ▼ ┌──────────────┐ Spreads transitions uniformly; │ SCRAMBLER │ ←─ improves spectral properties; └──────┬───────┘ doesn't add bits │ ▼ ┌──────────────────┐ Buffers 128 bits; adds 2-bit │ 128b/130b ENCODER│ ←─ sync header (10=data, 01=ordered set) └──────┬───────────┘ Overhead = only 1.5% │ ▼ ┌─────────────────────┐ │ PARALLEL→SERIAL │ → Differential signaling (D+/D-) └─────────────────────┘ At Receiver: Serial→Parallel → 128b/130b Decode → Descramble
Multilane Distribution (PCIe ×4 Example)
Byte stream: B0 B1 B2 B3 B4 B5 B6 B7 ... Lane 0 (128b/130b): B0, B4, B8, ... Lane 1 (128b/130b): B1, B5, B9, ... Lane 2 (128b/130b): B2, B6, B10, ... Lane 3 (128b/130b): B3, B7, B11, ... (1 byte at a time, round-robin)
Result
Using these techniques, PCIe 3.0 achieves 16 GB/s per ×16 link (each direction). Each transmission begins/ends with 8-bit framing for receiver synchronization.

PCIe Split Transaction & TL Operations

Transaction Types by Response
  • Split transactions: Memory/I/O/Config reads — completion packet returns later with data
  • Posted transactions: Memory writes, Message requests — no response expected
  • Locked operations: Atomic read-modify-write sequences; root complex locks PCIe link during transaction set
TLP Packet Attributes
  • no-snoop — bypass cache coherency check for non-cacheable data
  • relaxed-ordering — allow reordering for performance
  • priority — route time-sensitive data (streaming video, audio) first
