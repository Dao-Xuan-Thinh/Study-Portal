§8.1 OS Objectives & Services

Two Core Objectives
  • Convenience — Makes the computer easier to use for programmers & users
  • Efficiency — Allows system resources to be used in an efficient manner
Key OS Services
  • Program Creation — Editors, debuggers, utilities accessible through the OS
  • Program Execution — Load instructions & data into memory, initialize I/O & files
  • Access to I/O Devices — Abstracts device-specific control signals to simple reads/writes
  • Controlled File Access — Understands device format; provides protection in multi-user systems
  • System Access — Controls access to shared/public resources; resolves resource contention
  • Error Detection & Response — Handles hardware faults, overflow, forbidden memory access
  • Accounting — Collects usage stats, monitors performance, supports billing in multi-user systems
OS as Resource Manager

The OS controls the movement, storage, and processing of data. Unlike typical control mechanisms, the OS is itself a program executed by the processor — it relinquishes control so other programs can run, then regains control via interrupts or service calls.

System Interface Layers

Three Key Interfaces (Figure 8.1)
InterfaceDescription

Instruction Set Architecture		Boundary between HW & SW. User ISA = subset for apps. System ISA = privileged instructions only for OS (manages system resources)
ABI

API

Hardware Features Required by OS
  • Memory Protection — Prevent user programs from overwriting monitor/OS memory
  • Timer — Prevents any single job monopolizing CPU; triggers interrupt on expiry
  • Privileged Instructions — Can only execute in kernel/monitor mode (e.g., I/O instructions)
  • Interrupts — Give OS flexibility to relinquish and regain CPU control
Key Insight
The OS relies on hardware features (interrupts, timer, privilege modes) to maintain control over user programs running on the processor.

  • Waiting Time — Total time spent in ready queue
  • Response Time — Time from request submission to first response

Process Control Block (PCB)

Definition
Each process is represented in the OS by a Process Control Block — a data structure holding all information needed to manage and resume the process.
PCB Fields (Figure 8.8)
FieldContent
IdentifierUnique process ID (PID)

Context Switch Mechanism

When a process is preempted or blocked, the processor saves context data + program counter into the PCB. When the OS resumes the process, it restores this information so execution continues exactly where it left off.

Queue Management
OS maintains: Long-term queue (jobs waiting admission), Short-term queue (ready processes), I/O queues (one per device for waiting processes).

Preemptive vs Non-Preemptive
Non-preemptive: Process runs until it voluntarily releases CPU (I/O, exit). Preemptive: OS can forcibly remove a running process (timeout, higher-priority arrival). Preemptive needed for time-sharing and real-time systems.

§8.3 Swapping & Partitioning

Why Swapping?

Even with multiprogramming, all processes in memory may be waiting on I/O. Swapping moves blocked processes to an intermediate queue on disk, freeing memory for other processes. Disk I/O is the fastest I/O, so swapping improves performance.

Fixed-Size Partitioning
  • Memory divided into fixed-size chunks (equal or unequal size)
  • Process placed in smallest available partition that fits
  • Problem: Internal fragmentation — unused space within a partition
Variable-Size Partitioning
  • Each process allocated exactly the memory it needs
  • Problem: External fragmentation — many small holes accumulate over time
  • Solution: Compaction — OS periodically shifts processes to consolidate free space (time-consuming)
Logical vs Physical Addresses
Logical Address → [base address + logical addr] → Physical Address Base address = current starting location of process in memory CPU hardware performs translation automatically
Key Point
Processes are not always loaded at the same physical address each time. Logical addressing allows relocation without changing the program code.

Paging — Mechanism & Address Translation

Core Concept

Memory divided into equal fixed-size frames. Each process divided into same-size pages. Pages can be loaded into any available frames — no contiguity required. Wastes at most a fraction of the last page (minimal internal fragmentation).

Address Translation
Logical Address = [ Page Number | Offset ] Physical Address = [ Frame Number | Offset ] Page Table: maps Page# → Frame# for each process CPU: uses page number to index page table → get frame number → combine with offset
Page Table Entry (PTE) Fields
Bit/FieldMeaning
Frame NumberPhysical frame in main memory holding this page
Valid/Present (P)1 = page is in main memory; 0 = page fault on access
Dirty (D)1 = page has been written (modified); must be written back on eviction
Accessed (A)Set to 1 on any read or write; used by page replacement algorithms
ProtectionRead-only, Read/Write, Execute permissions
Inverted Page Table (Alternative)

One entry per physical frame (not per virtual page). Uses hash function on page number → index into inverted table. Fixed memory for tables regardless of number of processes. Used in PowerPC, UltraSPARC, IA-64.

Hash(Virtual Page#) → entry in Inverted Page Table → (PID, Page#, Frame#)

Virtual Memory & Demand Paging

Demand Paging

Pages are loaded only when needed (on demand). Only a few pages of a process need be in memory at any time. Exploits the principle of locality: execution is confined to a small working set of pages over any short interval.

Key Concepts
TermDefinition
Page FaultAccess to a page not currently in memory; triggers OS to load it from disk
Page ReplacementWhen memory is full, OS must evict a page to load the needed one
ThrashingCPU spends more time swapping pages than executing; process working set > available frames
Real MemoryActual physical main memory where process executes
Virtual MemoryProgrammer's view — entire disk-sized address space; much larger than RAM
Working SetSet of pages actively used by a process over a recent time window; key concept for avoiding thrashing
Key Benefit

A process can be larger than physical memory. Programmer deals with a huge virtual address space; OS + HW handle the mapping to actual RAM and disk transparently.

Thrashing Warning
If the OS evicts pages just before they are reused, the CPU spends most of its time on page swaps. Prevention: working set model — keep all pages of the working set in memory.

Translation Lookaside Buffer (TLB)

Problem & Solution

Without TLB: every virtual memory access requires two physical memory accesses — one for the page table entry, one for the data. This doubles memory access time.

The TLB is a small, fast hardware cache of recently-used page table entries. Exploits temporal locality: most accesses hit recently-used pages.

TLB Access Flow (Figure 8.18)
  1. CPU generates virtual address (page# + offset)
  2. Check TLB for page table entry → TLB Hit: get frame# directly (fast)
  3. TLB Miss: access page table in main memory; update TLB with new entry
  4. Check if page is in main memory → Page Hit: combine frame# + offset → physical address
  5. Page Fault: OS loads page from disk; perform page replacement if memory full; update page table & TLB; retry instruction
TLB + Cache Interaction (Figure 8.19)
Virtual Addr → [TLB hit?] → Real Addr [tag|remainder] → [Cache hit?] → Data ↓ miss ↓ miss Page Table Main Memory
Performance
Effective Access Time = h × t_TLB + (1-h) × (t_TLB + t_mem) h = TLB hit ratio (typically 0.90–0.99) t_TLB = TLB access time t_mem = main memory access time
x86 TLB
Intel x86 TLB holds 32 page table entries. The buffer is cleared each time the page directory register is changed (process context switch).

Page Replacement Algorithms

Bélády's Anomaly (FIFO only)
Adding more page frames can sometimes increase the number of page faults. This counter-intuitive behavior only affects FIFO. LRU and OPT are stack algorithms and are immune to Bélády's Anomaly.

Segmentation

Core Concept

Segmentation divides memory into variable-length, programmer-visible segments. Unlike paging (invisible to programmer, fixed-size), segments correspond to logical program divisions: code, data, stack, heap, shared libraries.

Logical Address Format
Logical Address = [ Segment Number | Offset ] Segment Table: maps Segment# → (Base Address, Limit, Access Rights) Physical Address = Segment Base + Offset (if Offset ≤ Limit)
Advantages of Segmentation
  • Dynamic growth — Data structures assigned their own segment; OS expands/shrinks as needed
  • Independent recompilation — Programs altered independently without relinking entire set
  • Sharing — A utility or data table segment can be addressed by multiple processes
  • Protection — Access privileges assigned per segment (read-only, execute-only, read/write)
Paging vs Segmentation
FeaturePagingSegmentation
Programmer visible?NoYes
Unit sizeFixed (e.g., 4 KB)Variable
Internal fragmentationSmall (≤ page size)None
External fragmentationNoneYes
Sharing/protectionDifficultNatural fit
Virtual memory supportExcellentModerate
Combined Approach
Many systems (e.g., x86, UNIX System V) use paged segmentation: segmentation provides logical organization and protection; paging provides efficient physical memory allocation within segments.

§8.5 ARM — Memory System Organization

ARM Memory Management Hardware (Figure 8.22)

ARM provides a versatile MMU (Memory Management Unit) tailored for embedded system needs. Key components:

  • TLB — Cache of recent page table entries; supplies physical address directly on hit
  • Virtual Memory Translation Hardware — Performs 1 or 2-level table walk on TLB miss
  • Access Control Hardware — Checks access permissions; issues Abort signal if denied
  • Cache & Write Buffer — Data exchanged between processor and main memory via cache
Page/Section Size Options
TypeSizeLevels NeededNotes
Supersection16 MBL1 onlyOptional; 16 identical L1 entries required (4-bit overlap)
Section1 MBL1 onlySingle L1 lookup; efficient for large memory regions
Large Page64 KBL1 + L2L2 entries replicated 16 times (4-bit overlap); subpages of 16 KB
Small Page4 KBL1 + L2Standard page; subpages of 1 KB; most granular control
Design Goal
Sections/supersections use a single TLB entry for large memory regions — efficient when mapping large OS regions. Small pages used for fine-grained user process memory.

§8.5 ARM — Access Control & Domains

AP (Access Permission) Bits

Every page table entry and TLB entry contains AP bits that control access to that memory region. A Permission Fault is raised if access is attempted without the required permissions.

Access TypeDescription
No AccessAccess denied (fault generated)
Read-OnlyOnly read operations permitted
Read-WriteFull read and write access
Privileged OnlyOS/kernel access only; denied to user applications

ARM supports 16 domains. A domain is a collection of sections and/or pages sharing access permissions. The Domain Access Control Register (2 bits per domain) controls access to each domain rapidly.

Exam Tips & Common Exam Questions

Q: Why does paging eliminate external fragmentation?

Frames are all equal-size; any free frame can hold any page. No holes — pages don't need contiguous memory.

Q: What hardware features does multiprogramming require?

Interrupt-driven I/O (or DMA), memory protection, timer interrupt, privileged instructions, context save/restore.

Q: Why is TLB necessary for virtual memory performance?

Without TLB, every memory access requires 2 physical accesses (page table + data), doubling access time. TLB hit rate ~90-99% reduces this overhead dramatically.

Q: Difference between paging and segmentation?

Paging: fixed-size, invisible to programmer, no external fragmentation. Segmentation: variable-size, visible/logical, supports sharing/protection naturally.

Q: What causes thrashing and how is it prevented?

Thrashing: working set larger than available frames → constant page faults. Prevention: working set model (keep all pages of working set in memory), reduce multiprogramming degree.

Q: What is Bélády's Anomaly?

With FIFO page replacement, increasing the number of page frames can increase page faults. Does not affect LRU or OPT (stack algorithms).
