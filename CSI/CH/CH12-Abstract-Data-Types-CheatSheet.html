ADT Concept

ADT = data + operations defined mathematically, independent of any implementation

An Abstract Data Type (ADT) separates what a data structure does from how it does it.

Abstraction vs Implementation
Abstraction (ADT)Implementation
Defines behavior via operationsSpecifies structure in memory
Language- and machine-independentTied to a language / platform
Hides internal detailsExposes memory layout
Stated as a contract / interfaceArray, linked list, tree, etc.
Interface vs Implementation
  • Interface — the set of allowed operations and their signatures
  • Implementation — the concrete data structure + algorithms that fulfill the interface
  • Users depend only on the interface; internals may change freely
Encapsulation

Encapsulation enforces the ADT contract: data is accessed exclusively through the defined operations. No direct manipulation of internal representation is permitted.

ADT as a Contract
An ADT is a contract between the implementor and the user: the user gets predictable behavior; the implementor retains freedom to optimize internals.

Stack ADT

Stack: ordered list; insertions and deletions restricted to one end (the top) — LIFO

A Stack enforces LIFO (Last-In, First-Out) access. Think of a pile of plates.

Formal Operations
OperationPre-conditionPost-condition
push(x)stack is not fullx on top; size+1
pop()stack is not emptytop element removed; returned
peek()stack is not emptytop element returned; unchanged
isEmpty()—returns true if size = 0
isFull()—returns true if size = capacity
Pre / Post-Condition Pattern
pre: stack not empty post: top ← stack[top-1]; top--; return value
Applications
  • Function call / return (call stack)
  • Expression evaluation & conversion (infix → postfix)
  • Undo / redo mechanisms in editors
  • Backtracking algorithms (maze, DFS)
  • Balanced-parentheses / syntax checking
Array-Based Stack — Key Formulas
push(x): stack[++top] = x // pre: top < capacity-1 pop(): return stack[top--] // pre: top >= 0 peek(): return stack[top] // pre: top >= 0 isEmpty: top == -1 | isFull: top == capacity-1
Stack Overflow
Pushing onto a full array-based stack → stack overflow. Popping from empty → underflow. Always guard with isFull() / isEmpty().

Queue ADT

Queue: ordered list; insertions at rear, deletions at front — FIFO

A Queue enforces FIFO (First-In, First-Out) access. Think of a waiting line.

Formal Operations
OperationPre-conditionPost-condition
enqueue(x)queue not fullx added at rear; size+1
dequeue()queue not emptyfront element removed; returned
front()queue not emptyfront element returned; unchanged
isEmpty()—returns true if size = 0
isFull()—returns true if size = capacity
Circular Queue

Array-based queues waste space as front advances. A circular queue treats the array as a ring:

rear = (rear + 1) % capacity front = (front + 1) % capacity full when (rear+1)%capacity == front
Priority Queue ADT
  • Each element has an associated priority
  • dequeue() removes the highest-priority element, not simply the oldest
  • Typical implementation: binary heap
  • Applications: OS scheduling, Dijkstra's algorithm, Huffman coding
Queue vs Stack
Both restrict access to a single end; the queue differs in having a separate insertion end (rear) and removal end (front), producing FIFO instead of LIFO order.
Deque (Double-Ended Queue)
  • Generalizes both stack and queue: insert/delete allowed at both ends
  • Operations: addFront, addRear, removeFront, removeRear
  • Can emulate a stack (use one end only) or a queue (add rear, remove front)

Position vs Index
List ADT positions are often 1-based in textbook definitions. Always check whether an implementation uses 0-based or 1-based indexing to avoid off-by-one errors.

Tree ADT

Tree ADT: hierarchical structure; one root, each node has ≥ 0 children, no cycles
Binary Tree ADT
  • Each node has at most two children: left and right
  • Operations: insert, delete, search, traverse (pre/in/post-order)
  • Full vs Complete binary tree distinctions matter for implementation
BST (Binary Search Tree) ADT

Invariant: left subtree keys < node key < right subtree keys.

search: O(log n) avg, O(n) worst (degenerate) insert: O(log n) avg | delete: O(log n) avg

Heap Sort
Phase 1: heapify the array — O(n).  Phase 2: repeatedly extract-max into sorted position — O(n log n). Total: O(n log n), in-place, not stable.

ADT Implementation Strategies

Choose implementation by access patterns, memory constraints, and language support
Array-Based vs Pointer-Based
FactorArray-BasedPointer-Based
Cache localityExcellentPoor (scattered)
Random accessO(1)O(n)
Mid-list insert/deleteO(n)O(1) with ptr
Max size known?Required (static)Not required
Memory per elementData onlyData + pointer(s)
Time / Space Trade-offs
  • Hash table: O(1) avg search but O(n) worst; uses extra load-factor space
  • Balanced BST (AVL/Red-Black): O(log n) guaranteed; rotation overhead
  • Sorted array: O(log n) binary search; O(n) insert/delete
  • Skip list: O(log n) expected; probabilistic; pointer overhead
Generic / Template Types

A well-designed ADT is generic: parameterized over the element type so that one Stack definition works for int, float, or any user type.

Stack<T> // C++ template Stack<T> // Java generic // Single implementation; compiler generates type-safe code
Design Principle
Define the ADT interface first. Select the implementation only after analyzing the expected operations and their required complexities — not the other way around.
Information Hiding
Changing an array-based stack to a linked-list stack should require zero changes to any code that uses the Stack ADT interface. This is the payoff of encapsulation.

Key Principles to Remember
  1. Separation of concerns: define the ADT interface before choosing an implementation.
  2. Encapsulation: users interact through operations only — never touch internal representation.
  3. Pre/post-conditions form a formal contract; violating pre-conditions is the caller's bug.
  4. LIFO vs FIFO: Stack removes newest; Queue removes oldest.
  5. Heap property gives O(1) access to min/max and O(log n) removal.
  6. Generic types let one ADT definition serve any element type safely.
Forouzan's ADT Framework
Forouzan defines ADTs with three components: (1) the type definition (domain of values), (2) the operations with pre/post-conditions, and (3) exceptions for pre-condition violations. This three-part structure applies uniformly to Stack, Queue, List, Tree, and all other ADTs in the chapter.
Common Exam Pitfalls
Confusing ADT with data structure — an ADT is a specification, not a memory layout. A stack can be implemented as an array or a linked list; both are valid realizations of the same ADT.
