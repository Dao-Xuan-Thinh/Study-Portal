SE Fundamentals

Software Engineering = disciplined application of scientific & mathematical principles to design, develop, test, and maintain software economically and reliably

Software Engineering emerged in response to the software crisis of the 1960s–70s: projects were over budget, late, and unreliable.

The Software Crisis
  • Projects routinely exceeded cost and schedule estimates
  • Delivered software often failed to meet user needs
  • Code was difficult to maintain and evolve
  • No systematic methods — programming was an art, not engineering

SE vs. Programming
SE addresses the entire lifecycle — planning, analysis, design, coding, testing, deployment, maintenance — not just writing code.
Correctness vs Reliability
Correctness is a static, specification-based property. Reliability is a statistical, run-time property measured over many executions.

SDLC Models

SDLC = Software Development Life Cycle — structured process guiding a project from conception to retirement

A SDLC model defines the phases, their sequence, and how they interact.

Waterfall Model
  • Phases (linear): Requirements → Design → Implementation → Testing → Maintenance
  • Pro: Simple, easy to manage; clear milestones
  • Con: Rigid; late discovery of errors is costly
  • Use: Well-understood, stable requirements (e.g., embedded firmware)
Incremental Model
  • System built and delivered in increments; each adds functionality
  • Pro: Early working software; easier to accommodate change
  • Con: Architecture may degrade without careful management
  • Use: Large systems where partial delivery has value
Spiral Model
  • Iterates four quadrants: Plan → Risk Analysis → Engineer → Evaluate
  • Pro: Explicit risk management; supports prototyping
  • Con: Complex; requires risk expertise; expensive
  • Use: High-risk, large-scale projects
Agile (XP / Scrum)
  • Short sprints (1–4 weeks); continuous delivery and feedback
  • Pro: Flexible; customer collaboration; fast response to change
  • Con: Requires active customer involvement; hard to predict end date
  • Use: Evolving requirements; web/mobile products
Model Comparison Shortcut
Waterfall = sequential & predictive · Incremental = builds in layers · Spiral = risk-driven iterations · Agile = people & change over process & plans

Requirements Engineering

Requirements = description of what the system must do and the constraints under which it must operate
Functional Requirements

Define what the system does — specific behaviours, computations, outputs for given inputs.

  • Example: "The system shall allow users to log in with a username and password."
  • Directly testable; each maps to one or more test cases
Non-Functional Requirements

Define how well the system does it — quality constraints.

  • Performance — response time, throughput
  • Security — authentication, encryption standards
  • Usability — learnability, accessibility
  • Scalability — concurrent user load
  • Reliability — uptime SLA, MTBF
Use Cases

A use case describes a goal-driven interaction between an actor (user or external system) and the system.

  • Components: actor, precondition, main flow, alternate flow, postcondition
  • Expressed in structured natural language or UML use case diagrams
User Stories (Agile)
As a [role], I want [feature] so that [benefit].
  • Short, informal; acceptance criteria attached
  • Refined into tasks during sprint planning
Requirements Specification (SRS)
  • IEEE 830 / ISO 29148 define standard SRS structure
  • Must be: complete, consistent, unambiguous, verifiable, traceable
Requirements Creep
Uncontrolled addition of new requirements after baseline is set. Mitigate with formal change-control process.

Design Principles

Good design = high cohesion + low coupling + clear abstraction + information hiding
Modularity

Modularity partitions a system into independent modules. Benefits: parallel development, easier testing, isolated fault containment.

Cohesion — Higher is Better ✓

Cohesion measures how closely related the responsibilities inside one module are.

TypeDescriptionQuality
FunctionalAll elements contribute to one well-defined taskBest
SequentialOutput of one element feeds the nextGood
CommunicationalElements operate on same dataOK
CoincidentalElements grouped arbitrarilyWorst
Coupling — Lower is Better ✓

Coupling measures how much one module depends on another.

TypeDescriptionQuality
DataModules share only parametersBest
ControlOne module controls flow of anotherOK
ContentOne module modifies internals of anotherWorst
Abstraction & Information Hiding
  • Abstraction: expose only essential details; suppress irrelevant implementation
  • Information Hiding: each module conceals its design decisions (Parnas principle)
  • Enables change in one module without impacting others
Design Patterns (Overview)
  • Creational — Factory, Singleton, Builder
  • Structural — Adapter, Decorator, Facade
  • Behavioral — Observer, Strategy, Command
Design Patterns
Reusable solutions to recurring design problems (GoF, 1994). They encode expert knowledge, not ready-to-use code.

Testing

Testing = executing a program with the intent of finding errors (Dijkstra: testing shows presence, not absence of bugs)
Testing Levels
  1. Unit Testing — individual functions or classes; typically by developer; uses stubs and drivers
  2. Integration Testing — combined modules; verifies interfaces; bottom-up, top-down, or big-bang
  3. System Testing — complete system against functional & non-functional requirements
  4. Acceptance Testing — customer validates system meets business needs; alpha/beta variants

Exhaustive Testing is Impossible
For any non-trivial program, the input space is infinite. Strategy: select a representative subset using risk and coverage criteria.
Regression Testing
Re-running tests after a change to ensure no previously passing tests now fail. Essential in Agile and CI/CD pipelines.

Project Management

Project Management = planning, organising, staffing, directing, and controlling a software project to meet scope, schedule, and budget goals
Work Breakdown Structure (WBS)

Hierarchical decomposition of all project work into manageable tasks. Each leaf node is a work package that can be estimated and assigned.

Scheduling Tools
ToolDescription
Gantt ChartBar chart; tasks vs time; shows parallelism and progress
PERTNetwork diagram; critical path; handles uncertainty via optimistic/pessimistic/most-likely estimates
CPMCritical Path Method; identifies the longest path → minimum project duration
PERT Time Formula
T_e = (T_o + 4·T_m + T_p) / 6

T_o = optimistic · T_m = most likely · T_p = pessimistic

Effort Estimation
  • LOC (Lines of Code): straightforward but language-dependent; derived from historical data
  • Function Points: measure functionality from user perspective (inputs, outputs, queries, files, interfaces); language-independent
  • COCOMO: Constructive Cost Model — effort = a × (KLOC)^b with calibration constants
Risk Management
  1. Identify risks (technical, schedule, resource, business)
  2. Analyse likelihood × impact → risk exposure
  3. Plan mitigation strategies (avoid, transfer, reduce, accept)
  4. Monitor risks throughout project
Team Structures
  • Chief Programmer Team: one senior architect directs; efficient but bottleneck risk
  • Democratic (Egoless): flat structure; peer review culture; good for research-type work
  • Mixed: hierarchical but collaborative; most common in industry
Brooks's Law
"Adding manpower to a late software project makes it later." New members need onboarding time and increase communication overhead.

Quick Review — Must-Know Facts
SE Definition
Software engineering applies engineering discipline to software: systematic, disciplined, quantifiable approach to development, operation, and maintenance.
SDLC Model Selection
Choose Waterfall for stable requirements; Spiral when risk is high; Agile when requirements evolve rapidly; Incremental when early delivery provides value.
Cohesion & Coupling Rule
Maximise cohesion within modules; minimise coupling between modules. These two principles are the foundation of modular design.
Testing ≠ Debugging
Testing reveals the existence of faults; debugging identifies their location and cause so they can be fixed.
Quality Attributes Recap
Correctness · Reliability · Robustness · Efficiency · Maintainability · Portability
Mnemonic: CR·REMP or "Can Really Robust Engineers Make Portable code?"
PERT Expected Time
Te = (To + 4·Tm + Tp) / 6 — weighted average giving most weight to the most-likely estimate.
Function Points
Count user-visible functionality (inputs, outputs, queries, internal files, external interfaces), apply complexity weights → language-independent size metric.
Requirements Must Be…
Complete · Consistent · Unambiguous · Verifiable · Traceable. Any violation leads to downstream defects that are exponentially more costly to fix.
