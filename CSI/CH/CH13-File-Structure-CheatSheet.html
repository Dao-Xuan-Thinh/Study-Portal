File Concepts

File = named, persistent collection of related records stored on secondary storage
Data Hierarchy

Bit → Byte → Field → Record → File → Database

Key Definitions
UnitDescription
FieldSmallest named unit of data; has a type and length (e.g. age INT)
RecordCollection of related fields; describes one entity (e.g. one employee)
FileCollection of related records; named and persisted on disk
Key fieldField whose value uniquely identifies a record within the file
File Attributes (Metadata)

name · type · size · location · timestamps (created/modified/accessed) · owner · permissions · FCB pointer

Text vs Binary Files
TypeEncodingUse
TextPrintable ASCII/Unicode; newline-delimitedSource code, config, CSV
BinaryRaw byte patterns; compact, machine-readableImages, executables, DBs
File Operations
  • Create — allocate space; add entry to directory
  • Open — load FCB into memory; return file descriptor
  • Close — flush buffers; release file descriptor
  • Read — copy bytes from file into process buffer
  • Write — copy bytes from process buffer into file
  • Seek — reposition file pointer (random access)
  • Delete — release space; remove directory entry
Open-File Table
The OS maintains a per-process open-file table. Each open() call returns an integer file descriptor (fd) that indexes into this table; the table stores the current file pointer, access mode, and a reference count.

Sequential File Organization

Records stored one after another in physical (insertion) order; access is strictly serial
Fixed-Length Records

Every record occupies exactly R bytes. The byte offset of record i can be computed directly:

offset(i) = i × R (0-based index) Example: R=128 B, record 5 → offset = 640 B
  • Simple implementation; O(1) random access by record number
  • Wasted space when fields are shorter than their maximum size
  • Deletion leaves "holes" — must compact or mark as deleted
Variable-Length Records
  • Records differ in size; use length-prefix or delimiter approach
  • Length-prefix: first 2–4 bytes of each record store its byte length
  • Delimiter: special byte sequence (e.g. \n, \0) marks record end
  • Enables efficient storage of VARCHAR fields; no wasted padding
  • Random access requires scanning from beginning — O(n) in worst case
Advantages
  • Simplest organization to implement and understand
  • Excellent cache/disk locality for sequential scans
  • Efficient for full-file batch processing
Disadvantages
  • No efficient random/keyed access — must read from start
  • Insertion in sorted order forces shifting all subsequent records
  • Deletion requires compaction or tombstone marking
Ideal Use Cases
Payroll processing, monthly billing runs, transaction log replay, data backups — any workload that processes every record in order.

Indexed File Organization

A separate index structure maps search-key values to record addresses, enabling O(log n) keyed access
Index Structure

An index is a file of (search-key, pointer) pairs, sorted by key. The pointer gives the disk address (block + offset) of the matching record.

Dense vs Sparse Index
TypeEntriesRequirement
DenseOne entry per recordWorks on any file order; larger index
SparseOne entry per blockFile must be sorted on search key; smaller index
Primary vs Secondary Index
  • Primary index: built on the ordering key of the data file; one entry per block (sparse is typical)
  • Secondary index: built on a non-ordering attribute; must be dense; may have duplicate keys → bucket of pointers
B-Tree Index

A self-balancing tree where all leaves are at the same depth. Used by virtually every DBMS for its index structures.

Tree depth = ⌈log_m(N)⌉ m = order (max pointers per node) N = number of indexed keys
  • Each internal node holds up to m−1 keys and m child pointers
  • Search, insert, delete: O(logm N)
  • B+ tree variant: all data in leaves; inner nodes hold keys only → efficient range scans via leaf linked list
Indexed Lookup Process
  1. Binary-search the index (or traverse B-tree) for target key
  2. Retrieve the block pointer from the matching index entry
  3. Read that disk block into memory buffer
  4. Scan block for exact record match
B+ Tree Advantage
Because all records are in leaves linked in sorted order, a range query (e.g. WHERE age BETWEEN 20 AND 30) traverses only the leaf level — no backtracking through internal nodes.

Hashed File Organization

A hash function maps a record key directly to a bucket address — average O(1) access with no index traversal
Hash Function

A hash function h(k) converts a key value to a bucket number in the range [0, B−1].

Division method: h(k) = k mod B Folding: split k into parts, add them, mod B Mid-square: square k, take middle digits, mod B
  • Choose B as a prime number to reduce clustering
  • Bucket = disk block (or page) holding one or more records
Collision Handling

A collision occurs when two keys produce the same bucket address.

StrategyDescription
Open AddressingProbe next available slot: linear (+1), quadratic (+i²), double hashing
ChainingEach bucket points to an overflow linked list; no clustering
Overflow AreaSeparate file region reserved for overflow records
Load Factor
α = n / B (n = records stored, B = buckets) Ideal range: 0.70 – 0.80 α > 0.9 → performance degrades rapidly
Rehashing
  • When α exceeds threshold, allocate ~2× buckets and redistribute all records
  • Extendible hashing: splits only the overflowing bucket using a directory; avoids full rehash
  • Linear hashing: splits buckets in a fixed round-robin order as load grows
  • Cost of rehash: O(n) — all n records must be re-hashed and rewritten
Range Queries
Hashing destroys key order. Do not use hashed organization for range queries (e.g. WHERE salary BETWEEN 50000 AND 80000) — use an indexed file instead.
When to Use Hashing
Exact-match lookups on a primary key with uniform key distribution: e.g., employee lookup by ID, DNS record lookup, symbol tables in compilers.

Directory Structures

A directory is a symbol table mapping file names to their File Control Block (FCB / inode) metadata
Single-Level Directory
  • All files in one flat namespace; simplest implementation
  • Problem: name collisions in multi-user systems; no grouping
Two-Level Directory
  • Master File Directory (MFD) → per-user User File Directory (UFD)
  • Isolates namespaces; users may have identically named files
  • Problem: users cannot share files; no subdirectory support
Tree-Structured Directory
  • Arbitrary depth; each node is a file or a directory
  • Current Working Directory (CWD) maintained per process
  • Enables intuitive hierarchical organisation of files
  • Absolute path: from root → every directory → file
  • Relative path: from CWD; .. = parent directory
Acyclic Graph Directory
  • Shared files/subdirectories via links — one physical file, multiple directory entries
  • Hard link: second directory entry pointing to same inode; reference-counted
  • Symbolic (soft) link: special file containing target path; can cross file systems
  • No cycles allowed → deletion is safe using reference counts
Mounting

A file system on a separate device is mounted at a mount point — a directory in the existing tree. After mounting, the root of the new file system replaces the mount point.

General Graph Directories
If cycles are permitted (e.g. via hard links to directories), simple reference counting fails. The OS must run a garbage-collection cycle to reclaim unreachable storage.

File Access & Protection

Access Methods
MethodDescriptionBest For
SequentialRead/write in order; pointer auto-advances; can rewindLog files, tape backups
Direct (Random)Seek to any block number; O(1) for fixed-length recordsDatabases, OS paging
IndexedLook up key in index, then direct-access the blockDBMS table access
Access Rights
  • Read (r) — read file contents; list directory entries
  • Write (w) — modify or truncate file contents
  • Execute (x) — run file as a program; traverse directory
  • Append (a) — add data to end of file only; no overwrite
  • Delete (d) — remove file from directory; free its space
  • List (l) — view directory entries and file names
Unix Permission Model
Permissions: rwx rwx rwx owner group others Example: 754 = rwx r-x r-- owner: read+write+exec group: read+exec others: read only Special bits: setuid(4), setgid(2), sticky(1)
Access Control List (ACL)

An ACL stores a list of (user or group, allowed operations) pairs for each file or directory.

  • More flexible than Unix 3-class model — grant access to arbitrary users
  • Windows NTFS and POSIX.1e both implement ACLs
  • Can combine with Unix bits: check ACL first, then fall back to mode bits
  • ACL entry format: (alice, rw), (dev-team, r-x), (*, ---)
Principle of Least Privilege
Grant each user/process only the minimum access rights needed to perform its task. This limits the damage that can result from accidental or malicious misuse.
Write ≠ Delete
In many systems a user can hold write permission on a file without holding delete permission. Deletion requires write permission on the containing directory, not just the file itself.

Key Distinctions to Remember
  • Dense vs Sparse: dense = one entry per record; sparse = one per block (needs sorted file)
  • Primary vs Secondary index: primary is on ordering key; secondary can be on any field
  • Hard link vs Symbolic link: hard link shares inode; symlink stores path string
  • Write ≠ Delete: deleting a file requires write on the directory, not just the file
  • Hashing vs Indexing: hash for exact match; index (B+ tree) for range or ordered access
  • Open addressing vs Chaining: addressing keeps data in table; chaining uses overflow lists
Exam Tip
Know the trade-offs: sequential is fastest for batch scans, hashing for point lookups, B+ tree indexes for mixed workloads. Directory questions often ask about absolute vs relative paths and link types.
