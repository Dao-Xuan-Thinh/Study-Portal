Algorithm Concepts — §8.1

An algorithm is an ordered set of unambiguous steps that produces a result and terminates in a finite time.
Definition

An algorithm is a step-by-step procedure to solve a well-defined problem. It is independent of programming language and hardware — it is a logical construct.

Five Essential Properties
PropertyMeaning
FinitenessMust terminate after a finite number of steps — no infinite loops
DefinitenessEvery step must be precisely and unambiguously defined
InputZero or more well-defined inputs supplied before execution
OutputOne or more outputs — the result of the algorithm
EffectivenessEach step is basic enough to be carried out in finite time
Informal vs Formal Description
  • Informal: natural language description — easy to write, prone to ambiguity
  • Formal: pseudocode or flowchart — structured, unambiguous, closer to implementation
Pseudocode vs Flowchart
ToolFormatBest For
Pseudocode Text, structured English Sequential description, textbooks, code translation
Flowchart Diagram with symbols Visualising control flow, decision logic, loops
Pseudocode Conventions (Forouzan)
Keywords: Algorithm, Input, Output, if…else…end if, while…end while, for…end for. Indentation shows nesting; ← denotes assignment.

Three Constructs — §8.2

Any algorithm can be expressed using exactly three constructs: sequence, decision, and repetition (Böhm–Jacopini theorem).
1 · Sequence

Statements execute one after another in the order written. No branching or looping.

Algorithm AddTwo
  Input: a, b
  sum ← a + b
  Output: sum
2 · Decision (Selection)

Chooses between paths based on a condition. Two forms:

if (condition)
  action_A
else
  action_B
end if
switch (variable)
  case val1: action_1
  case val2: action_2
  default:   action_default
end switch
Switch vs If-Else
Use switch for multi-way selection on discrete values; if/else for range comparisons or complex boolean conditions.
3 · Repetition (Loop)
LoopTest PositionMin Executions
whilePre-test (top)0
forPre-test + counter0
do-whilePost-test (bottom)1
i ← 1
while (i ≤ n)
  process(i)
  i ← i + 1
end while
for i ← 1 to n
  process(i)
end for
Loop Invariant
A condition that holds true before, during, and after each loop iteration. Used to prove correctness of repetition constructs.

Algorithm Design — §8.3

Good algorithm design = decompose, refine, verify. Abstract first, detail later.
Top-Down Design

Start with the whole problem as a single module, then progressively break it into smaller, more manageable sub-problems. Each level is a refinement of the level above.

  1. Define the overall goal (main module)
  2. Identify major sub-tasks
  3. Decompose each sub-task recursively
  4. Implement at the leaf level
Modular Decomposition

Each module has a single responsibility, a clear interface (inputs/outputs), and can be designed, tested, and understood independently.

  • Promotes code reuse and maintainability
  • Reduces complexity at each level
  • Enables parallel development by teams
Stepwise Refinement

Iteratively replace abstract actions with more concrete steps until the algorithm is fully specified. Each pass adds detail without changing intent.

Recursion vs Iteration
AspectRecursionIteration
MechanismFunction calls itselfLoop repeats block
MemoryCall stack growsConstant extra space
TerminationBase caseLoop condition false
EleganceConcise for tree/graph problemsExplicit, easier to trace
RiskStack overflow if deepInfinite loop if guard fails
Recursive Requirement
Every recursive algorithm must have a base case that does not recurse, otherwise it will run infinitely.
Recursion Example — Factorial
factorial(n) = if n=0 then 1 else n × factorial(n−1)
Base case: n = 0 → returns 1 without recursing.

Sorting Algorithms — §8.4

Sorting: rearrange a list of n elements into ascending (or descending) order.
Selection Sort

Find the minimum element in the unsorted portion and swap it into position.

for i ← 0 to n−2
  min ← i
  for j ← i+1 to n−1
    if list[j] < list[min]
      min ← j
    end if
  end for
  swap list[i] ↔ list[min]
end for
Bubble Sort

Repeatedly compare adjacent pairs and swap if out of order; largest element "bubbles" to end each pass.

for i ← 0 to n−2
  for j ← 0 to n−i−2
    if list[j] > list[j+1]
      swap list[j] ↔ list[j+1]
    end if
  end for
end for
Insertion Sort

Build a sorted sub-list by inserting each new element into its correct position.

for i ← 1 to n−1
  key ← list[i]
  j ← i − 1
  while (j ≥ 0) and (list[j] > key)
    list[j+1] ← list[j]
    j ← j − 1
  end while
  list[j+1] ← key
end for
Complexity Summary
AlgorithmBestAverageWorstStable?
Selection O(n²) O(n²) O(n²) No
Bubble O(n) O(n²) O(n²) Yes
Insertion O(n) O(n²) O(n²) Yes
When to Use
All three are O(n²) — fine for small n (≤ 1 000). For large n, prefer O(n log n) algorithms (merge sort, quicksort, heap sort).

Searching Algorithms — §8.5

Searching: locate a target value (key) within a collection of n elements.
Sequential (Linear) Search — O(n)

Examine each element one by one from the beginning until the key is found or the list is exhausted.

Algorithm SeqSearch(list, n, key)
  Input: list[0..n-1], target key
  i ← 0
  while (i < n) and (list[i] ≠ key)
    i ← i + 1
  end while
  if i < n
    Output: i          // found at index i
  else
    Output: −1         // not found
Preconditions
None. Sequential search works on any list — sorted, unsorted, with duplicates. The simplest search strategy.
Binary Search — O(log n)

Repeatedly halve the search interval by comparing the key to the middle element. Works only on a sorted list.

Algorithm BinSearch(list, n, key)
  Input: sorted list[0..n-1], target key
  lo ← 0 ; hi ← n − 1
  while lo ≤ hi
    mid ← (lo + hi) / 2
    if key = list[mid]
      Output: mid      // found
    else if key < list[mid]
      hi ← mid − 1    // search left half
    else
      lo ← mid + 1    // search right half
    end if
  end while
  Output: −1           // not found
Precondition: Sorted List Required
Binary search fails on unsorted data. Always verify the list is in ascending order before applying binary search.
Comparison
PropertySequentialBinary
PreconditionNoneList must be sorted
Best CaseO(1)O(1)
Worst / AverageO(n)O(log n)
Comparisons (n=1024)up to 1 024up to 10
Data structureAny array/listSorted array
Why O(log n)?
Each comparison eliminates half the remaining elements. After k steps, only n/2ᵏ elements remain. When n/2ᵏ = 1 → k = log₂ n.

Complexity Analysis — §8.6

Complexity measures how resource usage (time or space) grows as input size n increases — expressed using Big-O notation.
Big-O Notation

Big-O describes the upper bound of growth rate. We drop constants and lower-order terms.

T(n) = 5n² + 3n + 7 → O(n²)
T(n) = 2 log n + 1 → O(log n)
Common Complexity Classes (ascending)
NotationNameExample
O(1)ConstantArray index access
O(log n)LogarithmicBinary search
O(n)LinearSequential search
O(n log n)LinearithmicMerge sort, heap sort
O(n²)QuadraticBubble / selection sort
O(2ⁿ)ExponentialRecursive Fibonacci (naïve)
Growth Rate Ordering
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
Time vs Space Complexity
TypeMeasuresNotes
TimeOperations / stepsMachine-independent count
SpaceMemory usedIncludes auxiliary storage
Time-Space Trade-off
Algorithms can often trade time for space: caching computed results (memoization) uses more memory but reduces recomputation — a classic time-space trade-off.
Complexity Classes
  • P: solvable in polynomial time — tractable problems
  • NP: solutions verifiable in polynomial time — includes hard combinatorial problems
  • NP-complete: hardest problems in NP — no known poly-time solution
  • Exponential: grows faster than any polynomial — intractable for large n
Practical Cut-off
O(n²) is acceptable for n ≤ ~10 000. O(2ⁿ) becomes impractical for n > 40. Always analyse complexity before choosing an algorithm.

Space Complexity Amount of additional memory needed beyond the input itself
Modular Decomposing a problem into self-contained modules with defined interfaces

Bubble Sort Passes through the list swapping adjacent elements; O(n²) worst case
Insertion Sort Inserts each element into correct position in growing sorted sub-list
Quick Review
Algorithm Must-Haves
Finite · Definite · Has input(s) · Produces output(s) · Effective. If any one of these is missing, the procedure is not a valid algorithm.
Three Constructs Theorem
Böhm & Jacopini (1966) proved that any computable algorithm can be expressed using only sequence, decision, and repetition. No goto needed.
Top-Down vs Bottom-Up
Top-down: start from the whole, decompose downward (Forouzan's approach).
Bottom-up: build small primitives first, compose upward. Used in some OOP and incremental development.
Sorting Choice Guide
Small n (<1 000): insertion sort.
Nearly sorted: insertion or bubble.
General large n: merge sort O(n log n).
In-place with average good perf: quicksort.
