OS Concepts

OS = software layer between hardware and user programs; manages resources and provides services

An OS acts as a resource manager (CPU, memory, I/O) and as an extended machine hiding hardware complexity from applications.

OS Types
TypeDescription
BatchJobs queued and executed without user interaction; maximises CPU utilisation
Time-sharingMultiple users share CPU via rapid context switches; gives illusion of simultaneity
Real-timeGuarantees response within strict deadline; hard vs. soft RT
DistributedCoordinates multiple networked computers; transparent resource sharing
EmbeddedMinimal footprint OS in ROM for dedicated hardware (IoT, appliances)
Kernel vs User Space
  • Kernel space: privileged mode — direct hardware access, runs OS core (scheduler, MM, drivers)
  • User space: restricted mode — applications run here; must use system calls to request OS services
  • Mode switch triggered by syscall / trap / interrupt
OS Services Provided
  • Program execution — load, run, terminate programs
  • I/O operations — uniform interface for files and devices
  • File-system manipulation — create/delete/read/write files and directories
  • Communications — between processes on same or different machines
  • Error detection — hardware faults, program errors, I/O errors
  • Resource allocation — CPU cycles, memory, I/O ports shared fairly
  • Accounting & security — usage tracking, access-control enforcement
Dual-Mode Operation
Hardware provides at least two modes (user / supervisor). Privileged instructions execute only in kernel mode, protecting the OS from misbehaving programs.
Microkernel vs Monolithic
Monolithic: all OS services run in one large kernel binary (Linux, traditional UNIX). Fast but complex.
Microkernel: minimal kernel; services run as user-space servers (Mach, MINIX). More modular; slightly slower IPC.

Process Management

Process = program in execution; unit of work managed by the OS
Five-State Process Model
StateMeaning
NewProcess being created; resources not yet allocated
ReadyIn memory, waiting for CPU; sits in ready queue
RunningCPU executing its instructions (only one per CPU core)
WaitingBlocked on I/O or event; not eligible for CPU
TerminatedExecution finished; resources being reclaimed
PCB Fields
  • PCB: PID, process state, program counter, CPU registers
  • Memory limits, open file table, I/O status, accounting info
Context Switching

Save running process's PCB → load next process's PCB. Pure overhead — no useful work done during switch.

CPU Scheduling Algorithms
AlgorithmKey Property
FCFSSimple; may cause convoy effect (long jobs block short)
SJFOptimal avg wait time; requires burst-length prediction
RRFair time-sharing; performance depends on quantum size
PriorityHigher-priority first; can starve low-priority (use aging)
Turnaround = Completion − Arrival  |  Waiting = Turnaround − Burst
Threads vs Processes
  • Process: independent address space; expensive creation & context switch
  • Thread: shares code/data/heap; own stack & registers; cheaper switch
  • Concurrency within one process; true parallelism on multi-core
Interprocess Communication (IPC)
  • Shared memory: fastest; processes map same physical pages; needs synchronisation
  • Message passing: send/receive via OS kernel; safe; slower for large data
  • Pipes: unidirectional byte stream; unnamed (parent-child) or named (FIFOs)
  • Sockets: bidirectional; works across network
Preemptive vs Non-Preemptive
Non-preemptive (cooperative) scheduling — process runs until it voluntarily yields or blocks. Preemptive — OS can interrupt running process at any time (needed for time-sharing).

Memory Management

MM allocates physical memory to processes, enforces protection, and supports virtual address spaces
Contiguous vs Non-Contiguous
  • Contiguous: each process occupies one unbroken physical region; simple but causes external fragmentation
  • Non-contiguous: process split across physical frames/segments; eliminates external fragmentation
Paging

Logical address space divided into fixed-size pages; physical memory divided into frames of equal size.

Physical address = Frame# × Page_size + Offset
  • Page table maps page → frame; stored in memory (TLB caches translations)
  • Eliminates external fragmentation; internal fragmentation ≤ page size − 1
Segmentation

Logical address = <segment#, offset>; each segment is a variable-length unit (code, data, stack). Supports user view of memory; can cause external fragmentation.

Virtual Memory & Page Replacement

Only active pages reside in RAM; others on disk (swap). A page fault triggers loading from disk.

AlgorithmPolicy
FIFOEvict oldest; simple but suffers Bélády's anomaly
LRUApproximates OPT; good in practice; costly to implement exactly
OPTTheoretical minimum faults; requires future knowledge
Thrashing
Process spends more time paging than executing. Fix: increase frames or reduce degree of multiprogramming (working-set model).
Address Translation Steps (Hardware)
  1. CPU generates logical address <page#, offset>
  2. Check TLB for page# → frame# mapping (hit: skip step 3)
  3. On TLB miss: walk page table in RAM; load entry into TLB
  4. Compute physical address = frame# × page_size + offset
  5. If page not in RAM (page fault): trap to OS, load from swap, retry

File System

File = named, persistent collection of related data; OS provides uniform access interface
File Attributes
  • Name, type/extension, size, timestamps (create/modify/access)
  • Permissions (owner/group/other: read/write/execute)
  • Location: pointer to first block or inode on disk
Directory Structures
StructureCharacteristics
Single-levelAll files in one directory; simple but name collision for multiple users
Two-levelSeparate directory per user; solves collision; no grouping within user
Tree (hierarchical)Arbitrary depth subdirectories; absolute & relative paths; most common
Access Methods
  • Sequential: records read/written in order; simple; supports tapes
  • Direct (random): jump to any block by number; needed for databases
  • Indexed: index file maps keys → block addresses; enables fast keyed lookup
Disk Allocation Methods
MethodPros / Cons
ContiguousFast sequential & direct access; external fragmentation, fixed size
LinkedNo external fragmentation; poor direct access; pointer overhead per block
IndexedGood direct access; index block overhead; inode in Unix
FAT vs inode
FAT (File Allocation Table) is a linked-list-in-table variant; inode stores up to 12 direct + indirect pointers enabling very large files.
Disk Scheduling Algorithms
AlgorithmHead Movement Policy
FCFSService requests in arrival order; simple but high seek time
SSTFShortest Seek Time First; greedy; may starve far-away requests
SCAN (elevator)Move in one direction servicing all requests; reverse at end
C-SCANCircular SCAN; only services on forward sweep; more uniform wait

Device Management

I/O subsystem hides hardware diversity; provides uniform interface to devices via device drivers
I/O Subsystem Layers
  1. User-level I/O software (libraries, syscall wrappers)
  2. Device-independent OS layer (buffering, caching, error reporting)
  3. Device drivers (device-specific; supplied by hardware vendors)
  4. Interrupt handlers (bottom half; respond to hardware events)
Interrupt Handling
  • Device signals CPU via interrupt line when operation completes
  • CPU saves state, jumps to ISR in interrupt vector table
  • ISR acknowledges device, processes data, restores CPU state
  • Maskable vs. non-maskable (NMI) interrupts
DMA (Direct Memory Access)

DMA controller transfers blocks of data directly between device and RAM without CPU involvement. CPU sets up transfer then continues executing; DMA raises interrupt on completion.

CPU freed = transfer_size / bus_bandwidth (approx. CPU cycles saved)
Buffering & Spooling
  • Buffering: temporary storage to absorb speed mismatch between device and process
  • Caching: copy of frequently accessed data kept faster memory
  • Spooling: disk used as large buffer for devices that cannot interleave (e.g., printer)
Polling vs Interrupts
Polling burns CPU cycles checking device status. Interrupts are efficient for infrequent events; polling may be faster for very-high-rate I/O (avoids interrupt overhead).
I/O Performance Concepts
  • Seek time: time to move disk head to target track
  • Rotational latency: wait for target sector to rotate under head
  • Transfer time: time to read/write the data
  • Access time = seek + rotational latency + transfer time
  • SSDs eliminate seek & rotational latency; access time ~microseconds

Deadlock

Deadlock = set of processes permanently blocked, each waiting for a resource held by another in the set
Four Necessary Conditions (Coffman, 1971)
ConditionMeaning
Mutual ExclusionAt least one resource must be held non-shareably
Hold & WaitProcess holds ≥1 resource while waiting to acquire more
No PreemptionResources cannot be forcibly taken; only voluntarily released
Circular WaitChain P1→R1→P2→R2→…→Pn→Rn→P1 exists
All Four Must Hold
Deadlock occurs only if all four conditions hold simultaneously. Preventing any one condition prevents deadlock.
Prevention
  • Break Hold & Wait: require all resources requested at once
  • Break Circular Wait: impose global ordering on resource types
  • Allow Preemption: forcibly reclaim resources from waiting processes
Avoidance — Banker's Algorithm

Before granting a request, simulate allocation and check if remaining resources can satisfy all processes' maximum needs in some order (safe state). Requires advance declaration of maximum resource needs.

Safe state ⟺ ∃ safe sequence <P1,P2,…,Pn> where each Pi's needs ≤ available + held by Pj (j<i)
Detection & Recovery
  • Detection: wait-for graph cycle search (resource-allocation graph); run periodically or on low utilisation
  • Recovery: process termination (all or one-by-one) or resource preemption (with rollback)
Ostrich Algorithm
Most general-purpose OSes (UNIX, Windows) ignore deadlock — the frequency is too low to justify overhead of full prevention or avoidance.
Banker's Algorithm — Step-by-Step
  1. Maintain: Available[], Allocation[][], Max[][]; compute Need = Max − Allocation
  2. When process Pi requests resources: verify Request ≤ Need[i] and Request ≤ Available
  3. Tentatively allocate: Available −= Request; Allocation[i] += Request; Need[i] −= Request
  4. Run safety algorithm: find process whose Need ≤ Work; add its Allocation to Work; repeat
  5. If all processes found → safe state → grant request; else → unsafe → roll back & block Pi
Need[i][j] = Max[i][j] − Allocation[i][j]

OS Architecture Comparison
ArchitectureExamplesProsCons
MonolithicLinux, BSD, traditional UNIXFast inter-component calls; mature ecosystemLarge, hard to maintain; one bug can crash kernel
MicrokernelMach, MINIX, QNXFault isolation; extensible; small TCBIPC overhead; slower than monolithic for I/O
HybridWindows NT, macOS (XNU)Balance of performance and modularityComplexity of two approaches
ExokernelMIT Exokernel (research)Application-controlled resource managementComplex application design; less portable
Virtual Machine (VM) & Hypervisor
  • Hypervisor (VMM) abstracts hardware; runs multiple guest OSes simultaneously
  • Type 1 (bare-metal): runs directly on hardware — VMware ESXi, Hyper-V, KVM
  • Type 2 (hosted): runs on top of host OS — VirtualBox, VMware Workstation
  • Guest OS makes hypercalls (or uses paravirtualisation) for privileged operations
  • Containers (Docker) share host kernel; lighter than full VMs; less isolation
Protection Rings
x86 hardware defines rings 0–3. Ring 0 = kernel (most privileged); Ring 3 = user applications (least privileged). Hypervisors may use Ring −1 (VMX root mode on Intel VT-x).
PrimitiveTypeKey Property
MutexBinary lockOnly the locker can unlock; ownership enforced
SemaphoreInteger counterwait() decrements; signal() increments; any thread can signal
MonitorHigh-level constructMutual exclusion implicit; condition variables for wait/notify
SpinlockBusy-wait lockFast for very short critical sections on multi-core; wastes CPU
Race Condition
Outcome depends on exact interleaving of concurrent accesses to shared data. Prevent with mutual exclusion around critical sections.
Classic Synchronisation Problems
Producer-Consumer: bounded buffer with full/empty semaphores.
Readers-Writers: multiple concurrent readers OR one exclusive writer.
Dining Philosophers: illustrates circular-wait deadlock with resource allocation.
Exam Focus
FCS4e Ch. 7 exam questions frequently ask: (1) process-state transitions, (2) scheduling algorithm Gantt charts, (3) page-replacement trace tables, and (4) deadlock condition identification. Practise worked examples for all four.
Common Mistakes
Confusing starvation (indefinite postponement) with deadlock (circular wait). Starvation can occur without deadlock (e.g., priority scheduling always prefers high-priority jobs).
