§3.1 Data Types Overview

All computer data — numbers, text, images, audio, video — is stored as patterns of 0s and 1s

The meaning of a bit pattern depends on context (data type):

  • 01000001 = 65 (unsigned integer)
  • 01000001 = 'A' (ASCII)
  • 01000001 = something else (float component)

Note
The computer doesn't know what a bit pattern "means" — only the program's data type declarations give meaning to the raw bits stored in memory.

§3.2 Integers — Unsigned

Unsigned integers use ALL bits for magnitude — no sign representation

n-bit unsigned range: 0 to 2ⁿ − 1

640~1.8 × 10¹⁹Large file sizes, timestamps
n-bit unsigned range: [0, 2ⁿ − 1]
Overflow behaviour
  • Adding 1 to max wraps back to 0
  • 8-bit: 255 + 1 = 0 (wraps around)
⚠ Warning
Overflow is a common source of bugs and security vulnerabilities — if a program expects positive values and wraps to 0, logic errors can occur.

§3.2 Integers — Signed

Two's complement is the universal standard for signed integers in modern hardware
Sign-Magnitude (older / simpler)
  • MSB = sign bit: 0 = positive, 1 = negative
  • +25 = 00011001  |  −25 = 10011001 (8-bit)
  • Problem: TWO zeros (+0 = 00000000, −0 = 10000000)
  • Problem: Addition requires checking signs separately
Two's Complement (modern standard)
  • −N = flip all bits of N, then add 1
  • Example −25 in 8-bit:
    +25 = 00011001 → flip = 11100110 → +1 = 11100111
  • Only ONE zero representation (00000000)
  • Addition and subtraction use the SAME hardware circuit

§3.3 IEEE 754 Floating-Point

IEEE 754 is the universal standard for storing real numbers — supported by all modern CPUs

Scientific notation in binary: ±1.fraction × 2exponent

32-bit Single Precision layout
  • [sign 1 bit][exponent 8 bits][mantissa 23 bits]
  • Sign: 0 = positive, 1 = negative
  • Biased exponent: stored value = true exponent + 127 (bias)
  • Mantissa: fractional part after the implicit leading 1
Value = (−1)^sign × 1.mantissa × 2^(exponent − 127)

64-bit Double Precision: [sign 1][exponent 11][mantissa 52], bias = 1023

TypeBitsExpMantissaApprox RangePrecision
Single (float)328 bits23 bits±3.4×10³⁸~7 decimal digits
Double6411 bits52 bits±1.8×10³⁰⁸~15 decimal digits
Special Values
  • +Inf: exp=11111111, mant=0
  • −Inf: exp=11111111, mant=0, sign=1
  • NaN: exp=11111111, mant≠0
  • ±0: exp=00000000, mant=0
Encoding Example: +6.75₁₀
  • 6.75 = 110.11₂ = 1.1011 × 2²
  • Sign = 0
  • Exponent = 2 + 127 = 129 = 10000001₂
  • Mantissa = 10110000000000000000000
  • Final: 0 10000001 10110000000000000000000
⚠ Precision Warning
Floating-point arithmetic is NOT exact. 0.1 + 0.2 = 0.30000000000000004 in most languages. Never use == to compare floats — use |a−b| < ε.
Converting Decimal Fraction → Binary
  • Multiply fraction by 2 repeatedly; integer part of each result is the next bit
  • 0.625 × 2 = 1.25 → bit=1; 0.25 × 2 = 0.5 → bit=0; 0.5 × 2 = 1.0 → bit=1
  • 0.625₁₀ = 0.101₂ (terminates)  |  0.1₁₀ never terminates in binary!
Common Floating-Point Gotchas
  • Loss of significance: subtracting nearly-equal numbers loses precision
  • Catastrophic cancellation: (1 + ε) − 1 ≠ ε when ε is very small
  • Rounding modes: round-to-nearest (default), round-up, round-down, round-to-zero
  • Denormal numbers: very small values near zero with reduced precision

§3.4 Text — ASCII & Unicode

Text is stored as numeric code points — each character maps to a unique integer
ASCII (7-bit, 128 characters)
  • 0–31: control characters (NUL, TAB, LF, CR, ESC…)
  • 32–126: printable (space, A–Z, a–z, 0–9, punctuation)
CharDecCharDecCharDec
Space32'0'48'9'57
'A'65'Z'90'a'97
'z'122—
Unicode — all human writing systems (150,000+ chars)
  • Range: U+0000 to U+10FFFF
  • U+0041 = 'A'  |  U+4E2D = '中'  |  U+1F600 = '😀'
UTF-8 encoding (most common — web, Linux, modern software)
  • 1 byte: U+0000–U+007F (ASCII-compatible!)
  • 2 bytes: U+0080–U+07FF (Latin, Greek, Arabic, Hebrew…)
  • 3 bytes: U+0800–U+FFFF (CJK, most symbols)
  • 4 bytes: U+10000–U+10FFFF (emoji, rare scripts)
Other Encodings
  • UTF-16: 2 or 4 bytes; used in Windows internals, Java, JavaScript
  • UTF-32: fixed 4 bytes; simple indexing but 4× larger than UTF-8
String Storage in Memory
  • C strings: null-terminated — last byte is 0x00 (NUL)
  • Pascal / Java strings: length-prefixed — first bytes store character count
  • Python 3: internally uses Latin-1, UCS-2, or UCS-4 depending on contents
  • Byte Order Mark (BOM): optional prefix indicating UTF-16/32 endianness
Encoding Pitfalls
  • Mojibake: garbled text from reading with wrong encoding (e.g., UTF-8 as Latin-1)
  • Always specify encoding explicitly — never rely on OS default
  • HTTP Content-Type: text/html; charset=UTF-8 declares encoding

§3.5 Audio Storage

Audio is digitized by sampling the analog waveform at regular intervals
Analog-to-Digital Conversion (ADC)
  1. Sample: measure amplitude at regular time intervals
  2. Quantize: round to nearest discrete level
  3. Encode: store as binary number
Sampling Rate (Hz)
  • CD quality: 44,100 Hz (captures up to 22 kHz — Nyquist: must be 2× max freq)
  • DVD / Studio: 48,000–96,000 Hz
  • Telephone: 8,000 Hz
Bit Depth (bits per sample — resolution)
  • 8-bit: 256 levels (low quality)
  • 16-bit: 65,536 levels (CD quality)
  • 24-bit: 16.7 million levels (professional)
Uncompressed size = sample_rate × bit_depth × channels × duration / 8 (bytes)
Example: 3-min stereo CD = 44100 × 16 × 2 × 180 / 8 ≈ 31 MB
Common Audio Formats
  • WAV / AIFF — uncompressed PCM (lossless, large)
  • FLAC — lossless compression (~50–60% smaller)
  • MP3 / AAC / OGG — lossy compression (90%+ smaller)

Quantization noise decreases as bit depth increases.

Nyquist–Shannon Sampling Theorem
  • To accurately reproduce a frequency f, sample at ≥ 2f (Nyquist rate)
  • Human hearing: ~20 Hz – 20 kHz → CD uses 44,100 Hz (> 2 × 20,000)
  • Sampling below Nyquist causes aliasing — false low-frequency artefacts
Channels
  • Mono (1 ch) — telephone, podcasts
  • Stereo (2 ch) — CD, streaming music
  • Surround 5.1 (6 ch) — DVD, Blu-ray
  • Surround 7.1 (8 ch) — high-end home theatre

§3.6–3.7 Images & Video Storage

Raster (Bitmap) Images

Grid of pixels; each pixel stores color as numbers.

Color Depth (bits per pixel)
  • 1-bit: 2 colors (B&W)
  • 8-bit: 256 colors (indexed / paletted)
  • 24-bit true color: 8 bits each for R, G, B → 16.7 million colors
  • 32-bit: RGBA (+ 8-bit alpha transparency)
Image size (bytes) = width × height × (bits_per_pixel / 8)
1920×1080 true color = 1920×1080×3 ≈ 6 MB uncompressed

Video Storage

Video = sequence of frames + synchronized audio + metadata.

Frame Rates
  • 24 fps — film / cinema
  • 30 fps — broadcast TV
  • 60 fps — gaming / sports
  • 120+ fps — high-speed / slow-motion
Compression Techniques
  • Spatial: within each frame (similar to JPEG)
  • Temporal: only store differences between frames
  • I-frames (keyframes): complete image
  • P-frames: changes from previous frame
  • B-frames: interpolated between I and P frames
Codecs
  • H.264 / AVC — most widely used; excellent quality/size
  • H.265 / HEVC — 2× better than H.264; streaming 4K
  • VP9 — Google open-source; YouTube
  • AV1 — royalty-free; best compression (newest)
Raw video ≈ width × height × fps × color_depth × duration
1080p 30fps 1 min uncompressed ≈ 11 GB → H.264 ≈ 130 MB (98% smaller!)

codec = coder-decoder; container formats (MP4, AVI, MKV) wrap codec bitstreams.

Note — The "Missing Space" Mystery
Hard drive manufacturers use decimal (SI) prefixes — a "1 TB" drive = 10¹² bytes. Operating systems use binary — so it shows as ~931 GiB. This 7% "missing space" is just the unit difference.
Quick Size Reference
  • Plain text page ≈ 2–4 KB
  • MP3 song (4 min) ≈ 3–5 MB
  • HD photo (JPEG) ≈ 2–6 MB
  • Full HD movie (H.264) ≈ 1–4 GB
  • Blu-ray disc capacity ≈ 25–100 GB
SI vs Binary Prefix Quick-Convert
  • 1 GiB = 1,073,741,824 bytes  ≈  1.074 GB (decimal)
  • 1 TiB = 1,099,511,627,776 bytes  ≈  1.1 TB (decimal)
  • Ratio formula: (1024/1000)ⁿ where n = prefix order (1 for K, 2 for M, etc.)
  • IEC 80000-13 defines official KiB, MiB, GiB, TiB, PiB, EiB symbols
Context Matters
RAM and CPU cache are reported in binary (GiB). Network speeds use SI (Mbps). Storage manufacturers use SI (GB). Windows File Explorer labels binary sizes as "GB" — causing apparent "missing space." Always clarify which unit system is in use when communicating storage or memory capacities.
