Database Concepts

Database = organized collection of logically related data managed by a DBMS

A DBMS provides a systematic interface between programs/users and stored data, eliminating many limitations of traditional file systems.

DBMS vs. File System
File SystemDBMS
Data redundancy & inconsistencyCentralized control; reduced redundancy
Difficult concurrent accessBuilt-in concurrency control
No standard query languageDeclarative query language (SQL)
Limited security & integrityAuthorization, constraints enforced
Program–data dependencyData independence from programs
Data Independence
  • Physical independence: change storage structures without rewriting programs
  • Logical independence: change logical schema without rewriting programs
ANSI/SPARC 3-Level Architecture
External → Conceptual → Internal
  • External level: user/application views — what each user sees
  • Conceptual level: community view — entire logical structure of the database
  • Internal level: physical storage — how data is stored on disk
Database Users
  • DBA — manages schema, access rights, backups, tuning
  • Application developers — design and code DB-backed programs
  • End users — naive (forms/menus) or sophisticated (ad-hoc queries)
Key Benefit
Data independence separates what data exists (logical) from how it is stored (physical), enabling schema evolution without cascading rewrites.

Relational Model

Codd (1970): represent data as a set of relations (tables) with no ordering assumed
Core Terminology
Formal TermCommon TermMeaning
RelationTableSet of tuples with same attributes
TupleRow / RecordOne data instance
AttributeColumn / FieldNamed property with a domain
DomainData typeSet of allowed values
DegreeArityNumber of attributes
CardinalityRow countNumber of tuples
Keys
  • Super key: any attribute set that uniquely identifies a tuple
  • Candidate key: minimal super key — no attribute can be removed
  • Primary key — one chosen candidate key; must be unique & non-null
  • Foreign key — attribute referencing a primary key in another relation
Referential Integrity
Constraint
A foreign key value must either match an existing primary key value in the referenced relation, or be NULL. Violations are rejected by the DBMS.
Relational Algebra Operations
OperationSymbolDescription
SelectionσFilter rows by condition
ProjectionπKeep specific columns
Union∪Combine two compatible relations
Intersection∩Rows common to both
Difference−Rows in R1 but not R2
Cartesian Product×All row combinations
Join⋈Combine on matching attribute
Closed System
Every relational algebra operation takes one or more relations as input and returns a relation — enabling composition of complex queries.

SQL — Structured Query Language

SQL = standard language for defining, manipulating, and querying relational databases
DDL — Data Definition Language
  • CREATE TABLE t (col type [constraint], …) — define new table
  • ALTER TABLE t ADD|DROP|MODIFY col … — modify schema
  • DROP TABLE t — permanently remove table & data
  • CREATE INDEX idx ON t(col) — speed up lookups
DML — Data Manipulation Language
SELECT cols FROM table WHERE cond GROUP BY … HAVING … ORDER BY … LIMIT n
  • INSERT INTO t (c1,c2) VALUES (v1,v2)
  • UPDATE t SET c1=v1 WHERE cond
  • DELETE FROM t WHERE cond
WHERE Clause Operators
  • = <> < > <= >= — comparison
  • AND OR NOT — logical operators
  • BETWEEN v1 AND v2 — range test
  • LIKE 'pattern%' — wildcard string match
  • IN (v1, v2, …) — membership test
  • IS NULL / IS NOT NULL — null check
JOIN Types
JoinReturns
INNER JOINMatching rows in both tables
LEFT OUTER JOINAll left rows + matched right (NULLs for no match)
RIGHT OUTER JOINAll right rows + matched left
FULL OUTER JOINAll rows from both; NULLs where unmatched
CROSS JOINCartesian product of both tables
Aggregate Functions
  • COUNT(*) — number of rows
  • SUM(col) — total of numeric column
  • AVG(col) — arithmetic mean
  • MAX(col) / MIN(col) — extreme values
GROUP BY vs HAVING
WHERE filters rows before grouping; HAVING filters groups after aggregation. Never use an aggregate function in a WHERE clause.

Database Design — ER & Normalization

Design goal: faithful, non-redundant representation of the real-world domain
ER Model Components
ComponentNotationMeaning
EntityRectangleDistinguishable real-world object
Weak EntityDouble rectangleDepends on owner entity for identity
AttributeEllipseProperty of an entity
Key attributeUnderlined ellipseUniquely identifies entity instance
RelationshipDiamondAssociation among entity sets
Cardinality Constraints
  • 1:1 — one entity relates to exactly one other
  • 1:N — one entity relates to many others (most common)
  • M:N — many entities relate to many; requires junction table
ER-to-Relational Mapping Rules
  1. Strong entity → relation; key attribute → primary key
  2. Weak entity → relation; include owner PK as partial key
  3. 1:1 relationship → add FK to either side (choose less-null side)
  4. 1:N relationship → add FK on the N-side relation
  5. M:N relationship → create new junction relation with both PKs as composite PK
  6. Multi-valued attribute → new relation with entity PK + attribute value
Normalization

Remove update anomalies (insert, delete, modify) by decomposing relations into well-structured forms.

Normal FormRequirement
1NFAtomic values; no repeating groups
2NF1NF + no partial dependency on PK
3NF2NF + no transitive dependency
BCNFEvery determinant is a candidate key
Functional Dependency
X → Y means knowing X uniquely determines Y. Normalization eliminates problematic FDs by decomposing relations, always preserving lossless join & dependency preservation.

Transaction Management

Transaction = logical unit of work; either commits fully or rolls back completely
ACID Properties
PropertyGuarantee
Atomicity All-or-nothing execution
Consistency DB moves between valid states only
Isolation Concurrent Txns appear serial
Durability Committed changes survive crashes
Transaction States
Active → Partially Committed → Committed
Active → Failed → Aborted (Rolled Back)
Concurrency Problems
  • Lost update: two txns read & overwrite the same item; one update is lost
  • Dirty read: txn reads uncommitted data from another txn that later aborts
  • Non-repeatable read: same query returns different rows within one txn
  • Phantom read: new rows appear between two reads in same txn
Locking Protocol
  • Shared lock (S) — read-only; multiple holders allowed
  • Exclusive lock (X) — read/write; single holder only
  • Two-Phase Locking (2PL): growing phase (acquire), shrinking phase (release)
Deadlock
Deadlock Condition
Two or more transactions each hold a lock and wait for a lock held by the other. No progress is possible. Resolution: detect via wait-for graph and abort one victim transaction; or use timeouts.
Recovery — Write-Ahead Logging
Before any data page is written to disk, its log record must be flushed first (WAL rule). On crash, redo committed txns and undo uncommitted ones using the log.

Key Terms & Quick Review — Chapter 14

Key Terms
TermDefinition
DBMS Software system that stores, retrieves, and manages data in a database while enforcing integrity, security, and concurrency rules
relation A two-dimensional table of rows and columns satisfying the relational model properties (no duplicate rows, atomic values)
tuple A single row in a relation; represents one instance of the entity described by that relation
attribute A named column in a relation with a defined domain of allowed values (e.g., Age INTEGER, Name VARCHAR(50))
primary key The chosen candidate key that uniquely identifies each tuple; must be unique and NOT NULL
foreign key An attribute (or set) in one relation that references the primary key of another relation; enforces referential integrity
SQL Structured Query Language — the standard declarative language for defining (DDL) and manipulating (DML) relational databases
normalization Process of decomposing relations to remove redundancy and update anomalies; progresses through 1NF → 2NF → 3NF → BCNF
transaction A logical unit of database work that must execute completely (commit) or not at all (rollback); defined by BEGIN … COMMIT/ROLLBACK
ACID Four transaction properties: Atomicity, Consistency, Isolation, Durability — guarantee correctness even under failures and concurrency
concurrency Simultaneous execution of multiple transactions; managed via locking or optimistic schemes to prevent dirty reads, lost updates, etc.
deadlock Circular wait condition where each of two or more transactions holds a lock needed by another; resolved by aborting a victim transaction
ER model Entity-Relationship model — a high-level conceptual design tool using entities, attributes, and relationships to model a domain before physical implementation
NoSQL "Not Only SQL" — non-relational databases (key-value, document, column-family, graph) optimized for horizontal scalability and flexible schemas
data independence Ability to change a schema level (physical or logical) without requiring changes to higher-level programs or user views
referential integrity Constraint ensuring every foreign key value either matches an existing primary key in the referenced table or is NULL
Quick Review — Chapter 14
§14.1 Database Concepts
A DBMS centralizes data management, eliminating redundancy, enforcing integrity, and providing concurrency control. The ANSI/SPARC 3-level architecture (external → conceptual → internal) decouples user views from physical storage through data independence.
§14.2 Relational Model
Data is stored in relations (tables). Tuples are rows; attributes are columns. Keys uniquely identify tuples. Foreign keys enforce referential integrity. Relational algebra provides the formal query foundation; SQL is its practical syntax.
§14.3 SQL
DDL (CREATE, ALTER, DROP) defines schema; DML (SELECT, INSERT, UPDATE, DELETE) manipulates data. JOIN combines tables; aggregate functions summarize groups. WHERE filters rows; HAVING filters groups after aggregation.
§14.4 Database Design
ER modeling captures the conceptual structure using entities, attributes, and relationships with cardinality constraints. ER-to-relational mapping translates each construct to tables. Normalization (1NF→BCNF) removes redundancy and update anomalies by eliminating partial and transitive dependencies.
§14.5 Transaction Management
Transactions must satisfy ACID. Concurrency is managed by two-phase locking (shared/exclusive locks). Deadlocks arise from circular waits and are resolved by aborting a victim. Recovery uses write-ahead logging to redo committed and undo incomplete transactions after a crash.
§14.6 Database Models
Hierarchical and network models pre-date the relational model. Object-oriented databases handle complex types. NoSQL models (key-value, document, column-family, graph) sacrifice ACID for horizontal scalability and schema flexibility, following the BASE principle and CAP trade-offs.
Exam-Focus Points
  • Distinguish physical vs. logical data independence and which ANSI/SPARC level each applies to
  • Know all four ACID properties and give a concrete example of each
  • Be able to write a SELECT … JOIN … WHERE … GROUP BY … HAVING query from scratch
  • Explain the difference between 2NF and 3NF with a functional dependency example
  • Describe how 2PL prevents lost updates; explain deadlock detection via wait-for graph
  • Compare RDBMS (ACID/SQL) vs. NoSQL (BASE/scale-out) and know which NoSQL type suits a given use-case
