§18.1 Hardware Performance Issues

Why Single-Core Scaling Stopped

  • ILP WallBeyond ~6 parallel pipelines, hazards & resource dependencies prevent full utilisation; compiled binaries rarely expose more ILP
  • Power WallPower density grows exponentially with clock speed & transistor density — cooling becomes impractical
  • Memory WallDRAM latency improvements lag far behind CPU speed; cores stall waiting for data
  • Design WallIncreasing pipeline depth (e.g. Pentium 4 Prescott = 31 stages) adds logic, interconnects, and verification complexity
Pollack's Rule (1999)Performance ≈ √(Complexity). Doubling core logic gives only ~40% more performance — incentivising multicore over ever-larger single cores.

Power & Dennard Scaling Breakdown

P = C × V² × f
  • C — capacitance switched per cycle
  • V — supply voltage
  • f — clock frequency

Dennard Scaling assumed that as transistors shrink, V and f scale proportionally — keeping power density constant. This held until ~2004.

Dennard Scaling BreakdownVoltage could no longer be reduced proportionally (threshold voltage limits). Static leakage current increased. Power density soared — leading to the "Power Wall".
Dark SiliconAt advanced nodes (≤ 22 nm) designers cannot power all transistors simultaneously at full speed without exceeding thermal limits. Large fractions of the chip must remain "dark" (powered off) at any moment.

§18.2 Software Performance Issues

Amdahl's Law for Multicore

Speedup = 1 / [ s + (1−s)/N ]
  • s — serial fraction of program (1 − f)
  • f — parallelisable fraction
  • N — number of cores
Serial %N=2N=4N=8N=∞
0% (f=1.0)2×4×8×∞
2% (f=0.98)1.96×3.77×6.90×50×
5% (f=0.95)1.90×3.48×5.93×20×
10% (f=0.90)1.82×3.08×4.71×10×
Overhead EffectReal curves peak and then degrade — communication, cache coherence, and OS scheduling overhead grows with N, eventually dominating gains.

Modified law with Pollack's Rule: Speedup = 1/[(1−f)/perf(r) + f·r/(perf(r)·n)], where perf(r) ≈ √(2r)

Concurrency Hazards

  • Race ConditionTwo threads access shared data with at least one write; result depends on execution order → non-deterministic bugs
  • DeadlockThread A holds lock 1, waits for lock 2; Thread B holds lock 2, waits for lock 1 → both block forever
  • LivelockThreads continuously change state in response to each other without making progress (e.g. both backoff repeatedly)
  • StarvationA thread is perpetually denied access to a resource due to priority scheduling
  • False SharingTwo threads access different variables on the same cache line → unnecessary coherence traffic
Mutual ExclusionCritical sections must be protected so only one thread executes them at a time. Achieved via locks, semaphores, or monitors.

Threading Granularity

GranularityDescriptionTrade-off
Coarse Whole modules (rendering, AI, physics) assigned to separate cores Simple; ~1.2–2× speedup in practice (limited load balance)
Fine Loop iterations split into thousands of tiny threads Higher potential speedup; complex scheduling, high overhead
Hybrid Coarse for fixed-load modules (sound), fine-grain for heavy modules (scene rendering) Best scalability for 8–16 core systems (Valve's choice)
Valve Source EngineUsed a single-writer / multiple-readers concurrency model. Over 95% of accesses are reads — this model minimises lock contention while preserving correctness.

Parallel Application Classes

  • TLPMultithreaded native apps — small number of highly-threaded processes (e.g. games, simulations)
  • PLPMultiprocess apps — many single-threaded processes running concurrently (e.g. web servers)
  • JVMJava apps — JVM is inherently multithreaded; language-level thread support
  • ALPMulti-instance apps — multiple independent instances of the same program (isolate with VM/containers)
  • DBDatabase workloads — near-linear scaling shown in Oracle DSS, DB2 benchmarks when serial fraction minimised in all layers
§18.3 Multicore Organisation

Cache Organisation Alternatives

ConfigStructureExample
(a)Each core has its own L1 only; L2 off-chipARM11 MPCore
(b)Each core has private L1 + L2; no on-chip sharedAMD Opteron
(c)Each core has private L1; shared on-chip L2Intel Core Duo
(d)Private L1+L2 per core; shared on-chip L3 ✓ dominant todayIntel Core i7, Xeon

Advantages of Shared Higher-Level Cache:

  • Constructive interference — shared data loaded once for all cores
  • No replication of shared lines at shared cache level
  • Dynamic cache partitioning — threads with larger working sets get more
  • Easy inter-core communication via shared memory locations
  • Coherency problem confined to lower cache levels

Simultaneous Multithreading (SMT)

What is SMT?Each physical core exposes multiple hardware thread contexts (expanded register banks) sharing the same execution pipelines. To the OS, each SMT thread appears as a logical CPU.

§18.6 ARM Cortex-A15 MPCore

Cortex-A15 MPCore Overview

4

L2 Cache
  • Out-of-order, 15–24 stage pipeline; 3 instructions/cycle dispatch
  • Each core: dedicated 32 kB L1-I + 32 kB L1-D cache
  • Snoop Control Unit (SCU) maintains L1/L2 cache coherency
  • Generic Interrupt Controller (GIC) handles all interrupts centrally — supports masking, prioritisation, distribution, tracking
  • Generic per-core timer + watchdog; debug/trace unit
  • Memory-mapped I/O to GIC via private SCU interface

SCU Cache Coherency Optimisations

  • DDIDirect Data Intervention — copy clean data from one L1 to another L1 without accessing L2/main memory. Resolves local L1 miss via remote L1.
  • Dup Tag RAMsDuplicated tag RAMs — SCU maintains copies of all L1 tag arrays. Sends coherency commands only to CPUs that actually have the relevant line — reduces snooping power/traffic.
  • Migratory LinesMove dirty data from one L1 to another without write-back to L2, then re-read — saves memory bus bandwidth for frequently migrating modified lines.
L2 CoherencyL2 maintains snoop tag array (duplicate of all L1 data cache directories). Lines in M/E state belong to L1 system and must be serviced by L1. Shared/Invalid lines served directly from L2. Hybrid MESI + MOESI protocol.

GIC Interrupt Routing

The GIC can route any interrupt in three ways:

  • UnicastDirected to a single specific processor
  • MulticastDirected to a defined group; first accepting processor (typically least loaded) handles it
  • BroadcastDirected to all processors (e.g. TLB shootdown)

INTERRUPT SOURCES

  • ID0–ID15 Interprocessor Interrupts (IPIs) — software-triggered, private per CPU
  • ID16–ID31 Private timers & watchdog (ID29, ID30)
  • ID32+ Hardware interrupts — up to 224 external lines

EC12 Design Trade-offs & RAS

  • L1 kept small (fast access, 1 cycle) despite supporting 5.5 GHz core
  • Private L2/core avoids shared cache contention for hot working sets
  • eDRAM for L3 — slower than SRAM but fits 48 MB on-chip (proximity wins over speed for many workloads)
  • L3 processes every store — maintains data availability during core failure (RAS feature)
  • L4 spans books (MCMs) — accessing off-book L4 is faster than off-book DRAM
  • OS + resource manager co-locate workloads within book boundaries to minimise cross-book L4 traffic
RAS (Reliability, Availability, Serviceability)Write-through L1→L2→L3 chain means L3 always has a complete, up-to-date copy — enables on-the-fly core replacement without data loss.

Key Terms Quick Reference

Hardware Concepts

  • ILP Instruction-Level Parallelism — exploited by superscalar and SMT
  • TLP Thread-Level Parallelism — exploited by multicore
  • DVFS Dynamic Voltage & Frequency Scaling — power management
  • Dark Silicon Transistors that must remain powered off to stay within TDP
  • Pollack's Rule Perf ∝ √Complexity — motivation for multiple smaller cores
  • SoC System on Chip — integrates CPU, GPU, memory ctrl, I/O on one die
  • MCM Multi-Chip Module — multiple dies in one package (IBM EC12)

Cache / Coherence Terms

  • MESI Modified / Exclusive / Shared / Invalid — standard snoopy protocol
  • MOESI Adds Owned state — allows dirty sharing without write-back
  • ACE ARM Advanced eXtensible Interface Coherence Extensions
  • SCU Snoop Control Unit (ARM) — L1/L2 coherency manager
  • DDI Direct Data Intervention — L1-to-L1 transfer without L2
  • LLC Last-Level Cache — typically shared L3 (or L4 in mainframe)
  • eDRAM Embedded DRAM — lower density than SRAM but much larger capacity on-chip

Software / Parallelism Terms

  • Amdahl's Law Theoretical max speedup limited by serial fraction
  • Critical Section Code segment that must execute atomically
  • Mutex Mutual exclusion lock — binary semaphore
  • Deadlock Circular dependency on locks — system halts
  • False Sharing Different vars on same cache line → unnecessary invalidations
  • Memory Fence Instruction ensuring visibility ordering of memory ops
  • CAS Compare-And-Swap — fundamental lock-free atomic primitive
