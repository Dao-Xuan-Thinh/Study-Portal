§17.1 — Multiple Processor Organizations
🔀 Flynn's Taxonomy (1972) SISD / SIMD / MISD / MIMD
Class Full Name Description Examples
SISD Single Instruction, Single Data One processor executes one instruction stream on a single data stream — classic sequential computer Traditional uniprocessors
SIMD Single Instruction, Multiple Data A single instruction controls simultaneous execution across multiple processing elements, each with its own data memory (lock-step) Vector processors, GPU cores, array processors
MISD Multiple Instruction, Single Data Multiple processors execute different instructions on the same data stream — largely theoretical Not commercially implemented
MIMD Multiple Instruction, Multiple Data Each processor executes its own instruction stream on its own data; most powerful and general-purpose category SMP, clusters, NUMA, multicore
ℹ️

MIMD is the dominant modern paradigm — it subdivides into shared-memory (SMP, NUMA) and distributed-memory (clusters, MPP) systems based on how processors communicate.

🔗 MIMD: Shared-Memory Systems Tightly Coupled
  • Processors share a single physical memory (or pool)
  • Communication via reads/writes to shared memory
  • Requires cache coherence protocols
  • SMP (Symmetric Multiprocessor) — UMA, equal access time for all CPUs
  • NUMA — memory access time varies by region
✔

Easiest to program; OS presents single system image. Scales to ~64 processors.

🌐 MIMD: Distributed-Memory Systems Loosely Coupled
  • Each node has its own local memory
  • Communication via message passing over a network (LAN/WAN)
  • No global address space; no cache coherence needed
  • Clusters — commodity servers + high-speed interconnect
  • MPP (Massively Parallel Processor) — custom interconnect, 1000s of nodes
✔

Scales to thousands of nodes; higher latency communication than shared-memory.

🧠 Memory Organization Models UMA / NUMA / COMA
ModelFull NameAccess TimeDescription
UMA Uniform Memory Access Equal for all CPUs All processors have identical access latency to any memory location. Standard SMP is UMA.
NUMA Non-Uniform Memory Access Varies by region Local memory is faster; remote memory (via interconnect) is slower. CC-NUMA adds cache coherence.
COMA Cache-Only Memory Access Migration-based All main memory acts as cache; data migrates to the node that uses it. Complex coherence.
§17.2 — Symmetric Multiprocessors (SMP)
⚖️ SMP Defining Characteristics
  1. ≥ 2 similar processors of comparable capability
  2. Processors share main memory & I/O via bus/interconnect with equal access times
  3. All processors have equal access to I/O devices (same or different channels)
  4. All processors can perform the same functions (hence symmetric)
  5. A single integrated OS manages all processors and provides interaction at job/task/file/data levels
⬆️ SMP Advantages over Uniprocessor
  • Performance: parallel workloads execute across multiple CPUs simultaneously
  • Availability: single CPU failure → reduced performance, not system halt
  • Incremental growth: add processors to boost performance
  • Scaling: vendors can offer product range by varying # of CPUs
  • Transparent to user — OS hides parallelism, schedules threads/processes
⚠️

These are potential benefits; OS tools must be designed to exploit SMP parallelism.

🖧 SMP Bus/Interconnect Organization
  • Time-shared bus (common): Simplest; all processors share single bus. Low cost, easy to expand, but performance bottleneck at high CPU counts
  • Multiple buses: Redundant buses improve reliability and bandwidth
  • Crossbar switch: N×M switch matrix; any CPU can connect to any memory module simultaneously. Best bandwidth but highest cost/complexity
  • Each CPU has own L1 + L2 (±L3) cache to reduce bus accesses
⚠️

Local caches introduce the cache coherence problem — addressed by hardware (MESI protocol).

🖥️ SMP Operating System Design Considerations
  • Simultaneous concurrent processes: OS routines must be reentrant — multiple CPUs may execute the same OS code at once
  • Scheduling: Any CPU may perform scheduling; conflicts must be avoided; scheduler assigns ready processes to available CPUs
  • Synchronization: Mutual exclusion & event ordering for shared address spaces and I/O
  • Memory management: Paging and page protection mechanisms must function on all CPUs
  • Reliability & fault tolerance: OS must degrade gracefully when a CPU fails
  • Process migration: OS may move processes between CPUs to balance load
📐 Amdahl's Law for SMP Speedup Limit
// Speedup with N processors Speedup(N) = 1 ───────────────── (1 - f) + (f / N) f = fraction of code parallelizable N = number of processors (1-f) = sequential fraction (bottleneck)
  • As N → ∞: max speedup = 1/(1−f)
  • If f = 0.95 (95% parallel): max speedup ≈ 20×
  • Even small sequential portions dominate at large N
  • Real applications limited by synchronization overhead, memory bandwidth, and OS overhead
💡

Amdahl's Law sets the theoretical ceiling — practical SMP performance is always lower due to synchronization overhead and memory contention.

§17.3 — Cache Coherence & the MESI Protocol
🔄 The Cache Coherence Problem

In an SMP, multiple processors each have a private cache. If two caches hold copies of the same memory line and one processor modifies its copy, the other cache holds stale data — an inconsistent view of memory.

  • Write-back policy: Only writes to cache; memory updated only on eviction → stale copies possible in other caches
  • Write-through policy: Writes go to memory AND cache — inconsistency still occurs if other caches don't observe bus traffic
⚠️

Cache coherence is addressed at hardware level (transparent to programmer). Software-only solutions incur high overhead.

⚙️ Cache Coherence Approaches
🛠 Software Solutions
  • Compiler or OS marks shared memory pages non-cacheable
  • Compiler inserts cache flush/invalidate instructions at critical points
  • Simpler hardware but conservative — reduces cache effectiveness
  • Transparent to programmer but limits performance
⚡ Hardware Solutions — Protocols
  • Directory protocols: Central directory in main memory tracks which caches hold each line; centralized controller issues commands
  • Snoopy protocols: Each cache controller snoops the shared bus for transactions; decentralized, suited to bus-based SMPs
📡 Snoopy Protocol Strategies
✗ Write-Invalidate
  • Multiple readers, only one writer
  • Writer broadcasts invalidate signal; other caches mark line invalid
  • Writer gains exclusive ownership
  • Used by MESI
  • Fewer bus transactions for repeated writes
↺ Write-Update (Broadcast)
  • Writer broadcasts updated data to all caches
  • All cached copies updated simultaneously
  • More bus traffic per write
  • Better if multiple CPUs read same data frequently
  • Less commonly used
🗂️ MESI Protocol — Four States Table 17.1
M
Modified

Line present only in THIS cache and is dirty (differs from main memory). Cache responsible for writing back before any other CPU reads it.

E
Exclusive

Line present only in THIS cache, clean (matches main memory). Can be written without bus transaction (becomes Modified).

S
Shared

Line may be present in multiple caches, clean. Any write must first invalidate all other copies (via bus).

I
Invalid

Line is not valid in this cache (either absent or has been invalidated). Any access causes a cache miss.

ℹ️

Each cache line has its own 2-bit state field encoding its current MESI state. The protocol is used on x86 architecture.

↔️ MESI State Transition Table Processor-Initiated + Bus-Initiated Events
Event / Action Current State Next State Bus Transaction Description
Read Hit (RH) M, E, or S No change None Data served directly from local cache; no bus activity needed
Read Miss (RMS) — others share I → S BusRd Other caches signal they share; initiator reads from memory, enters Shared state
Read Miss (RME) — none share I → E BusRd No other cache holds line; initiator reads from memory, gets exclusive ownership
Read Miss — other has Modified I → S (both caches) BusRd + writeback Modified cache writes back to memory (dirty line copyback), then transitions M→S; initiator gets line in S state
Write Hit (WH) M or E → M None Already have exclusive ownership; just modify locally (E→M or stays M)
Write Hit (WH) S → M BusUpgr (Invalidate) Broadcast invalidate to all other caches sharing this line; gain exclusive ownership, then write
Write Miss (WM) I → M BusRdX (RWITM) Read-With-Intent-To-Modify: fetch line AND invalidate all other copies; modify locally, enter Modified
Snoop Hit on Read (SHR) E → S Signal shared Another CPU is reading this line; transition from Exclusive to Shared
Snoop Hit on Write (SHW) S or E → I — Another CPU is writing; invalidate this copy
📡

BusRd — bus read request (data needed, no modify intent)
BusRdX — bus read exclusive / RWITM (read + invalidate all copies)
BusUpgr — upgrade (invalidate without re-fetch, already have data in S state)

✔

At any moment a cache line exists in exactly one state. The MESI protocol guarantees global consistency by ensuring only one Modified copy can exist at any time.

§17.4 — Multithreading & Chip Multiprocessors
🧵 Thread & Process Concepts
Process
Program in execution; owns resources (memory, files, I/O). Switching processes is expensive (saves full context)
Thread
Dispatchable unit within a process; has PC, stack, registers. Multiple threads share process resources
Thread switch
Switch within same process — much cheaper than process switch (no resource reallocation)
IPC
Instructions Per Cycle — key performance metric
MIPS Rate = f × IPC f = clock frequency (MHz) IPC = avg instructions completed/cycle

Goal: maximize IPC by keeping pipeline stages busy using multithreading to hide latencies.

⚡ Approaches to Explicit Multithreading Four Methods
ApproachAlso Known AsMechanismBest ForTrade-off
Interleaved (Fine-grain) Fine-grain MT / Barrel processing Switch thread every cycle (round-robin). Zero-cycle thread switch. Hides ALL latency events Hiding all pipeline latencies, high-latency memory systems Hurts single-thread performance; threads compete for cache
Blocked (Coarse-grain) Coarse-grain MT Execute thread until a stall event (cache miss); then switch to ready thread. 1-cycle switch cost possible In-order pipelines; hiding long-latency events (L2/L3 miss) Cannot hide short latencies; wasted cycle on switch
Simultaneous MT (SMT) Intel Hyper-Threading Issue instructions from multiple threads simultaneously to a superscalar's execution units each cycle Superscalar CPUs with underutilized execution units Threads compete for L1/L2 cache; complex scheduling logic
Chip Multiprocessing (CMP) Multicore Multiple independent cores on single chip, each handling separate threads. Cores may each use SMT or MT internally Throughput workloads, server environments Cannot hide per-core latencies across cores; each core limited to own threads
🔀 SMT (Hyper-Threading) Detail
  • Combines wide superscalar issue with multiple thread contexts
  • Single physical CPU appears as multiple logical CPUs to OS
  • On each cycle, selects instructions from any ready thread to fill issue slots
  • When threads are blocked (cache miss), other threads' instructions fill the pipeline
  • Physical resources shared: execution units, caches, TLB
✔

SMT improves throughput by ~20–30% with minimal die area increase compared to adding a full core.

⚖️ SMT vs CMP Comparison Figures 17.7j & 17.7k
FeatureSMT (Simultaneous MT)CMP (Chip Multiprocessor / Multicore)
Physical structure One core with multiple thread contexts and shared execution units Multiple independent cores on single chip
Thread isolation Low — threads share all execution resources High — each core has own pipeline, registers, L1/L2 cache
Instruction-level parallelism Higher — can mix instructions from multiple threads Lower — cannot hide latency across cores
Latency hiding Excellent — issues from other threads during stall Limited — only within-core multithreading
Single-thread performance Moderate — shared resources cause contention Better — dedicated per-core resources
vs Superscalar Better — fills issue slots from multiple threads Better — avoids horizontal losses of single-thread superscalar
Scalability Limited by single-core resources Scales by adding cores
📌

CMP with per-core multithreading (e.g., each core does SMT) combines the benefits of both approaches — this is standard in modern processors.

§17.5 — Clusters
🖧 Cluster Definition & Motivations

A cluster is a group of interconnected complete computers (nodes) working together as a unified computing resource. Each node is an independent system with its own CPU(s), memory, and OS.

  • Availability: node failure → cluster continues (failover). Passive standby → active standby → mutual takeover
  • Performance: distribute workload across nodes; parallel execution
  • Scalability: add nodes incrementally without redesigning system
  • Cost: commodity hardware (vs custom SMP) — better price/performance
ℹ️

Failover: when a node fails, another takes over its workload. Failback: when repaired node resumes its original role.

🗄️ Cluster Configurations
😴 Passive Standby

One primary server handles all load; standby is idle. Heartbeat monitor detects failure. Simple but wastes resources. Not really a cluster.

📤 Shared Nothing

Nodes share no disks; each owns partitioned volumes. On failure, other node takes ownership of failed node's volumes. Simple and scalable.

💾 Shared Disk

All nodes access a common disk subsystem. Any node can serve any data. Requires distributed lock management. High availability.

🔗 Active Cluster (Mutual Takeover)

All nodes are active; they share workloads and cover for each other on failure. Data must be continuously replicated. Best availability + performance.

🗂️ Blade Servers
  • Self-contained server modules (blades) inserted into a blade enclosure/chassis
  • Each blade has CPU(s), memory, local storage, NIC — a complete server
  • Chassis provides shared power, cooling, networking backplane
  • Very high density — hundreds of servers in a rack
  • Interconnected via 100GbE / 40GbE Ethernet switches
  • Common in modern cloud data centers
✔

Blade servers reduce cabling, power, cooling overhead vs. rack-mounted 1U servers while providing similar or higher compute density.

🖥️ Cluster OS / Middleware Design Issues
  • Single system image: user sees one system — single login, single file tree, single job manager
  • Single entry point — user logs onto cluster, not individual nodes
  • Single file hierarchy — shared root directory across nodes
  • Single control point — management console for entire cluster
  • Single virtual network — any node reaches any other via virtual network
  • Single memory space — distributed shared memory for variable sharing
  • Single job management — scheduler submits jobs cluster-wide
💻 Cluster Parallel Programming Models
  • Parallelizing compiler: auto-detects parallel sections; splits to nodes at compile time. Difficult to implement; performance varies
  • Parallelized application: programmer explicitly uses message passing (MPI/PVM) between cluster nodes. High burden on programmer but best control
  • Parametric computing: run same algorithm many times with different inputs; trivially parallelized across nodes
  • Message passing (MPI): standard communication API between nodes
⚖️ Clusters vs. SMP — Comparison Table
FeatureSMPCluster
Architecture Multiple CPUs sharing one physical memory on one board/bus Multiple complete independent computers connected by high-speed network
Communication Shared memory reads/writes — very fast (<ns) Message passing over network — higher latency (µs–ms)

§17.6 — Nonuniform Memory Access (NUMA)
📊 NUMA Motivation & Organization

NUMA bridges the gap between SMPs (limited scalability) and clusters (high communication overhead). Each node is effectively a mini-SMP with local memory, connected via high-speed interconnect to other nodes.

CC-NUMA (Cache-Coherent NUMA)
  • Most practical NUMA variant — adds hardware cache coherence across nodes
  • Each node: multiple processors + L1/L2 caches + local main memory + directory
  • Nodes interconnected by switching fabric (ring, crossbar, mesh)
  • Single system-wide address space — transparent to software
  • Local memory access: fast; remote memory access: slower (via interconnect)
  • Directory in each node tracks which caches hold each memory line
// CC-NUMA node structure Node 1: [CPU 1-1] [CPU 1-m] ← L1/L2 caches [Main Memory 1] ← Local memory [Directory] ← Coherence info [I/O subsystem] ↕ Interconnect network Node 2: [CPU 2-1] ... [Main Memory 2] ... Node N: [CPU N-1] ... [Main Memory N]
📋 NUMA Pros, Cons & Comparison
✅ NUMA Advantages
  • Scalable beyond SMP limits — more nodes = more total memory bandwidth
  • Effective performance at higher parallelism than SMP
  • Transparent to software — no major code changes vs SMP
  • Bus traffic per node remains manageable (local accesses are local)
  • L1/L2 caches minimize remote accesses; OS can allocate local memory for locality
❌ NUMA Disadvantages
  • Performance degrades if many accesses go to remote nodes
  • Directory overhead adds complexity and memory cost
  • Inter-node latency is non-deterministic
  • OS must be NUMA-aware to allocate memory locally (first-touch policy)

§17.7 — Cloud Computing
☁️ Cloud — Essential Characteristics (NIST)
  • Broad network access: Capabilities available over the network; accessible by heterogeneous thin/thick clients (phones, laptops, tablets)
  • Rapid elasticity: Resources expand/contract on demand; appear unlimited to user
  • Measured service: Automatic resource control/optimization via metering; pay-per-use
  • On-demand self-service: Consumer provisions compute/storage automatically without human interaction with provider
  • Resource pooling: Multi-tenant model; resources dynamically assigned; location independence
ℹ️

Definition per NIST SP 800-145. Cloud is NOT just outsourcing — it requires all five characteristics.

🏗️ Cloud Service Models (IaaS / PaaS / SaaS) NIST
🔧 IaaS — Infrastructure as a Service
  • Consumer accesses raw compute, storage, networking infrastructure
  • Consumer controls OS, middleware, applications
  • Provider manages physical hardware, virtualization layer
  • Examples: AWS EC2, Azure VMs, Google Compute Engine
⚙️ PaaS — Platform as a Service
  • Consumer deploys applications on provider-managed platform
  • Consumer controls apps and data, not OS or infrastructure
  • Provider manages runtime, middleware, OS, hardware
  • Examples: Google App Engine, Heroku, Azure App Service
📱 SaaS — Software as a Service
  • Consumer uses applications running on cloud infrastructure
  • Consumer has minimal control — only user-specific config
  • Provider manages everything including application
  • Examples: Gmail, Salesforce, Office 365, Dropbox
🌐 Cloud Deployment Models
🌍 Public Cloud

Infrastructure available to the general public; owned by cloud provider. Advantage: low cost, no management overhead. Concern: security, limited control over data location.

🔒 Private Cloud

Infrastructure operated exclusively for a single organization. Can be on-premise or hosted. Better security and control; higher cost than public cloud.

🏢 Community Cloud

Shared by several organizations with common concerns (mission, security, compliance). Shared cost with higher control than public cloud.

🔀 Hybrid Cloud

Combination of ≥2 deployment models (public + private). Allows "cloud bursting" — run routine workloads on private, peak loads on public. Most flexible but complex.

🖥️ Virtualization's Role in Cloud
  • Resource pooling: Hypervisor creates multiple VMs on single physical server; each VM is isolated and independently managed
  • Rapid elasticity: VMs can be created, cloned, migrated, or deleted in seconds
  • Multi-tenancy: Multiple customers' VMs co-exist on shared hardware with isolation
  • Live migration: Running VM moved to different physical host without downtime
  • Blade servers are common physical substrate — high density + shared infrastructure (power, cooling, networking)
✔

Without virtualization, cloud computing's on-demand provisioning and resource pooling would not be practical at scale.
